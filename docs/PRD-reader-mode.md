## 0. TL;DR

把 **沉浸阅读 + TTS 朗读** 做成 `_shared/reader.{js,css}` 共享组件,所有当前与未来的报告自动获得「点一下右下角 ▶ 进入大字号沉浸阅读 + 一键听书」能力。

- **目标体验**:对标 Chrome 阅读模式 + 微信读书听书 + 小宇宙 mini player,做出**报告语义感知**的差异化。
- **关键约束**:GitHub Pages 纯静态站,零后端,零 API Key 成本,**只用浏览器原生 API**(SpeechSynthesis + MediaSession + localStorage)。
- **推荐方案**:**走 L0+L1+L2 三段**(约 2.7 天),L3 留扩展位。
- **核心洞察**:我们 v4 已有的 `.plain .callout.*` 语义结构是**为听觉优化的天然资产**,做听书模式比通用网页阅读器有结构红利。

## 1. 背景与契机

### 1.1 触发事件

- 用户在使用 Chrome 阅读模式听报告时,发现体验良好,提出"能不能加到我们框架里"。
- 同步反馈:Chrome 阅读模式会**把 SVG 标签里的文字也读出来**,造成"FIG 4.2 三大供能系统功率峰值"这种碎片混入正文,体验断裂。

### 1.2 项目演进逻辑

| 版本 | 主轴 | 用户感知 |
|---|---|---|
| v1 | 内容深度(12 章学术骨架) | 信息完整 |
| v2 | 阅读辅助(术语词典 + 5 分钟回顾 + 反共识索引) | 信息可导航 |
| v3 | 视觉重塑(Newsprint 浅色 + AI 章首图) | 信息可呼吸 |
| v4 | 可读性嫁接(plain/ops/you/protocol) | **可读懂 + 可落地** |
| **v5** | **多通道消费(看 + 听)** | **可走读 / 可助眠 / 可通勤** |

v1→v4 一直在解决"看懂 + 看得下去"的问题;**v5 解决"在不能看的场景里也能消费"** —— 这是产品的下一个增量。

### 1.3 为什么现在做

- v4 字数已破 137K,**单遍阅读 6.5 小时**。从"信息密度高 → 可消费"的桥梁,听是必经之路。
- 报告作为通勤 / 跑步 / 睡前内容,**听书是天然消费场景**。
- 写作约定一旦定下来(`READER_RULES.md`),未来所有报告**零成本兼容**。延迟越久,改造成本越高。

## 2. 目标用户与场景

### 2.1 用户画像

| 角色 | 优先级 | 设备 | 场景 |
|---|---|---|---|
| **深度读者** | P0 | 桌面浏览器 | 长时段深读,眼睛累了切听 |
| **通勤听者** | P0 | 手机 + 耳机 | 地铁/公交 30-60 min |
| **睡前听者** | P1 | 手机 + 耳机/外放 | 助眠,需要定时关闭 |
| **跑步听者** | P1 | 手机 + 耳机 | 跑步 30-90 min,锁屏控制 |
| **分享传播者** | P2 | 任意 | 把某段发给朋友 |

### 2.2 核心场景(按优先级)

**S1 · 桌面端深读 + 切换听**(P0)  
你在桌面看到某段没看进去,点 ▶ 让它读给你听,你跟着浏览。

**S2 · 手机通勤听完一章**(P0)  
地铁上戴耳机,从首页选一章点听,锁屏后继续放,到站后从同一段恢复。

**S3 · 睡前定时助眠**(P1)  
躺床上听,设"30 分钟后停",超时自动暂停不打扰睡眠。

**S4 · 跑步时手机后台播放**(P1)  
跑步开始前切到听书模式,手机锁屏,耳机控制 ⏯ / 下一段。

**S5 · 分享某段给朋友**(P2)  
把"Ch11.3 ACWR 协议"那段的链接发朋友,他打开就在那段。

### 2.3 反场景(明确不做)

- ❌ 多用户协作笔记
- ❌ 朗读时的评论互动(小宇宙式时间戳评论)
- ❌ 云端 AI 音色(Medium 的教训:成本扛不住)
- ❌ OCR / PDF 朗读(Speechify 那种)
- ❌ 多设备实时进度同步(需后端)

## 3. 竞品调研(横向对标)

### 3.1 微信读书「听书」

**核心定位**:看书无缝切听书,同一本书两种消费方式。

| 模块 | 设计 | 借鉴 |
|---|---|---|
| 入口 | 阅读页右上"听"按钮 + 退出后底部 mini player 浮现 | ⭐⭐⭐ |
| 音色 | 4-6 个 AI 音色(含明星音色) | ⭐⭐ |
| 倍速 | **9 档** 0.5/0.75/1.0/1.25/1.5/1.75/2.0/2.5/3.0/3.5 | ⭐⭐⭐ |
| 定时关闭 | 5/10/15/30/60/90 min + **本章末** | ⭐⭐⭐ "本章末" |
| 跟读高亮 | 当前段落自动滚到中央 | ⭐⭐⭐ |
| 章末衔接 | 自动接下章 | ⭐⭐⭐ |
| 后台播放 | 锁屏控制 ⏯ + 下一章 | ⭐⭐⭐ Media Session |
| 跨设备 | 手机→桌面同步进度 | ⭐⭐ |
| 听 ≠ 看进度 | 听书和阅读分两个进度 | ⭐⭐ |
| 去除背景音 | AI 默认带轻 BGM,一键关 | ⭐ 我们不用 |

### 3.2 小宇宙「播客播放器」

**核心定位**:文字-时间双向绑定的播客 player,我们应该借鉴"双向绑定"范式。

| 模块 | 设计 | 借鉴 |
|---|---|---|
| **shownotes 时间戳** | `02:35 开场` `15:40 嘉宾观点` 点击跳到对应时间 | ⭐⭐⭐ **双向跳转** |
| 评论时间戳 | 用户在 23:15 留评论 → 评论挂在 23:15 时间点 | ⭐ 我们不做社交 |
| 章节标签 | 进度条上可见章节断点 | ⭐⭐ |
| **mini player** | 底部常驻折叠条 + 上滑全屏 | ⭐⭐⭐ |
| 倍速 | 6 档 0.5-3.0 | ⭐⭐ |
| 定时关闭 | 15/30/45/60 min + 本节目结束 | ⭐⭐⭐ |
| 节目主题色 | 节目封面自动渲染播放页背景 | ⭐ 视觉小惊喜 |

## 4. 核心洞察

### 洞察 1 · 共性的「必备七件套」

把所有评分 4+ 星的产品做交集,这 7 件**不做就不及格**:

```
1. 段落级高亮 + 自动滚动到屏幕中央
2. 倍速 6+ 档(含 0.75 / 1.25 / 1.75 微调档)
3. 定时关闭(必含「本章末」)
4. 后台 / 锁屏继续播(Media Session API)
5. 进度记忆(关闭再开继续)
6. 章节跳转 + 一章末自动接下章
7. mini player(折叠条,可一边浏览一边听)
```

### 洞察 2 · 我们 v4 的语义结构是**听觉资产**

v4 嫁接的 5 类组件几乎都是为听觉优化的:

| 组件 | TTS 友好度 | 增益 |
|---|---|---|
| `.plain` 白话翻译 | ⭐⭐⭐⭐⭐ | "人话:..." 自带提示词 |
| `.callout.sharp` 反共识 | ⭐⭐⭐⭐⭐ | "反共识:" 前缀 + 强调 |
| `.callout.ops` 3 件事 | ⭐⭐⭐⭐⭐ | 自动 ①②③ 分点朗读 |
| `.callout.you` 剧本 | ⭐⭐⭐⭐⭐ | 第二人称天然听感好 |
| `.callout.protocol` 协议 | ⭐⭐⭐⭐⭐ | 编号 + STOP 视觉强调 |

**这是 Chrome / Speechify 等通用工具做不到的差异化** —— 它们不懂我们的语义。

### 洞察 3 · GitHub Pages 静态站的**约束转机会**

| 通常路径 | 我们的约束 | 转化为 |
|---|---|---|
| 上云端 TTS | ❌ 零后端 + 零 API Key | 用浏览器原生 SpeechSynthesis(免费) |
| 跨设备同步 | ❌ 零账号系统 | URL hash 分享(分享段落) |
| 收费音色 | ❌ 零商业化 | 系统音色枚举(让用户挑) |
| 笔记云存 | ❌ 零云存 | localStorage(单设备,但够用) |

Medium 当年下架 Listen 的原因(成本)正好不在我们身上。

### 洞察 4 · 「本章末」是杀手锏

微信读书的"听完本章自动停"档,在其他产品都没有。

- 通勤场景:听到地铁站,自然在章末停
- 助眠场景:听完一章可能正好入睡
- 是**比固定时长定时更聪明的设计**

### 洞察 5 · 桌面 vs 手机是两个**完全不同的产品**

| 维度 | 桌面 | 手机 |
|---|---|---|
| 主场景 | 边看边听 / 切换听 | 锁屏听 / mini player |
| 触发位置 | 右下角悬浮 | 阅读页右上"听"按钮 + 底部 mini |
| 控制条位置 | 顶部(像 Chrome) | **底部**(拇指可达) |
| 退出方式 | ESC + ╳ | 上滑收 mini |
| 字号默认 | 22-24px | 18-20px |
| 主操作大小 | 40px | 56px(拇指目标) |

**不能用一套 UI 硬适配两端,要做"响应式 + 行为差异化"。**

### 7.5 报告写作约定 · `READER_RULES.md`

```markdown
# Reader Rules · 报告写作约定

为让报告兼容阅读模式 + TTS,新报告写作时遵守:

## ✅ 鼓励
- 用语义化标签:<h2><h3><h4><p><ul><ol><li>
- 实操内容用 .callout.{plain|sharp|ops|you|protocol|story|note}
- 装饰性图表用 <svg>(默认会被阅读模式隐藏)
- 数据表用 <table>(默认隐藏,加 data-summary="..." 给 TTS 摘要可选)

## ⚠️ 注意
- SVG 内部的 <text> 标签也会被一起跳过
- 关键数字尽量同时出现在 <svg> 和正文 <p> 里(冗余但听觉友好)
- 章节标题统一用 h3,小节用 h4

## ❌ 避免
- 用 <span class="some-style">把正文藏进 SVG 容器
- 用 <img> 装可读文字(没法被 TTS)
- 在 <table> 里放关键洞察(应该同时写一段正文)
```

## 11. 附录

### A · 关键代码草图

```js
// _shared/reader.js (简化版核心)

class Reader {
  constructor() {
    this.config = this.loadConfig();
    this.progress = this.loadProgress();
    this.tts = new TTSEngine();
    this.ui = new ReaderUI();
  }

  open() {
    const chapter = this.detectVisibleChapter();
    const chunks = this.extractChunks(chapter);
    this.ui.render(chunks, this.config);
    this.resumeFromLastChunk();
  }

  extractChunks(chapter) {
    return Array.from(chapter.querySelectorAll(INCLUDE_TAGS.join(',')))
      .filter(el => !this.isExcluded(el))
      .map(el => ({
        id: this.genId(el),
        el,
        text: this.semanticAugment(el),  // 反共识加前缀等
        type: this.classify(el),
      }));
  }

  semanticAugment(el) {
    if (el.closest('.callout.sharp')) return '反共识。' + el.textContent;
    if (el.closest('.callout.protocol')) return el.textContent;
    if (el.closest('.callout.ops')) return el.textContent;
    return el.textContent;
  }
}
```
