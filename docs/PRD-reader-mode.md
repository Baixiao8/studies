## 0. TL;DR

### 0.1 产品一句话定义

> **不是给报告加播放按钮,是把研究报告从「久坐型阅读品」升级为「随身型消费品」。**

技术形态上,这是一个 `_shared/reader.{js,css}` 共享组件;产品形态上,这是研究报告族的**消费通道扩展** —— 在"看"之外打开"听"的第二条路径。

### 0.2 核心问题(为什么用户需要)

v4 跑步报告字数已破 **137,411 字 / 单遍阅读 6.5 小时**,但当前用户消费方式 100% 被锁定在"坐下 + 看屏幕 + 全程主动注意力"。**通勤 / 跑步 / 家务 / 助眠 这四类「不能看屏幕」的场景日均占用户 2.5-4.5 小时,目前消费完全空白**。这是深度报告类内容的最大未满足需求。

### 0.3 用户证据(决策的强信号)

> **用户已经在用 Chrome 阅读模式听我们的报告(自反馈触发本 PRD)。**

这是产品-市场契合度的硬信号:**需求真实存在,但被通用工具勉强满足**。Chrome 阅读模式跑通用网页,在我们重图重表的报告里:读 SVG 内部标签("FIG 4.2 · 三大供能系统功率峰值")混入正文 / 朗读表格行 / 朗读章节地图与参考文献编号 / mini-TOC 触发跳回顶部 —— **体验在我们的语义结构上断裂**。

### 0.4 价值主张(为什么必须是我们做,不是用 Chrome)

我们 v4 已有的语义资产 —— `.plain` 白话翻译(99 处)/ `.callout.sharp` 反共识(15 处)/ `.callout.protocol` 章末协议(12 处)/ `.callout.you` 第二人称剧本(31 处)—— **本身就是为听觉优化的内容结构**。Chrome / Speechify / Apple Books 等通用工具做不到这一档,因为它们不懂我们的语义。

**这不是抄一份阅读模式,这是做一份比通用工具懂内容的体验。**

### 0.5 关键约束 → 转化为机会

| 约束 | 一般会怎么破 | 我们的转化 |
|---|---|---|
| GitHub Pages 纯静态 / 零后端 | 上 Cloudflare Workers | 用浏览器原生 API 闭环 |
| 零 API Key / 零商业化预算 | 接 OpenAI TTS(成本高) | 浏览器原生 SpeechSynthesis(免费) |
| 不存账号系统 | 上 Firebase | localStorage 单设备记忆 + URL 分享 |

**Medium 当年因为云端 TTS 成本扛不住下架了 Listen 功能** —— 这条陷阱不在我们身上,因为约束规避了它。

### 0.6 首发版本范围

```
v5.0 = L0 写作约定 + L1 基础阅读+TTS + L2 听书化升级
工期  ≈ 2.7 天(分 2-3 次推进)
范围  覆盖现有跑步报告 + 未来全部报告(_shared/ 框架级)
不含  L3 段落双向跳转 / 分享段落 URL(留扩展位,看反馈决定是否加)
```

### 0.7 4 周成功标准

| 指标 | 目标值 | 含义 |
|---|---|---|
| 阅读模式打开率 | ≥ 15% | 至少 1/7 访问会话会进入阅读模式 |
| 朗读完成率 | ≥ 30% | 进入后 30%+ 用户朗读 > 1 段 |
| 移动端占比 | ≥ 40% | 听书的核心场景在手机 |
| 设置调整率 | ≥ 40% | 进入用户调过字号/配色/速度任一 = 在意 |
| 单次会话时长 | ≥ 5 min | 不是好奇点开,是真有人在用 |

### 0.8 [DECISION] 一句话推荐

> **走 L0+L1+L2(2.7d 起步),不做激进 L3** —— 先把"必备七件套"做到位,L3 的双向跳转/语义朗读看 4 周实际使用数据再决策。

## 1. 背景与契机

### 1.1 资产现状(数据先行)

```
v4 跑步报告
├── 137,411 字 · 12 章 · 单遍阅读 6.5 小时
├── 30+ SVG 数据图 · 80+ 数据卡 · 12 张 AI 章首图
├── 99 个白话翻译(.plain)
├── 69 个 3 件事清单(.callout.ops)
├── 31 个第二人称剧本(.callout.you)
├── 12 个章末协议清单(.callout.protocol)
└── 53 个可落地训练协议 · 30+ 反共识洞察
```

**这是一份内容密度高、结构语义化、写作精细的产品** —— 但当前 100% 消费场景被锁定在"坐下 + 看屏幕"。

### 1.2 真实的问题在哪里(三层痛点拆解)

#### 痛点 1 · 消费场景被锁死,深度报告难以"消费完整"

成年知识工作者日均"能坐下专注看屏幕"的时间窗口约 **1-2 小时**(扣除工作 / 通勤 / 家务 / 运动 / 社交 / 睡眠 / 碎片)。v4 报告单遍 6.5 小时意味着**用户几乎不可能在合理周期内消费完整份报告**。

**实际行为表现**(基于公开访问数据合理推断):多数用户消费片段(挑章节看)而非整本,完整知识结构难以建立。

#### 痛点 2 · 用户已尝试自救但体验断裂(强信号)

> **本 PRD 的触发事件:用户主动反馈"我在用 Chrome 阅读模式听这份报告"。**

这是**需求被通用工具勉强满足**的产品-市场契合度强信号。但 Chrome 阅读模式跑通用网页,在我们这种重图重表的研究报告里有 4 处明确断裂:

| Chrome 阅读模式行为 | 我们报告里的表现 | 体验影响 |
|---|---|---|
| 读 `<svg>` 内的 `<text>` 标签 | "FIG 4.2 · 三大供能系统 ATP CP 糖酵解 有氧" | 数字与名词碎片插入正文 |
| 读 `<table>` 全部行内容 | "风险因素 证据等级 OR 可改造性" | 表头表格反复念出 |
| 读 mini-TOC / 章节地图 / 数据卡 | "CHAPTERS 12 DEPTH 137K DIAGRAMS 30+" | 数字洪流打断阅读流 |
| 朗读到 `<p class="ref">` 参考文献 | "Brooks 2018 Cell Metab Joyner Coyle 2008" | 文献名密集且毫无意义 |

**Chrome 阅读模式的体验下限,就是我们要超越的起点。**

#### 痛点 3 · "不能看"的高潜场景完全空白

| 用户场景 | 日均时长(知识工作者) | 当前我们的消费 | 替代方案 |
|---|---|---|---|
| 通勤(地铁/公交) | 60-90 min | 0(无法看) | 播客 / 听书 App |
| 跑步 / 健走 | 30-60 min | 0 | 音乐 / 播客 |
| 家务(做饭 / 打扫 / 通勤步行) | 30-60 min | 0 | 播客 / 音乐 |
| 睡前最后 30 min | 30 min | 0(屏幕影响入睡) | 听书 App |
| **可用窗口总计** | **2.5-4.5 h/d** | **0(完全空白)** | **被其他产品瓜分** |

**用户每天有 2.5-4.5 小时**可以消费深度内容,但**我们目前在这块的占比是 0**。深度报告的潜在消费时长,被听书 App 和播客瓜分了。

### 1.3 项目演进逻辑 · v5 是质变,不是迭代

```
v1  内容深度        ─→ 让报告值得读
v2  阅读辅助        ─→ 让报告读得动
v3  视觉重塑        ─→ 让报告读得久
v4  可读性嫁接      ─→ 让报告读得懂 + 用得上
    ═══════════════ 以上 4 版,全部是"看"通道的精化 ═══════════════
v5  双通道消费      ─→ 让报告"听得着" → 真正可消费
    ═══════════════ v5 是开辟第二条消费通道 ═══════════════
```

v1-v4 走的是同一条路径(优化"看"的体验);**v5 是**消费方式的质变,不是再加一个 nice-to-have 功能。

### 1.4 机会窗口(为什么现在做最划算)

#### 1.4.1 正向理由

| 因素 | 当下状态 | 时间紧迫性 |
|---|---|---|
| **语义资产已具备** | v4 的 `.plain` `.callout.*` 是天然听觉结构 | ⏰ 现在做边际成本最低 |
| **浏览器原生 API 已成熟** | Chrome/Edge/Safari 全部稳定支持 SpeechSynthesis + MediaSession | ⏰ 不依赖第三方 |
| **报告数量少** | 目前只有 1 份报告需要适配 | ⏰ 现在适配 1 份 vs 未来适配 N 份 |
| **用户主动反馈触发** | 真实需求被表达 | ⏰ 响应窗口期 |

#### 1.4.2 不做的代价(反向论证)

| 不做的代价 | 影响 |
|---|---|
| 每加一份新报告,**改造成本累积**(估计每报告事后适配 ~0.5d) | 第 N 份报告时已累积 0.5N 天技术债 |
| 用户继续用 Chrome 阅读模式 | 体验断裂可能被归咎为"我们报告写得不好" |
| 写作时不考虑朗读兼容 | 新报告产生新的"反 TTS 内容"(密集表格、未标记 SVG) |
| 错失"听觉化研究报告"类目 | 目前国内尚无同类竞品,先发优势窗口期短 |

#### 1.4.3 关键洞察:写作约定优先做

**`_shared/READER_RULES.md` 的优先级 > 实际功能开发。**

写作约定一旦定下:
- 现有 v4 报告:已经 80% 兼容(感谢 v4 语义化设计)
- 未来报告:从写作第一行就遵守 → 零适配成本
- 反例:微信公众号文章 90% 不兼容 Chrome 阅读模式(没语义化)

### 1.5 产品愿景(3 / 6 / 12 月画面)

#### 3 个月后(2026-08 · v5 上线 4 周)

- v5.0 上线 4 周,**指标全部命中**(参见 0.7 节)
- 1-2 份新报告写作时**第一行就遵循 READER_RULES**,零适配上线
- 用户社群出现"在地铁上听完了 Ch10"这类反馈

#### 6 个月后(2026-11 · 累积 3-4 份报告)

- "听报告"成为产品族的**标配消费方式**,与"看报告"对等
- Mini player 跨章节连听完成率 > 50% → 用户真在用"听书"模式
- 阅读模式 + 听书功能成为产品演示时的 **top 卖点**

#### 12 个月后(2027-05 · v5 第一周年)

- 4-6 份报告全部双通道兼容
- 形成"深度内容听觉化"的**产品类目认知**
- 写作约定 + 共享组件被其他研究报告项目借鉴(开源价值)
- 是否需要 L3 / L4(双向跳转 / 词级跟读 / 笔记)由真实使用数据驱动决策

## 2. 目标用户与场景

### 2.1 用户分层(2×2 矩阵 · 消费场景 × 设备)

我用「消费场景类型」× 「主用设备」二维矩阵划分用户群体:

```
                          桌面浏览器              手机浏览器
                       ┌────────────────────┐  ┌────────────────────┐
深读场景  P0           │  Q1 桌面深读+辅助听 │  │  Q2 手机碎片深读    │
(主动消费 / 高专注)    │     切换式 / JOB-1  │  │     阅读模式优化     │
                       └────────────────────┘  └────────────────────┘
                       ┌────────────────────┐  ┌────────────────────┐
听书场景  P0           │  Q3 桌面伴随听      │  │  Q4 手机锁屏纯听    │
(被动消费 / 中专注)    │     一边干别的      │  │     通勤/跑步主场   │
                       └────────────────────┘  └────────────────────┘
```

#### 用户群体行为特征

| 象限 | 角色 | 典型行为 | 设备使用 | 痛点严重度 |
|---|---|---|---|---|
| **Q1** 深度读者 | 周末/夜晚长时段读 | 桌面 + 笔记本,可能两个屏 | 主桌面 / 切手机 | ★★(可看,但累) |
| **Q2** 碎片化读者 | 等人 / 排队 / 短暂空闲 | 手机滚屏 5-15 分钟 | 100% 手机 | ★★(屏幕小) |
| **Q3** 桌面多任务 | 写代码 / 整理资料时背景听 | 桌面前坐着 | 桌面 + 耳机 | ★★★(没法看,只能纯听) |
| **Q4** 移动场景 | 通勤 / 跑步 / 家务 / 睡前 | 锁屏 / 手机塞口袋 | 100% 手机锁屏 | ★★★★★(完全无解决方案) |

**核心结论**:Q4 是最大未满足象限,Q1 是当前唯一被服务的象限,Q2 / Q3 介于其间。

### 2.2 JTBD 框架 · 4 类核心情境

按 Jobs-to-Be-Done 框架,拆解每类用户在什么情境下、雇用本产品做什么工作、用什么标准衡量、当前替代方案是什么。

#### JOB 1 · 桌面深读 + 切换辅助听(Q1 主场景)

```
情境  · 周末晚上 / 工作日午休,坐在桌前 30-90 min
触发  · 某段反复读不进去 / 眼睛累 / 想换种节奏 / 长 callout 想听
雇用  · 让产品念给我听,我跟着滚动浏览或闭目休息 3-5 min
成功  · 没流失,继续深读 / 关键内容被听到 + 看到双重接收
失败  · 朗读体验断裂 → 关产品去刷手机 / 朗读读错重要数字
替代  · ① Chrome 阅读模式(读 SVG 标签) / ② 复制段落到第三方 TTS(成本高)
        / ③ 让 AI 助手念出来(没有产品体验)
现状  · 用户已尝试 Chrome 阅读模式 → 反馈"读 SVG 文字" → 触发本 PRD
```

#### JOB 2 · 手机通勤听完整章(Q4 第一主场景)

```
情境  · 地铁 / 公交 / 步行 30-60 min,戴耳机,手机在口袋,屏幕锁
触发  · 出门时主动启动,或承接前一次未听完的章节
雇用  · 让产品在锁屏下连续朗读,通勤时长内听完 1 章
成功  · 到站时正好章末 + 关键内容(反共识 / 协议)记得
失败  · 锁屏 30s 后停 / 找不到上次位置 / 数字读错 / 听完没记住
替代  · ① 播客 App(但播客内容是别人的,不是我自己的报告)
        / ② Pocket Listen / Speechify(无针对中文长报告优化)
        / ③ 不消费(放弃这块时间)
现状  · 完全空白(0% 满足)
```

#### JOB 3 · 睡前定时助眠(Q4 第二场景)

```
情境  · 躺床上,屏幕关或调到最暗,需要平静节奏入睡
触发  · 睡前 20-30 min 想消费但不能强刺激
雇用  · 低音量朗读,设"30 分钟后停",超时自动暂停
成功  · 入睡时产品自动停,第二天能从同一段恢复
失败  · 没停继续放(吵醒) / 第二天找不到位置 / 朗读太亢奋
替代  · ① 喜马拉雅 / 微信读书(要单独导入我们的内容)
        / ② Apple Podcasts 定时器(同样要导入)
        / ③ 不消费
现状  · 完全空白
```

#### JOB 4 · 跑步锁屏听(Q4 第三场景)

```
情境  · 跑步 30-90 min,手机臂带,屏幕锁,只能耳机线控
触发  · 跑步前主动选定章节,跑步中靠耳机控制
雇用  · 让产品后台播放,耳机 ⏯ / 双击下一段
成功  · 跑完时进度合理(15-30K 字),关键洞察被听到
失败  · iOS 锁屏 30s 后停(系统级硬伤) / 没有"下一段"控制
        / 漏掉关键反共识 / 跑出汗手没法操作屏幕
替代  · ① 听播客(内容不是自己的)/ ② 音乐(只是背景)
现状  · 完全空白 + iOS 锁屏限制(系统级,需特别处理)
```

### 2.3 核心场景优先级矩阵

按 **频次 × 单次价值 × 当前满足度 × 实现复杂度** 综合排序:

| ID | 场景 | 频次 | 单次价值 | 当前满足 | 复杂度 | **优先级** | 关联 JOB |
|---|---|:---:|:---:|:---:|:---:|:---:|:---:|
| S1 | 桌面深读 + 切听 | ★★ | ★★★ | ★(Chrome 勉强) | 低 | **P0** | JOB 1 |
| S2 | 手机通勤听书 | ★★★ | ★★★ | ★(限制大) | 中 | **P0** | JOB 2 |
| S3 | 睡前助眠 | ★ | ★★ | 0 | 中 | **P1** | JOB 3 |
| S4 | 跑步锁屏 | ★★ | ★★★ | 0 | 高(iOS 限制) | **P1** | JOB 4 |
| S5 | 家务时听 | ★★ | ★★ | 0 | 中(同 S2) | **P2** | JOB 2 衍生 |
| S6 | 分享段落 URL | ★ | ★★ | 0 | 低 | **P2** | 传播 |

**决策含义**:
- **S1-S2(P0)** 必须首发就完美 → L1 + L2 范围
- **S3-S4(P1)** 在 L2 听书化阶段补齐 → "本章末"定时 / Media Session
- **S5-S6(P2)** 看 4 周反馈再决定 → L3 范围

### 2.4 关键用户旅程图(以 JOB 2 通勤听书为典型)

```
时间轴 →  T-5min   T0        T+15min       T+30min     T+45min

        ├────────┼─────────┼──────────────┼─────────┼──────────────┤
用户    │ 选章节  │ 戴耳机   │ 走路 / 看书   │ 出地铁站│ 重新启动       │
行为    │ 点 ▶   │ 按 ⏸ 开始 │ 偶尔耳机控制 │ 关 app  │ 自动从同一段续 │
        ├────────┼─────────┼──────────────┼─────────┼──────────────┤
产品    │ 章节列表│ TTS 启动 │ 锁屏 +       │ 段落级  │ localStorage │
要做的  │ + 听书入│ + 段落级 │ Media Session│ 进度记忆│ 恢复 +       │
        │ 口可见  │ 高亮     │ + 后台播放   │         │ mini player  │
        └────────┴─────────┴──────────────┴─────────┴──────────────┘

关键体验断点(产品必须避免):
✗ 听书入口不显眼 → 用户找不到怎么开始
✗ 锁屏后 30 秒 TTS 自动停(iOS 系统行为)
✗ 关 app 后下次找不到上次位置
✗ 耳机线控只能 ⏯,没办法跳下一段
✗ 章末突然停顿 → 用户以为坏了 → 解锁手机操作

关键体验亮点(差异化):
✓ 章末自动接下章(微信读书式)
✓ 锁屏通知栏显示当前章节 + 段落进度
✓ "本章末停"定时(微信读书独家功能)
✓ 反共识 / 协议读到时音调强调(我们独家)
```

### 2.5 非目标用户与反场景(明确划界)

为聚焦核心场景,以下用户群体 / 场景**明确不服务**:

| 不服务的用户 / 场景 | 不做的理由 |
|---|---|
| 学术研究者打印 PDF 阅读 | 我们的目标是数字消费,不是纸质 |
| 教学场景需要笔记同步 | 需要后端 + 多用户系统,目标外 |
| 视障用户全屏幕辅助朗读 | 应该用系统级屏幕阅读器(VoiceOver / TalkBack) |
| 英文报告 / 翻译朗读 | 当前内容 100% 中文,留作未来 |
| 朗读时的评论互动(小宇宙式时间戳) | 我们不是社交内容 |
| 多设备实时进度同步 | 需后端,本期不做 |
| 收费 AI 音色 / 明星音色 | 商业模式不允许,Medium 教训 |
| 词级跟读(Speechify 式) | Web Speech API 不可靠 |
| OCR 朗读截图 / 图片 | 目标外 |
| 离线下载 / 缓存到本地 | 报告本身就是文本,无意义 |

**反场景列出的目的**:避免产品在 4 周后被"为什么不支持 X"的扩张需求拉偏。这些是**有意识的不做**,不是漏做。

## 4. 核心洞察

> 洞察 ≠ 总结。这一节列出的 5 条,都是**通过竞品对比、用户行为和技术约束综合推断出的、对产品方向有 directional 决策价值的非显然结论**。每条都附"如果反过来想会怎样"以确认洞察成立。

### 洞察 1 · "必备七件套"在我们身上变成"五件套",共性 ≠ 起点

竞品矩阵显示 7 件功能是行业共性,但**我们的约束让其中 2 件无法做** —— 跨设备进度同步(需后端)和云端 AI 音色(成本)。这意味着:

```
通用产品的及格线 = 七件套
我们的及格线    = 五件套 + 用差异化补两个缺口

缺口 1: 跨设备同步 → 用 URL 锚点分享段落补偿
缺口 2: 优质 AI 音色 → 用语义朗读强化补偿(我们独家)
```

**反向验证**:如果硬上云端 TTS 追平七件套 → 重蹈 Medium 覆辙;如果只做五件套不补缺口 → 体验明显不如微信读书。**两条腿走路:基础对标 + 差异化补偿,缺一不可。**

### 洞察 2 · TTS 体验的主要变量是「内容结构」而非「音色质量」

业界惯性认为 TTS 升级路径是音色升级(原生 → 系统增强 → 云端 AI → 真人录音)。但我们的处境提供了反例:

**同样用浏览器原生 SpeechSynthesis 朗读**:
- 通用网页(微信公众号 / 知乎文章):体验 ★★(标签密度低,句子忽长忽短,数字与名词混淆)
- 我们的报告(v4 语义化):体验 ★★★★(段落语义明确,callout 自带提示词,数字有上下文)

**这意味着:在我们这种重写过的内容上,原生 TTS 的体验上限,可能高于 Medium 当年用云端 TTS 跑普通文章。**

**反向验证**:让 Chrome 阅读模式读我们的 v4 报告 → 体验比读知乎文章好;让 Chrome 阅读模式读 v1 时期的旧报告 → 体验明显差。证实"内容结构 > 音色"。

**决策含义**:把工程预算花在 **黑名单过滤 + 语义朗读强化** 上,**不在音色升级上花钱**。

### 洞察 3 · 听书产品的护城河是"章节节奏",不是"单点功能"

观察:微信读书 + 小宇宙 + Pocket 三家听书相关产品都共享一对组合 ——

```
"本章末停" + "自动接下章" = 章节为消费单元的节奏
```

**这对组合不能拆开**,单独的"定时关闭"和"自动接下章"在其他产品(Apple Books / Chrome)有,但缺乏"章节"这个中间粒度。

**为什么这是护城河**:
- 用户对深度内容的消费记忆单元是"章"(我读到第 3 章了 / 听完了 Ch4)
- 通勤 30-60 min ≈ 1 章自然时长
- 助眠 / 跑步同理
- 没有"章"这个粒度,听书体验退回"听一段是一段"的散乱感

**反向验证**:Apple Books 有定时关闭但**没有"本章末"**,导致它的听书体验比微信读书低一档,即使它的音色更好。

**决策含义**:`F-101 定时关闭(含本章末)` + `F-102 章末自动接下章` 这两项必须组合上线,不能 P1 部分上线。

### 洞察 4 · 桌面 vs 手机是两个产品,共享代码但分开设计

通常 PRD 会写"做响应式适配两端"。但更深的观察是:

```
桌面端阅读模式  ≈ "深读时的辅助功能"(切换式)
手机端阅读模式  ≈ "独立的媒体消费方式"(独立式)
```

**两端的核心体验不同**:

| 维度 | 桌面(切换式) | 手机(独立式) |
|---|---|---|
| 用户预期 | 我在深读,需要片刻辅助 | 我要消费内容,但不能看屏幕 |
| 触发位置 | 阅读页右下角悬浮(不打扰深读) | 阅读页显眼入口 + 启动后全屏 |
| 控制条 | 顶部横向铺开(像 Chrome) | 底部(拇指可达) |
| 退出 | ESC 回到深读,精确位置 | 收 mini player 继续听 |
| 主操作 | 40px 鼠标精度 | 56px 拇指目标 |
| 字号默认 | 22-24px(显示器距离) | 18-20px(手机距离) |
| 主流场景 | JOB 1 切换辅助 | JOB 2 / JOB 3 / JOB 4 锁屏听 |

**反向验证**:如果把桌面 UI 等比缩到手机 → 控制条变小、拇指点不到、字号过小;如果把手机 UI 拉大放桌面 → 控制条占用屏幕、像低端 app。证实"一套 UI 硬适配两端"会同时伤害两端。

**决策含义**:`reader.js` 单代码,但 `reader.css` 分桌面 + 手机两套布局规则。

### 洞察 5 · Mini Player 不是 nice-to-have,是听书场景的"中枢"

**观察**:微信读书 / 小宇宙 / Pocket / Speechify 都有 mini player;Chrome 阅读模式没有 → Chrome 完全没有"听完关页面后继续听"的体验路径。

**Mini Player 的真正作用**:

```
没有 mini player → 听书必须停留在阅读模式 → 不能浏览其他章节 → 听书是封闭场景
有 mini player → 听书时可以滚到其他章节 / 看 TOC / 查反共识索引 → 听书是开放场景
```

**这就是为什么微信读书听书体验完爆 Chrome 阅读模式** —— 不是音色更好,不是控制条更精致,而是 mini player 让"听 + 浏"可以并行。

**反向验证**:如果只做 L1 不做 mini player,用户会怎么用?
- 桌面用户:听 5 分钟想看下一章预告 → 关阅读模式 → 听书停了 → 烦
- 手机用户:听通勤,想看微信/邮件 → 切到后台 → iOS 30s 后停了 → 体验断了

**决策含义**:Mini player 必须在 L2 上线,不能延到 L3 / P2。它是听书场景能不能闭环的关键基础设施,不是装饰。

### 洞察小结 · 5 条对应 5 个设计决策

| 洞察 | 直接决策 |
|---|---|
| 1 · 五件套 + 差异化补偿 | 不上云端 TTS,但要做语义朗读强化 |
| 2 · 内容结构 > 音色 | 工程投入黑名单过滤 + 语义增强,不投音色 |
| 3 · 章节节奏是护城河 | "本章末" + "自动接下章" 必须组合上线 |
| 4 · 桌面手机是两个产品 | 单代码 + 双 CSS 布局,不做硬适配 |
| 5 · Mini player 是中枢 | L2 必须有 mini player,不能延后 |

## 6. 设计方案

### 6.1 信息架构

```
┌─ 报告页(正常浏览) ─────────────────────────────┐
│                                                  │
│   ...正常报告内容...                              │
```

## 11. 附录

### A · 关键代码草图

```js
// _shared/reader.js (简化版核心)

class Reader {
  constructor() {
    this.config = this.loadConfig();
    this.progress = this.loadProgress();
    this.tts = new TTSEngine();
    this.ui = new ReaderUI();
  }

  open() {
    const chapter = this.detectVisibleChapter();
    const chunks = this.extractChunks(chapter);
    this.ui.render(chunks, this.config);
    this.resumeFromLastChunk();
  }

  extractChunks(chapter) {
    return Array.from(chapter.querySelectorAll(INCLUDE_TAGS.join(',')))
      .filter(el => !this.isExcluded(el))
      .map(el => ({
        id: this.genId(el),
        el,
        text: this.semanticAugment(el),  // 反共识加前缀等
        type: this.classify(el),
      }));
  }

  semanticAugment(el) {
    if (el.closest('.callout.sharp')) return '反共识。' + el.textContent;
    if (el.closest('.callout.protocol')) return el.textContent;
    if (el.closest('.callout.ops')) return el.textContent;
    return el.textContent;
  }
}
```
