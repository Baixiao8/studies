## 0. 摘要

**Q**:Studies 项目要不要新增"导出报告到 epub / PDF"的功能?

**A**:**暂不做导出。**

**理由(三句话)**:

1. 需求来源是作者一人(N=1),真实诉求是"想听好",epub 是 workaround,真正的 root cause 是播放器有 bug
2. 在《加功能三问》宪法下,导出功能 Q2(叠加新场景)+ Q3(失败代价高)两问红灯
3. 修 bug 成本 < 1 天,做导出工程成本 + 长期维护成本远高于此;先修 bug 观察,需求大概率自然消失

**优先级**:

- **P0** · 修两个 bug(导航定位 / 段落循环) — 不在三问限制内
- **P1** · bug 修后观察作者本人还想不想跳出去微信读书听
- **P2**(如真需要)· 做 PDF,不做 epub
- **P3**(可能永远不到)· 做 epub

## 1. 需求背景

### 1.1 需求来源(诚实记录)

| 项 | 描述 |
|---|---|
| 提出人 | **项目作者(白笑)本人** |
| 提出方式 | 在自己使用 Studies 报告时,觉得当前播放器体验有问题 |
| 提出动机 | **想绕过当前播放器,去微信读书等成熟听书 APP 听** |
| 用户量级 | **N = 1** |
| 是否有数据支撑 | 无 — 没有他人反馈、使用率、痛点频率统计 |

**这是一条诚实的招供**:需求不是来自用户群,是作者本人的痛点。这条事实是后续所有判断的基础,必须白纸黑字。

### 1.2 真实需求拆解

| 表层诉求 | 真实需求 | 是否同一件事 |
|---|---|---|
| "想要 epub 导出" | "想要好的听书体验" | **❌ 不是** |

epub 在这里是**手段**,不是目的:

```
真实链路:
  播放器有 bug → 听起来不爽 → 想去微信读书听 → 想要 epub 格式

反过来推:
  播放器修好 → 听起来爽 → 不需要跳出去 → epub 需求消失
```

也就是说,**epub 这个需求的存在,完全依赖于"播放器 bug 不修"这个前提**。前提一去,需求随之消失。

### 1.3 关联的 root cause:播放器两个 bug

| Bug | 现象 | 根因(初步诊断) | 修复成本 |
|---|---|---|---|
| **导航定位** | 点击导航不能切换到当前章节,要点好几次才能定位 | scroll spy 与 click 事件打架——点击触发 `scrollIntoView`,滚动过程中 scroll spy 又把 active 改回去 | 30 min - 2 h |
| **段落循环** | 点击播放后会循环播放当前段落 | **Chrome SpeechSynthesis 长 utterance(>~15s)自动重启 bug** — 整章塞进一个 utterance 必中(项目用 Web Speech API,不是 HTML5 `<audio>`,详见 §4.1 实修订正) | 1 - 2 h |

**总成本 < 1 个工作日**。两个 bug 修复完,真实需求"想听好"自然满足。

## 2. 用《加功能三问》拷问导出功能

> 项目宪法 · `PRINCIPLES.md` 第十节
> v5-v6 没认真问 → 1850+ 行听书代码 + 227MB 音频 → 最终砍掉

### Q1 · 用户明确说要了吗?(不是一句话引申)

| 评估维度 | 答案 |
|---|---|
| 有人说要吗? | ✅ 有 — 项目作者本人 |
| 数量? | 1 人 |
| 强度(具体场景描述)? | 弱 — 指向"想绕过 bug",不是"我就需要 epub 这个格式" |
| 是否有其他用户反馈? | ❌ 无 |

**评分:🟡 黄灯**——技术上"有人提",但本质是 vocal minority 的极端版(N=1),且需求强度弱(指向格式 ≠ 指向价值)。

### Q2 · 强化核心 还是 叠加新场景?

| 项目核心 | 这次需求 |
|---|---|
| **高密度高质量的研究报告消费**(读 + 听 + 在 Studies 内停留) | "把报告导出到外部 APP 消费" |

**评分:🔴 红灯**——明确是**叠加新场景**。

更糟的是:导出本质是**让用户离开**,跟项目核心"留住用户深度消费"是**反向**的——做了反而稀释核心价值。

### Q3 · 失败的代价是什么?

| 代价类型 | 评估 |
|---|---|
| 工程开发成本 | 中-高 · SVG / callout / tooltip / mini-TOC / 学科色卡片 / 章末回顾 全要转换 |
| 长期维护成本 | **高** · 不同 epub reader(Calibre / Apple Books / 微信读书 / ReadEra)渲染不一致,长期 case by case 修 |
| 设计降级 | **高** · Newsprint 报刊风 + Inter Tight + 苹方 + SVG 颜色变量 在 epub 阅读器里**几乎全损**(详见 §3.1) |
| 留存断点 | **高** · 导出后用户离开项目,流量与互动归零 |
| 一致性维护 | **高** · 每次报告更新(已有 v8.x 多次迭代)epub 也要重新生成,workflow 复杂度翻倍 |
| 心智复杂度增加 | 中 · 多一个入口、多一个失败可能、多一份文档维护 |

**评分:🔴 红灯**——综合代价远高于不做。

| 三问 | 评分 |

## 3. epub vs PDF 详细对比

即便未来重新评估需要导出,以下对比说明**为什么 PDF > epub**——这样以后启动导出时,可以直接跳过 epub 评估环节。

### 3.1 设计语言保留度

| 元素 | epub | PDF (weasyprint / pagedjs) |
|---|---|---|
| 米黄背景 #F0EBDD | ⚠️ 部分(各 reader 自适应) | ✅ 完整 |
| 报刊红 #A8423B 强调色 | ✅ | ✅ |
| Inter Tight 标题字体 | ❌ 多数 reader 替换为默认衬线 | ✅ 嵌入字体 |
| 苹方 PingFang SC 正文 | ❌ 同上 | ✅ 嵌入字体 |
| JetBrains Mono 等宽数据 | ❌ 同上 | ✅ 嵌入字体 |
| 3px double border 章节分隔 | ❌ 大概率消失 | ✅ |
| inline SVG 图表 | ⚠️ 各 reader 表现不一 | ✅ 矢量保留 |
| 学科色 4px section border | ❌ 多数会丢 | ✅ |
| Mini-TOC / Tooltip / Progress | ❌ 全部丢失(epub 无 JS) | ❌ 同样丢失 |
| 章末 5 分钟回顾卡片 | ⚠️ 视觉降级 | ✅ |
| Callout(sharp/ops/you/note/story) | ⚠️ 边框 + 图标可能丢 | ✅ |

**结论**:Newsprint 报刊风的视觉资产在 epub 里**损失 ~60%**,在 PDF 里**损失 < 10%**(主要是交互组件,视觉基本保留)。

### 3.2 技术实现路径

| 维度 | epub | PDF |
|---|---|---|
| 技术栈 | 需引入 pandoc / epub.js / 自己拼装 EPUB3 zip | weasyprint(Python)/ pagedjs(JS)— **`PRINCIPLES.md` 已选好** |
| 学习成本 | 高 — EPUB3 spec、OPF、NCX、container.xml | 中 — weasyprint 文档丰富 |
| 与现有 build 流水线集成 | 需独立流程 | 可在 `_shared/build.py` 之后追加 |
| 字体嵌入 | 复杂(reader 支持差异大) | weasyprint 原生支持 |
| 调试成本 | **高** — 需在多个 reader 上验证 | 低 — PDF 渲染高度统一 |

### 3.3 用户体验

| 场景 | epub | PDF |
|---|---|---|
| 微信读书听书 | ✅ 直接支持 | ⚠️ 需转换或不直接支持 |
| 苹果图书 | ✅ | ✅ |
| 桌面阅读 | ⚠️ 需安装 reader | ✅ 系统原生 |
| 打印 | ❌ 不优化 | ✅ 原生为打印设计 |
| 在线分享 | ⚠️ 需 reader 打开 | ✅ 浏览器直接打开 |
| 离线归档 | ✅ | ✅ |

**唯一 epub 胜出的场景:微信读书听书**——而这正好是作者一人的诉求来源。

### 3.4 综合评分

| 维度 | epub | PDF |
|---|---|---|
| 设计语言保留 | 🔴 损失 60% | 🟢 损失 < 10% |
| 技术实现 | 🟡 复杂 | 🟢 项目已选型 |
| 维护成本 | 🔴 高 | 🟡 中 |
| 用户场景覆盖 | 🟡 偏窄(听书) | 🟢 广 |
| **总评** | **🔴 不推荐** | **🟡 可做但非现在** |

## 4. 替代方案 — 修 bug

### 4.1 修复方案

**Bug 1:导航定位**

- **根因**:scroll spy 监听滚动事件,代码触发 `scrollIntoView` 时也会触发滚动事件,scroll spy 把 active 改回原状态
- **修复**:加 `isProgrammaticScroll` flag,点击导航时设 true、滚动结束(`scrollend` 事件或 `setTimeout`)设 false,scroll spy 在 true 期间不更新 active
- **代码量**:十几行(在 `_shared/progress.js` 或 sticky-nav 处理逻辑里)
- **预计时间**:30 分钟 - 2 小时

**Bug 2:段落循环**

- **根因(2026-05-26 实修订正)**:**Chrome SpeechSynthesis 长 utterance(>~15s)自动停并从某段重启** 的长期已知问题。项目用 `window.speechSynthesis`(Web Speech API),**不是** HTML5 `<audio>` — PRD v1.0 初版诊断 "audio loop 误设" 是错的。代码把整章(1-3K 字,朗读 5-10 分钟)塞进一个 `SpeechSynthesisUtterance`,必中
- **修复**:在 `_shared/reader.js` 把整章文本切成段落级队列(`this._queue`),每段 < 15s,`_speak()` 拆成 `_speak()` 初始化 + `_speakNext()` 串行推进。详细 diff 见 CHANGELOG v8.7
- **代码量**:+20 行净增(4 处:构造函数 / `_speak` / `play` / `rateSel.onchange`)
- **实际工时**:已修(2026-05-26)

> **诚实留痕**:PRD v1.0 初版诊断错误,2026-05-26 实修中发现真根因是 Chrome SpeechSynthesis bug,已在 v1.1 修订。这种"主动承认 PRD 诊断错误"的痕迹本身就是文档价值——出错时不修订就是文档失信。

### 4.2 修复后预期体验

| 场景 | 修复前 | 修复后 |
|---|---|---|
| 点击导航定位 | 要点 2-3 次 | 一次到位 |
| 听书播放 | 卡在当前段循环 | 顺序播下一段、章末自动接下章 |
| 想去微信读书听的冲动 | 高 | **大概率消失** |

### 4.3 修复后的观察窗口

修复后**观察 1-2 周**,记录:

- [ ] 自己使用时是否还想跳出去到微信读书?
- [ ] 是否还有其他用户提到导出需求?
- [ ] 现有听书功能的使用频率(主观感受即可)

观察结果决定 P1 是否触发(重新评估导出需求)。

### 4.4 这类 bug 的根因复盘(留给项目长期参考)

这两个 bug 的共同模式都是**多事件源同时写一个共享状态,没有中央协调层**:

- 导航定位:`active section` 状态有"用户点击"和"用户滚动"两个写入源,代码触发的 `scrollIntoView` 让两个源打架
- 段落循环:`SpeechSynthesisUtterance` 的"播完"和业务的"播下一段"是两个状态层,中间缺少"播放队列管理器"抽象(v8.7 修后:`_queue` + `_queueIdx` 就是这个抽象)

**预防策略**(写入 PRINCIPLES.md 后续版本):

1. **任何共享状态有 ≥ 2 个写入源,必须先画状态图、定优先级**(active section / 播放进度 / Wake Lock 状态 等)
2. **引入"意图层"**:代码区分"用户主动" vs "代码触发",同事件源走不同处理路径(用 `isProgrammaticScroll` / `isAutoAdvance` 这类 flag)
3. **设计评审 / code review 第一问**:这个状态有几个写入源?≥ 2 立刻警觉
4. **测试组合操作**:不要只测"点击对不对"和"滚动对不对",要测"点击 + 同时手动滚动"——这种组合一抓一个准

底层心法:**用户操作 vs 系统自动行为冲突时,默认用户优先,系统让步**。

## 5. 结论与建议

### 5.1 决议:暂不做导出

| 方案 | 决议 |
|---|---|
| **epub 导出** | 🔴 **不做**(三问过不了 + 设计降级严重) |
| **PDF 导出** | 🟡 **暂缓**(技术路径明确,但当下无明确触发需求) |

### 5.2 行动清单

| 优先级 | 行动 | 负责 | 时间 |
|---|---|---|---|
| **P0** | 修 bug 1 — 导航定位 | 前端 | 30 min - 2 h |
| **P0** | 修 bug 2 — 段落循环 | 前端 | 1 - 2 h |
| **P0** | 修复后回归测试(组合操作:点击导航 + 同时手动滚动 / 播放 + 切段 / 锁屏切换) | 前端 | 30 min |
| **P1** | 1-2 周观察期,记录是否还有"想跳出去"的冲动 | 作者 | 2 周 |
| **P2** | 如观察结果显示仍有强烈需求,启动 PDF 导出 PRD(**不启动 epub**) | 作者 + 前端 | 待定 |

### 5.3 触发"重新评估导出"的条件

满足以下**任意一条**,可重新评估导出功能:

1. **N ≥ 5**:有 ≥ 5 个独立用户(非作者本人)明确提出导出需求,且能说出具体使用场景
2. **bug 修后仍有跳出**:作者自己在 bug 修复后,1 个月内仍 ≥ 3 次想跳到微信读书等外部 APP 听
3. **战略价值出现**:Studies 项目要做付费 / 商业化 / 跨平台分发,需要 epub/PDF 作为发行格式
4. **`PRINCIPLES.md` 更新**:项目方主动把 "epub 导出" 列入"未来扩展"清单

**任何一条不满足,这份决议持续生效**。

### 5.4 禁区(决议生效期间不做)

- ❌ 不要为单个用户(包括作者本人)的临时不爽就反复评估这个功能
- ❌ 不要因为"看到 Notion / Substack 有导出"就觉得"我们也该做"——他们的核心场景不同
- ❌ 不要在 README / 首页埋"敬请期待导出"之类的话——给用户预期 = 给自己挖坑
- ❌ 不要为了"显得功能丰富"做出 v1 后维护不动——这是 v5-v6 的覆辙

## 附录 A · 微信读书听书功能盘点(为什么作者想去)

| 功能 | 微信读书 | Studies v8.6 | 差距 |
|---|---|---|---|
| TTS 朗读 | ✅(预录人声 + AI 语音) | ✅(浏览器 Web Speech API) | 微信读书音质显著优 |
| 倍速 | ✅ 0.5x-3.0x | ✅ 9 档 0.5x-3.0x | 持平 |
| 章末接下章 | ✅ | ✅ | 持平 |
| 后台播放(锁屏继续) | ✅ APP 级 | ⚠️ Web 平台硬伤(iOS Safari 主动锁屏 30s 停 JS) | **APP 优 vs Web 硬伤** |
| 车载控制 | ✅ Media Session + CarPlay | ✅ Media Session(无 CarPlay) | 接近 |
| 断点续听 | ✅ 跨设备同步 | ✅ localStorage 本地持久化 | 单设备持平,跨设备不行 |
| 定时关闭 | ✅ + 本章末 | ✅ + 本章末 | 持平 |
| 内容范围 | 海量书 + 部分公众号 | 当前 2 份报告 | 完全不同维度 |

**关键洞察**:微信读书 > Studies 的真正差距,**只在"后台播放 / 跨设备同步 / 音质"3 项**。其他都持平。

而 3 项差距中:

- 后台播放是 **Web 平台硬伤**,Studies 没法补
- 跨设备同步需要后端,投入巨大且违反项目"零后端"原则
- 音质需要预录 MP3,**已在"主动不做"清单**

**所以即便做了 epub 导出让用户去微信读书,本质上是承认"Studies 在这 3 项追不上微信读书"——这是 Web 项目 vs APP 项目的结构性差异,不是 Studies 自身缺陷**。

承认结构性差异 > 用一个新功能去掩盖它。

## 附录 B · PDF 技术实现路径(未来若启动)

> 仅作技术储备记录,不代表近期启动。

### 技术栈选型(`PRINCIPLES.md` 已选)

- **主选**:weasyprint(Python)— 与现有 `_shared/build.py` 同语言、可集成
- **备选**:pagedjs(JS)— 浏览器端打印,所见即所得

### 集成路径

```bash
# 1. 装配 HTML(现有流程)
python3 _shared/build.py reports/<slug>

# 2. 新增:HTML → PDF
python3 _shared/build_pdf.py reports/<slug>   # 待开发

# 3. 输出
reports/<slug>/index.pdf  # 部署到 GitHub Pages
```

### 需处理的关键点

- 字体嵌入(Inter Tight / 苹方 / JetBrains Mono / Cormorant Garamond)
- 分页(章节首页强制断页,SVG 图不能被切)
- 学科色 border / callout 在打印 CSS 媒体查询下的表现
- 章末 5 分钟回顾的分页排布
- 反共识索引页是否单独成节
- 文件命名:`<slug>-vX.X.pdf` 跟 README 版本号对齐
- 文件体积控制(220K 字 + 12 张 hero 图 → 估计 5-15 MB,在 GitHub Pages 单文件 100 MB 限制内)

### 不需要处理的

- ❌ TOC 交互导航(PDF 阅读器原生支持书签)
- ❌ Tooltip(术语解释,PDF 里直接展开 inline 或丢弃)
- ❌ Mini-TOC(PDF 不需要)
- ❌ Reader 播放器(PDF 是纯阅读)

### 启动该工作的前提

参见 §5.3 触发条件。
