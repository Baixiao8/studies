故事 / 2022 年 11 月 30 日

OpenAI 在 2022 年 11 月 30 日发布 ChatGPT 时，内部预期是「研究预览，一周内可能几万人用」。两周后，用户破百万；两个月后，破一亿——成为史上增长最快的消费级产品。

但少有人注意到：ChatGPT 背后的模型 GPT-3.5，从 2020 年开始就以 API 形式开放给开发者，有一个叫 playground 的网页版可以直接对话。能力差距没那么悬殊。真正引爆 ChatGPT 的不是模型，是把「开发者输入框 + JSON 输出」换成了「对话气泡 + 流式打字」。就这么一个看上去不起眼的 UX 改动，撬开了消费级 AI 时代的大门。

0.1起源 · 一个让我们困惑的现象

这个故事告诉我们一件事：AI 产品的成败，越来越多地取决于 UX，而不是模型本身。

但当我们去翻 UX 教科书，会发现一个尴尬的事实——Don Norman《设计心理学》写于 1988 年，最近一次大修订是 2013 年。Jakob Nielsen 可用性十大启发法发表于 1994 年。Steve Krug《点石成金》写于 2000 年。

这些经典在 GUI 和触控时代被反复验证过，但它们面对 ChatGPT / Claude / Cursor / v0 / Krea 这一代产品时，开始漏风:

✕ 经典 UX 原则的「AI 时代漏洞清单」
  • 「系统状态可见」——在生成式 UI 里怎么定义？界面本身在被生成。
  • 「用户控制与自由」——在 Agent 里怎么定义？用户让 AI 自动执行时，要给多少干预入口？
  • 「错误预防」——在会幻觉的 LLM 里怎么定义？错误是必然的，不是异常。
  • 「一致性」——在每次输出都不一样的生成式产品里怎么定义？不一致才是常态。

我们需要新的语言、新的判断标准、新的设计原则。这本书就是想试着搭建这些。

0.2这本书为什么会被写出来

我长期在一线做这类 AI 设计产品——图像生成与编辑领域的 SaaS，目标用户是设计师、营销人员、创作者。本书的观察，大半来自这种日复一日的实战。

做这类产品，每天都在拍大大小小的 UX 决策：

  • 生成中的进度指示是用百分比还是「骨架闪烁」？
  • 输入框的 placeholder 应该写「请输入您的需求」还是「试试：一只在月球上喝咖啡的橘猫」？
  • AI 改出来的图用户觉得不对，应该让他继续 prompt 调，还是直接拖拽局部重绘？
  • 当模型输出失败时，是用红色弹窗告诉他「生成失败」，还是用一段灰色小字解释「这个 prompt 触发了内容安全策略，建议换成 XXX」？
  • 高级用户的 prompt 写出 1500 字，要不要给他一个「压缩」按钮？这种功能算「贴心」还是算「越权」？

这些决策能 case by case 地拍板，但始终缺一个能让所有相关角色对齐判断的「上位语言」。

  • 设计师说「这样符合 Material Design」——但 Material Design 不教你怎么处理幻觉。
  • 产品说「用户希望更多控制」——但「控制」在 AI 产品里有 5 种不同的定义。
  • 研发说「模型这次输出有 80% 的置信度」——但置信度怎么呈现给用户、要不要呈现，是 UX 问题不是工程问题。
i 这本书想解决的

它要做的，是让下一次讨论「是否在生成结果旁边显示 confidence score」这类问题时，设计、产品、研发能站在同一个坐标系里争论，而不是各说各话。

0.4本书的四个立场

写一本书没有立场就是写百科。本书想在开篇就把几个有点「锋芒」的立场亮出来——这样读者后面看到论证时不会觉得突兀，反对的读者也能从一开始就知道在反对什么。

✕ 立场 1 · UI 不会消失，但 UI 的中心地位会让位给「意图层」

对话 / Agent 不会取代 GUI，但 GUI 的设计重心会从「承载操作」转向「承载意图的表达和确认」。这是为什么 Claude Artifact、v0 这种「AI 生成 UI」会成为新主流—— UI 变成了对话的副产品。设计师的工作不再是「画完所有静态界面」，而是「设计 AI 怎么帮用户生成合适的界面」。

✕ 立场 2 · 信任比可用性更重要

在 GUI 时代，可用性是产品能不能用的底线。在 AI 时代，信任才是产品能不能用的底线——用户不信任 AI 的输出，再可用也是负担（甚至是危险）。所以 AI 产品的 UX 设计师，首先是信任设计师。每一个 confidence 显示、每一处 citation 链接、每一段 disclaimer 文案，都是在做信任设计。

✕ 立场 3 · 错误不是异常，是 UX 的核心场景

传统软件的错误处理是「边角料」——主流程跑通了再补错误页。AI 产品的错误处理是「主菜」——「AI 出错时用户怎么继续」这个问题必须在第一稿设计就回答，而不是放到 v2 再补。因为 LLM 的错误是必然的、高频的、用户无法预防的。

i 立场 4 · AI 产品的设计师，是新一代「系统工程师」

AI 产品的 UX 设计不再是「画几张稿」。它跨越前端、后端、模型、prompt、安全策略、商业策略。设计师必须懂模型的能力边界（不然会做出模型做不到的设计），必须懂 prompt 工程（不然写不好 placeholder 和 system prompt），必须懂 token 经济（不然不知道为什么某个流式效果代价昂贵）。这是设计师角色的「系统化」升级。

这四个立场会贯穿全书。如果你看完一章发现「咦这里似乎在反复说一个观点」，大概率是这四面旗在背后挥动。

每次范式跃迁 = 新输入设备（鼠标 / 触屏 / LLM）+ 新认知模型（桌面 / 卡片 / 对话）。新范式不「替代」旧范式，只是「重心位移」——但失控风险逐代上升，这是 Agent 时代最重要的新议题。

故事 / 旧金山 · 1984 年 1 月 24 日

旧金山 Flint Center。Steve Jobs 从一个布袋里拿出一台 Macintosh，按下电源键。屏幕亮起，Mac 用一个合成的声音说：「Hello, I『m Macintosh. It sure is great to get out of that bag.」

台下掌声雷动。但如果我们把镜头拉到现场观众的脸上，会发现一个细节：很多人是第一次看到屏幕上的「窗口」重叠、第一次看到「图标」代表文件、第一次看到鼠标指针在屏幕上移动。在那一刻之前，「电脑」对他们的认知模型是：黑屏 + 闪烁的命令行光标 + 一长串记不住的命令。

那个「Hello」不只是营销噱头，是一个范式宣告——电脑从此不再是「专家工具」，而是「普通人也能用的东西」。

1.1GUI 时代（1984-2007）· 把「操作」从专家手里夺过来

1980 年之前，使用电脑要记住一长串命令——cd /usr/local、grep -rn "foo" *.txt、mv file1 file2。这意味着电脑是少数人（程序员、研究员、企业 IT）的专属工具。普通办公室职员买不起也用不起。

关键突破:Xerox PARC 1973 年做出 Alto 原型机——窗口、图标、菜单、指针，俗称 WIMP。Apple 1984 年用 Macintosh 把 WIMP 商业化。Microsoft 1985 年用 Windows 跟进。

WIMP 的设计哲学，被 Don Norman 在《设计心理学》里凝结成四条原则：

VISIBILITY
可见性
把所有能做的操作摆在用户面前
AFFORDANCE
示能
按钮看起来就像按钮，让人知道能按
FEEDBACK
反馈
每次操作都有视觉 / 听觉响应
MAPPING
映射
把「想做什么」翻译成「在哪里点击」

Norman 这四条原则，后来成了几乎所有数字产品的地基，值得记一下：一是 可见性，能做的操作都摆出来让人看见；二是 示能，东西的样子要暗示它怎么用，按钮就得长得像能按；三是 反馈，每动一下都得有回应；四是 映射，把心里想做的事顺顺当当对应到该去哪儿动手。

留下的遗产

  • ✅ 「图标 + 标签」的视觉语言，至今仍是所有数字产品的基础
  • ✅ 「撤销 / 重做」的概念
  • ✅ 「复制 / 粘贴」的剪贴板隐喻
  • ✅ 「桌面 / 文件 / 文件夹」的空间隐喻

甩给下一个时代的坑

  • ❌ 鼠标 + 键盘需要一张办公桌，不能随身
  • ❌ 「学会用」还是有门槛——很多老年人到死也没学会双击 vs 单击的区别
  • ❌ 「可见性」过度堆砌导致的「功能膨胀」—— Microsoft Word 2003 的菜单层级深到没人能找到自己想要的功能，这是 GUI 时代最有名的反面教材
i GUI 时代的最大一课

好的设计是把复杂藏在简单背后，而不是把所有复杂都摆到台面上。但这个教训直到下一个时代才被真正贯彻。

1.2触控时代（2007-2022）· 手指 + 上下文 = 普及的极限

电脑虽然「普及」了，但它绑在桌子上、绑在键鼠上、绑在「专注一件事」的工作模式上。生活场景里的大量需求（地图、拍照、付钱、聊天、看车）都没法在 PC 上自然完成。

关键突破:2007 年 1 月 9 日，Jobs 在 Macworld 拿出第一代 iPhone。最大的创新不是「放进口袋」——之前已经有 Palm、Windows Mobile 这些。最大的创新是用手指直接操作，并且抛弃了实体键盘。

触控时代的设计哲学有几个新概念，Apple HIG 把它们系统化了：

DIRECT
直接操作
手指就是指针，没有中间设备
GESTURES
手势
捏合 / 滑动 / 长按 · 新交互词汇
CONTEXT
上下文感知
GPS / 加速度 / 相机 · 设备「知道」用户在哪
MINIMAL
极简界面
屏幕小，只摆「现在最重要的东西」

iPhone 还附带了一个革命：App Store 模型。它把「软件」从「安装包」变成「应用」，让「一个功能 = 一个图标」成为可能，也让小团队能直接触达全球用户。

留下的遗产

  • ✅ 「下拉刷新 / 滑动删除」这些手势成为通用语言
  • ✅ 「通知中心」作为多任务的承载形式
  • ✅ 「权限弹窗」作为隐私的最后一道防线（虽然没人真的看）
  • ✅ 全屏化、卡片化、极简化的视觉趋势
  • ✅ 「深色模式」作为屏幕越来越大的副产品

甩给下一个时代的坑

  • ❌ 输入还是麻烦——手指打字慢、容易错、长文输入是灾难
  • ❌ 「信息过载」在小屏更严重——推送、红点、tab 数字像永远关不掉的水龙头
  • ❌ 「意图表达」还是要「自己找路径」——想订一张明天去上海的高铁票，需要打开 12306 → 输入起点终点 → 选时间 → 选车次 → 选座位 → 付钱，每一步都是用户在操作
  • ❌ 跨 APP 的「任务流」几乎无法连贯—— APP 之间是孤岛，分享一张图都要走系统级 share sheet
i 触控时代的最大一课

最好的交互是没有交互——直接用手指点，没有中间设备。但即便如此，用户还是在「操作」，只是操作变得更直接了。真正的解放，是让用户表达意图，让系统去操作。这就是 Agent 时代的承诺。

1.3Agent 时代（2022—）· 意图直接执行

触控时代解决了「在哪里」和「用什么」，但没解决「为什么」和「做什么」。用户面对一个 APP，仍然要把自己的真实意图（「我想给老妈寄一束生日花」）翻译成一系列具体操作。这个翻译过程，就是 GUI 和触控时代都没消除的「摩擦」。

关键突破:2022 年 11 月 30 日，ChatGPT 发布。它的革命性不在「对话」—— Siri、小冰早就能对话。它的革命性在于：第一次让自然语言成为可靠的交互层。

不是「假装能理解你」，而是真的能理解。不是「给你一堆链接让你自己看」，而是直接给答案。不是「按一两个固定命令」，而是任何你能说出来的，它都能尝试。

Agent 时代的设计哲学正在被多家公司争夺定义权，但可以看到几个共识：

INTENT
意图层
用户说出目标，系统拆解步骤
COLLAB
协作
用户和 AI 是合作关系，不是命令-执行
EXPLAIN
可解释
AI 要让用户知道「为什么这么做」
INTERRUPT
可中断
用户能随时介入、修改、取消

Anthropic 在 2024 年 10 月发布 Claude Computer Use—— AI 可以直接操作鼠标键盘。这是 Agent 时代的一个里程碑：计算机变成了 AI 的工具，AI 变成了用户的延伸。

正在出现的新设计语言

  • 🔄 流式输出(Streaming)——用户能看着 AI 一个字一个字「想」出来
  • 📝 Artifact / Canvas—— AI 生成的内容有自己的容器，可被持续修改
  • 🎯 Slash Commands——在自然语言里嵌入快速指令（/edit、/translate）
  • 🪜 Step-by-step 反馈—— Agent 执行多步任务时，每一步都报告「我在做什么」
  • ⏸️ Pause & Edit——用户随时可以暂停 AI、修改它的下一步
  • 🔖 Memory & Projects—— AI 记得过往对话和用户偏好
✕ Agent 时代还没解决的问题（这就是 Part II 要讨论的）
  • 信任—— AI 会瞎编，怎么让用户知道哪些能信？
  • 控制—— AI 自动做了 50 步，用户只看了 5 步，怎么保证用户没失控？
  • 错误—— AI 改坏了，怎么撤销？怎么从灾难里恢复？
  • 上下文——用户和 AI 聊了 100 条消息，AI 还能记住第 3 条说的偏好吗？
  • 边界—— AI 该自己做哪些事、该问用户哪些事？

1.4三个时代横向对比

把三个时代摆到一块儿比，能看到一些深层规律。

维度 GUI 时代 触控时代 Agent 时代
输入设备鼠标 + 键盘手指 + 传感器自然语言 + 多模态
核心隐喻桌面（Desktop）卡片（Card）对话（Conversation）
用户角色操作者触摸者协作者
系统角色工具助理队友
设计师工作画界面设计交互流设计协作机制
错误的位置边角料通知中心主菜（必然发生）
学习成本高（需要培训）低（几分钟）极低（说人话就行）
失控风险低中高（AI 在自动执行）

这一比，有两件事值得拎出来。一是用户和系统的关系在变：从 GUI 时代你是操作者、机器是工具，到触控时代你是触摸者、机器是助理，再到 Agent 时代，你成了协作者，机器更像队友。二是错误的待遇在变——过去它是边角料、是事后补救，到了 Agent 时代，AI 会自动跑很多步，出错从此是必然发生的事，得当成正餐来设计，而不是擦屁股的附属品。

这里最该划重点的，是 失控风险逐代往上走。这是 Agent 时代最重要的 UX 议题：当用户把决策权交给 AI 时，怎么保留「我还能拿回来」的安全感？

✕ 经典原则的「驾驶座假设」在 Agent 时代崩了

Don Norman 那一套原则，假设用户始终在驾驶座上。而 Agent 时代，用户经常坐到了副驾，AI 在开车。副驾的座椅怎么设计、刹车按钮放在哪、什么时候要让副驾接管？这是全新的 UX 议题。我们会在第 14 章系统重写这些原则。

1.5范式跃迁的三条底层规律

最后总结三条贯穿三个时代的底层规律，作为后续章节的认知地基。

规律 1 · 每次范式跃迁的驱动力 = 新输入设备 + 新认知模型

时代新输入设备新认知模型
GUI鼠标桌面隐喻
触控触摸屏卡片 + 上下文隐喻
AgentLLM对话 + 协作隐喻

只有设备升级（如智能手表多了陀螺仪）而没有认知模型升级，跃迁不成立——这就是 Apple Watch 没有撑起「穿戴时代」的核心原因，它只是 iPhone 的延伸而不是新范式。

规律 2 · 范式跃迁不是「取代」，是「重心位移」

PC 没死，触控时代仍有人用 PC 工作；GUI 没死，触控产品里也有大量 GUI 元素；触控也不会死，Agent 时代我们依然要用手机。新范式的核心改变，是「哪些事情现在主要用新方式做」。

文字内容创作的「主战场」已经从 Word 移到 ChatGPT / Claude，但 Word 没消失——它仍然是写正式合同、写论文、写需要复杂排版内容的最佳选择。

✓ 这给我们一个判断标准

不要为了「AI 化」而 AI 化。判断一个功能要不要 AI 化的标准是：用户在做这个功能时，最大的痛点是不是「意图到操作的翻译」？如果是，AI 化有价值；如果不是（比如就是要精确点击某个按钮），AI 化反而是负担。

规律 3 · 每个时代的设计师都把上一代的「遗产税」当成新的设计议题

  • GUI 时代的设计师没料到「功能膨胀」会成为触控时代的反面教材
  • 触控时代的设计师没料到「信息过载」会让 Agent 时代不得不重新思考「通知」这个概念
  • 我们这一代 AI 产品设计师，正在制造一些未来的「遗产税」——可能是「prompt 焦虑」，可能是「AI 依赖症」，可能是「信任侵蚀」，可能是「创造力萎缩」

为这个未来负责，是这本书最长远的动机。

五分钟回顾

范式跃迁要点

  1. 三个时代:GUI(1984-2007) → 触控(2007-2022) → Agent(2022-),每个时代有一个'Hello'时刻(Macintosh / iPhone / ChatGPT)。
  2. 每个时代的核心矛盾:GUI 解决专家工具民主化 · 触控解决随身使用 · Agent 解决意图直接执行。
  3. 失控风险逐代上升:GUI 低(每步手动)→ 触控中(手势误触)→ Agent 高(AI 自动执行)。这是 AI 时代最重要的新议题。
  4. 跃迁不是替代是重心位移:Word 没死,只是文字创作主战场移到 ChatGPT。不要为了 AI 化而 AI 化。
  5. 底层规律:每次跃迁 = 新输入设备 + 新认知模型;旧时代的'遗产税'变成新时代的设计议题。

▼ ELEVATOR PITCH

AI 产品的 UX 不是「传统 UX + AI」，而是一套需要重新定义的设计语言。Don Norman / Nielsen 启发法没过时，但它们面对的对象变了——验证机制、错解退路、生成式 UI、多步介入、记忆资产，这 5 件事在 GUI 时代都不存在。

故事 / 两个搜索框

打开 Google，搜索「明天上海会下雨吗」，你会得到 10 条蓝色链接，最上面通常是中央气象台或天气网，你点进去看一眼天气预报，自己得出结论。

打开 Perplexity，搜同样的问题，你会得到：「根据中央气象台 5 月 25 日预报，上海明天（5 月 26 日）多云转阴，傍晚有阵雨可能，气温 18-24°C，建议带伞出门。[1][2][3]」下面是 3 个引用源小卡片。

这两个产品都叫「搜索引擎」，但它们的 UX 不是同一种 UX。Google 假设你是「研究者」——我给你链接，你自己判断。Perplexity 假设你是「提问者」——你想要答案，我直接给你，附带证据。

从 UX 的角度看，从 Google 到 Perplexity 至少多了 5 个全新的设计议题：

  • 答案的可信度怎么表达？（链接列表是「我不知道你看吧」，直接答案是「我知道请信我」，后者风险大得多）
  • 引用怎么呈现？（不引用就是黑盒，引用太多就抢眼球）
  • 答案错了怎么办？（链接错是「用户挑的链接错」，直接答案错是「系统的责任」）
  • 多轮追问怎么承接？（Google 每次搜索独立，Perplexity 要记住上一次问什么）
  • 答案的「温度」和「立场」怎么设计？（「建议带伞出门」是一种立场，Google 永远不会给你这种建议）

这就是这一章要回答的问题：AI 产品和传统产品的 UX，到底差在哪里？下面用 5 组对比，逐一展开。

2.1对比 1 · Gmail Smart Compose vs GitHub Copilot

两个产品乍看一样——都是「灰色字预测，Tab 接受」。但 UX 的本质完全不同。

维度 Smart Compose(2018) Copilot(2021)
预测长度几个词几行到几十行
用户验证成本低（一眼能看完）高（需要读懂代码逻辑）
错了的代价低（多写几个字而已）高（可能引入 bug、安全漏洞）
预测的语义客套话（高度可预测）业务逻辑（不可预测）
用户的角色写信人，AI 是秘书程序员，AI 是结对编程伙伴
✕ 特质 1 · 当 AI 输出超过「用户能秒判」的复杂度，UX 需要重新设计「验证机制」

Smart Compose 的「灰色字 + Tab」完全够用，因为用户一眼能判对错。Copilot 也是「Tab 接受」，但这个 UX 在长代码补全时其实是有问题的——它鼓励用户不读就接受，这是 Copilot 早期被批评的核心点。

后续 Copilot 推出了 Copilot Chat、Inline Chat、PR Review——都是在补「验证机制」这个洞。

2.2对比 2 · Photoshop 抠图 vs Photoroom 一键去背

PS 选择魔棒 → 设置容差 → 点击背景 → Shift+点击多选 → 边缘羽化 → 反选 → Delete。学过的人 30 秒搞定，没学过的永远学不会。Photoroom 上传图 → 自动识别主体 → 一键去背。3 秒搞定。

这看起来是「功能更强」，但 UX 上发生的是更深的变化：

  • PS 的 UX 假设：用户知道自己要做什么操作（「我要用魔棒选区域」），软件提供精准工具。
  • Photoroom 的 UX 假设：用户只知道自己要什么结果（「我要去背」），AI 帮用户做出操作。

这两种 UX 哲学的差距，是「工具型软件」和「意图型软件」的差距。但意图型软件也有新的 UX 难题：

  • 如果 AI 识别错了主体怎么办？（PS 至少错了你能立刻看到选区重新选，Photoroom 错了你只能重传）
  • 用户想精细调整边缘怎么办？（PS 有羽化收缩扩张，Photoroom 早期只有「成功 / 失败」两个状态）
  • 批量处理怎么办？（PS 录制动作可以批量，Photoroom 需要重新设计「批量 AI 处理」）

Photoroom 后来加了「细化边缘」、「手动涂抹保留 / 删除」等功能——是在用户「意图被错误理解」时给一个回退通道。

✕ 特质 2 · 必须为「AI 理解错了用户意图」准备一个体面的退路

这是从「工具型」到「意图型」软件最大的 UX 升级议题——意图被错解时的优雅降级。

2.3对比 3 · Notion 模板 vs Notion AI

传统 Notion：你想做会议纪要，打开模板库选「Meeting Notes」套用。模板是死的，永远是那个结构。Notion AI：你输入「/ai 帮我做一个产品评审会的纪要模板，参与方包括设计师、产品、工程师」，Notion AI 现场生成一个适合你具体场景的模板。

差距在于「模板」这个概念本身被解构了：

  • 传统 Notion 的「模板」是预先存在的资产，由模板作者制作
  • Notion AI 的「模板」是按需生成的实例，每次都不一样

这带来一系列 UX 新问题：

  • 生成出来的东西，用户怎么保存复用？（这次生成的好，下次还能用吗？）
  • 生成的版本之间怎么对比？（用户可能让 AI 重生成 3 次，怎么选最好的？）
  • 生成的内容能被进一步编辑吗？（如果能，被编辑后的版本算谁的？算 AI 的还是算用户的？）

Notion AI、Claude Artifact、v0 都在面对同样的问题——这就是第 5 章要专门讲的 「生成式 UI」。

✕ 特质 3 · UI 元素从「被设计的资产」变成「被生成的实例」

需要全新的「生成 / 修改 / 保存 / 复用」四态设计。

2.4对比 4 · 滴滴打车 vs Cursor Composer

滴滴：点「叫车」→ 系统派单 → 司机接单 → 司机到达 → 上车。每一步都有明确的状态显示，每一步用户都能取消。Cursor Composer：你说「帮我把这个组件的状态管理从 useState 改成 Zustand，并且更新所有使用它的地方」，Composer 开始执行：

i Composer 的多步执行典型流程
✓ 读取 components/UserProfile.tsx
✓ 读取 components/UserList.tsx
✓ 读取 store/userStore.ts
⠹ 修改 components/UserProfile.tsx (使用 Zustand)
⠹ 修改 components/UserList.tsx
⠹ 创建 store/userStore.ts
⠹ 修改 app.tsx 注册 store

每一步用户都能看，但用户能「取消」吗？取消的成本是什么？

如果 Composer 改到第 5 步时用户想取消，前 4 步已经改了的代码怎么办？回滚？保留？滴滴每一步是离散的、可逆的。Cursor Composer 每一步是连续的、相互依赖的——一旦开始改，「回到原状」不是按一个按钮就能实现的。

✕ 特质 4 · Agent 自动执行多步任务时，需要全新的「中断 / 回滚 / 介入」设计

不能套用传统的「取消」按钮。Cursor 的解法是接入 git——每次 Composer 执行前自动 stage 当前状态，用户可以 git reset 回去。这是把「回滚」外包给了 git。但其他 Agent 产品（让 AI 帮你回邮件、订机票的 Agent）没有 git 这个安全网，怎么办？这是第 8 章要专门讨论的。

2.5对比 5 · 百度搜索 vs 豆包对话

百度：每次搜索都是独立事件，今天搜「上海餐厅推荐」，明天再搜，结果差不多。豆包（或 ChatGPT / Claude / Kimi）：你今天告诉豆包「我是素食主义者」，明天问「推荐上海餐厅」，豆包会自动过滤掉非素食店。

这是「记忆」的引入。但记忆带来的 UX 问题，前所未有：

  • AI 记住了什么？用户能看到吗？（如果 AI 记错了，用户怎么改？）
  • 记忆什么时候触发？（用户上次说的话，下次还算数吗？什么时候过期？）
  • 记忆能被分享吗？（如果换设备登录，记忆能跟过去吗？）
  • 隐私怎么办？（用户敏感信息，AI 记不记？记多久？）

ChatGPT 在 2024 年推出 Memory 功能时，专门做了一个「记忆面板」——用户能看到 AI 记住了哪些，可以删除、修改。这是把「AI 的记忆」做成了可见、可控、可审计的资产，而不是黑盒。

✕ 特质 5 · 「上下文 / 记忆」从无变成核心资产

需要专门设计可见性、可控性、可移植性。这不只是「加个记忆功能」那么简单，是一个全新的 UX 议题。

故事 / 纽约 · 2023 年 6 月

纽约律师 Steven Schwartz 用 ChatGPT 准备了一份法律文书，引用了 6 个判例支持他的论点。文书提交给法院后，对方律师查不到这些判例。法官也查不到。原因是—— 这 6 个判例全都是 ChatGPT 编出来的。案件名、法官名、判决日期、引用编号，一切看起来都很真实，但全部不存在。

Schwartz 被罚款 5000 美元，案件被驳回，职业生涯严重受损。事后他向法庭解释：「我以为 ChatGPT 是一个超级搜索引擎。」这件事被全球媒体报道，成了「AI 幻觉」问题的标志性事件。

但对我们做产品的人来说，更值得问的问题是：ChatGPT 当时的 UX，有没有做任何努力让 Schwartz 意识到「这些判例可能是编的」？答案是：几乎没有。Schwartz 案是一次 UX 失败，不只是用户的轻信。

3.1什么是幻觉？为什么 LLM 必然会幻觉？

幻觉（Hallucination）是 AI 领域的专门术语：模型生成了看似合理但实际错误或不存在的内容。幻觉不是「bug」，是 LLM 的结构性特征。

原因是：

  1. LLM 的本质是「下一个 token 的概率分布」——它不「知道」事实，它只「生成」看起来像事实的内容
  2. 训练目标是「流畅」而不是「正确」——模型被优化成「说人话」，但「人话」不等于「对的话」
  3. 训练数据不可能覆盖所有事实——模型不知道的事，它会「猜」，猜出来的东西通常看起来很真
  4. 没有「我不知道」的强训练信号——早期 LLM 几乎从不说「我不知道」，因为训练数据里也很少有人这么说

幻觉有几种典型类型，做产品时要分别处理：

类型 例子 用户检测难度
事实幻觉编造历史事件、人物经历中（可以 Google 验证）
引用幻觉编造论文、判例、新闻链接低（链接打不开就露馅）
数字幻觉报错统计数字、价格、日期高（用户经常没意识到）
逻辑幻觉推理链条断裂、自相矛盾极高（需要专业知识）
指令幻觉编造软件功能、API 接口中（试了不存在就知道）

这几类里，真正值得做产品的人警惕的，是用户自己最难抓出来的那两种。引用幻觉其实最善良——编的论文、判例、链接，一点开打不开就露馅了。事实幻觉也还好，随手能查证。但数字幻觉（报错一个统计值、一个价格）和逻辑幻觉（推理链悄悄断掉、自相矛盾）往往溜得过去，前者用户根本没意识到要核对，后者得有专业知识才看得出。所以信任 UX 的力气，优先该花在这两类上。

i 对图像生成类产品而言

以某国产 AI 设计平台这类图像生成工具为例，「幻觉」表现为生成图像和用户意图不符——要求「5 个人围坐」画出 6 个人，要求「红色裙子」画成粉色。机制略有不同，但 UX 议题一致：用户怎么知道模型这次是否靠谱？

3.2信任的三个层次

做信任 UX，要分清楚「信任」是分层的，不是一个单一概念。

LAYER 1
任务级
「这次输出对吗？」
LAYER 2
系统级
「这个产品大概率靠谱吗？」
LAYER 3
模型级
「AI 这个东西能信吗？」
∑
三层叠加
缺任一层都会「信任崩塌」

分层具体是这么分的。第一层是任务级，回答的是「这次输出到底对不对」；第二层是系统级，问的是「这个产品整体大概率靠不靠谱」；第三层是模型级，问的是「AI 这个东西本身能不能信」。三层是叠加的，缺了任何一层，用户的信任都会整个崩掉。接下来这几个产品，各自主攻的就是其中一层。

不同产品的「信任战场」不同：

  • Perplexity 的战场在 Layer 1——每次回答都要让你能立刻验证
  • ChatGPT 的战场在 Layer 2——让你建立「它平时靠谱，偶尔出错」的整体印象
  • Apple Intelligence 的战场在 Layer 3——通过「在设备本地运行」「不上传云端」建立 AI 技术本身的信任感
✓ 图像生成类产品的信任主战场

像某国产 AI 设计平台这类工具，主战场在 Layer 1 + Layer 2：每次生图都要让用户能快速判断好坏，并且让「该平台大部分时候出图质量稳定」沉淀成产品口碑。Layer 3 主要靠行业大势，不是这类产品的主战场。

3.3表达可信度的 5 种 UX 模式

如果「幻觉是必然的」，那 UX 的核心任务就变成了：帮用户区分「哪些输出可信、哪些不可信」，并把这个判断的成本降到最低。主流 AI 产品在过去 3 年发展出了 5 种典型模式。

模式 1 · Confidence Display（置信度显示）

直接把模型的置信度数字 / 颜色 / 强度展示出来。

  • 正面案例:GitHub Copilot 早期会用灰色字代表「低置信」补全，黑色字代表「高置信」；Adobe Generative Fill 会给生成的图加「细节饱满度」提示；一些医疗 AI 会显示「诊断置信度：92%」。
✕ Confidence 显示的三个坑
  • 直接显示置信度数字（「95%」）对普通用户没有意义—— 95% 是高是低？
  • 用户会过度信任高置信结果——心理学上叫「automation bias」
  • 模型的「置信度」经常和「准确率」不挂钩—— LLM 经常对错的事很自信

模式 2 · Citation（引用 / 溯源）

把生成内容的来源标注出来，用户可以点进去验证。

  • 正面案例:Perplexity 每个段落后面跟 [1][2][3]，每个数字 hover 出来源；Claude 在做 web 搜索时会在回答里嵌入引用链接；Notion AI Q&A 会显示「答案来自这 3 个文档」，并 highlight 出处。
✕ Citation 的三个坑
  • 引用本身也可能是幻觉（早期 Bing Chat 会引用不存在的网页）
  • 引用源不可信（引用了一个营销稿当作权威）
  • 引用太多让答案碎片化（每句话都带 [1]，读起来累）

关键设计判断：引用的目的不是「显得权威」，而是让用户能快速验证。如果点击引用要 3 步才能看到原文，引用就是装饰；如果一 hover 就出原文片段，引用才是真功能。Perplexity 的引用 hover-card 是行业标杆。

模式 3 · Source Transparency（来源透明）

不只是「有引用」，而是让用户了解 AI 在用什么知识、什么时候的知识。

  • 正面案例:ChatGPT 在回答中明确说「我的训练数据截止 2024 年 4 月」；Claude 在被问到时间敏感问题时主动说「我没有实时数据，建议你查 X」；Perplexity 会标记每个引用源的「日期」。

关键判断：透明度不是「自证清白」，是帮用户调整使用预期。当用户知道 AI 不掌握 5 月之后的事，他自然不会用 AI 查最新新闻。

模式 4 · Disclaimer（免责声明）

在易出错的输出旁边加警告语。

✕ 典型反面案例 · 永远在底部的废话 disclaimer
  • 永远在最下面有一行小字 「ChatGPT can make mistakes. Check important info.」——用户已经免疫，等于没说
  • 医疗 AI 默认加「本回答不构成医疗建议，请咨询医生」——没人看
  • 正面案例:Claude 在被问到法律 / 医疗等高风险问题时，会在回答开头明确说「这是一般性信息，不是专业建议」；Perplexity 在不确定的事实旁边会用问号图标；一些金融 AI 在给投资建议时，会强制弹窗「你要继续吗？」而不是只放一行小字。

关键判断：免责声明的位置 + 频率 + 视觉权重决定它有没有用。永远在底部用灰色小字 = 没用。在高风险输出的旁边、用差异化视觉、强制看见 = 有用。

模式 5 · Verify-before-Act（执行前验证）

当 AI 要执行一个不可逆操作（发邮件、转账、删除文件），强制要求用户最后确认。

  • 正面案例:Claude Computer Use 在执行任何「破坏性」操作前会停下来问用户；Cursor Composer 改完代码后默认要求用户点 「Accept」 才生效；Notion AI 在生成内容后默认进入「预览状态」，用户主动点「插入」才进入正文。
✕ Verify 的两个反面
  • 对低风险操作也要求确认 = 让用户疲劳
  • 确认弹窗变成「OK」按钮的肌肉记忆，用户不读就点

关键判断:Verify 的成本要和操作的不可逆程度匹配。低风险用 inline 预览，高风险用强制确认，灾难性用二次输入（比如让用户敲「DELETE」才能删库）。

3.5警惕「过度信任 UX」

最后必须说一个反面议题：信任 UX 不是越多越好。如果每次输出都让用户验证、每次都弹引用、每次都强制确认——用户会疲劳，进入「机械点击 OK」状态，反而让 UX 失效。

平衡的关键是根据「输出的可逆性 × 错误成本」来分级:

可逆性错误成本信任 UX 力度示例
高低不需要特别 UX推荐表情符号
高中inline 提示写邮件草稿
低中引用 + 预览态生成报告
低高强制确认 + 引用自动发邮件给客户
不可逆极高二次输入 + 强警告删除账户、转账

这个分级可以顺着一条线来记：操作越容易撤回、错了越没代价，信任 UX 就越该退到后台。像推荐个表情符号，改起来零成本，什么都不用加。往上一档，写邮件草稿这种，给个 inline 小提示就够。再往上，生成报告改起来麻烦了，配上引用加预览态。到了自动把邮件发给客户这种发出去就收不回的，得强制确认再加引用。最狠的一档是删账户、转账这类灾难性又不可逆的——这时候就该逼用户亲手敲一遍确认字样，让他没法靠肌肉记忆一路点过去。一句话，验证的成本要跟操作的不可逆程度对齐。

◐ 对这类产品的简短建议（详见第 15 章）

图像生成的「错误成本」中等（生成不好就重生成，时间和点数浪费但没灾难），可逆性中等（生成出来不喜欢可以重新生成，但可能丢了灵感）。所以这类产品的信任 UX 应该集中在 inline 预览 + 轻量「重生成」成本提示，而不是每张图都让用户走一道确认流。

详细应用建议留到第 15 章「这类产品体检」章节展开。下一章我们讨论一个紧密相关的议题—— 可控性悖论：当 AI 越能干，用户越焦虑「我还掌握着这件事吗」。信任和控制是 AI 产品 UX 的两条主腿，缺一不可。

五分钟回顾

信任 UX 要点

  1. 幻觉是 LLM 结构性特征,不是 bug——Schwartz 律师案不只是用户轻信,也是 UX 失败。
  2. 信任分三层:任务级(这次对吗)/ 系统级(平时靠谱吗)/ 模型级(AI 能信吗),要分别设计。
  3. 5 种信任 UX 模式:Confidence Display · Citation · Source Transparency · Disclaimer · Verify-before-Act。每种都有典型坑。
  4. 行业最佳实践:可点验引用(Perplexity)+ 主动说不知道(Claude)+ 关键时刻确认(Computer Use)+ 不堆废话 disclaimer + 可审计记忆(ChatGPT)。
  5. 警惕过度:按'可逆性 × 错误成本'分级,避免用户麻木——堆 disclaimer 不如关键时刻一次强提示。

可控性悖论：给用户多大缰绳？ The Control Paradox

可控性不是「给用户控制」或「不给」的二元选择，而是启动 / 过程 / 边界 / 撤销 / 学习五个维度上的独立设计。AI 介入度越高，用户控制感越低，但产品的「想象力」越大——这是 AI 产品设计的核心张力。

故事 / 大西洋上空 · 2009 年 6 月 1 日

2009 年 6 月 1 日，法航 447 航班从里约飞往巴黎，在大西洋上空消失。残骸经过两年才被打捞起来，黑匣子还原了事发经过。

飞机的「皮托管」（测速装置）被冰晶堵塞，自动驾驶因数据异常自动断开，把操控权交给了飞行员。从这一刻开始的 4 分钟内,3 名飞行员持续做出错误操作，导致飞机失速坠海，228 人全部遇难。

事后调查报告指出了一个反直觉的结论：正是因为他们平时太依赖自动驾驶，反而在自动驾驶突然交还操控权时手足无措。这个事故现在被 HCI 学界（如 Lisanne Bainbridge 1983 年的经典论文 Ironies of Automation）反复引用——自动化越强，人在关键时刻的失控就越严重。这就是「可控性悖论」。

4.1可控性悖论的两难

AI 产品的设计师，每天都在两难中拉锯：

  • AI 做太少 → 用户问「它有啥用？还要我自己一步步操作」（鸡肋）
  • AI 做太多 → 用户问「它怎么自己做的？我能信吗？要是错了怎么办？」（失控）

这条线的拉锯，决定了产品的 UX 形态。

产品 AI 介入度 用户控制感
Gmail Smart Compose极低（只补几个词）高（一眼能看）
GitHub Copilot Inline中（补一段函数）中（要读才能判）
Cursor Composer高（自动改多个文件）低（有 git 兜底）
Cursor Agent (background)极高（自己开始干活）极低（git 兜底 + diff review）
Devin（演示版）接近 100%（端到端完成）几乎为 0
✕ 一条规律

AI 介入度越高，用户控制感越低，但产品的「想象力」越大。Devin 在 2024 年因 Demo 太炸裂引发轰动，但实际产品落地一直困难——核心原因之一就是用户在「完全失控」状态下不知道该信任 AI 多少。

4.2可控性的 5 个维度

把「可控性」拆开看，至少有 5 个独立的维度。做 AI 产品的设计师，要在这 5 个维度上独立做决策——不是「给用户控制权」或「不给」这种二元判断。

维度 1 · 启动控制（Trigger Control）

问题:AI 什么时候开始介入？由用户主动触发，还是 AI 主动出现？

  • 用户主动触发：用户必须按按钮 / 输入命令 / 选菜单才会激活 AI（如 Cursor Composer、Notion 的 /ai）
  • AI 主动出现:AI 在监测到合适时机时自己冒出来（如 Gmail Smart Compose）
  • Always-on:AI 始终在后台，随时介入（如 Grammarly 检查所有输入）
✓ 经验

低能力 AI 适合「always-on」（错了影响小），高能力 AI 适合「主动触发」（让用户明确表达「我现在要 AI 帮忙」）。Cursor 的成功部分源于它把 Tab 补全做成 always-on（低门槛），把 Composer / Agent 做成主动触发（高授权）——不同能力对应不同触发方式。

维度 2 · 过程控制（Process Control）

问题:AI 执行的过程中，用户能不能干预？

  • 完全不能干预:AI 开始执行就锁死，做完才给结果（如早期一些 image generator）
  • 可中断：用户能随时停止（如 ChatGPT 的「Stop generating」）
  • 可修改：用户能修改 AI 的下一步（如 Cursor Composer 执行中可以发送新指令）
  • 可逐步审批：每一步都暂停等用户确认（如 Claude Computer Use 的 step-by-step mode）
✕ 至少要给「中断」

即便是「完全不能干预」的模型（如纯生图），也应该让用户能中断生成、不浪费 credit。ChatGPT 早期没有 Stop 按钮是一个典型的设计漏洞，2023 年后才补上。

维度 3 · 边界控制（Scope Control）

问题：用户能不能定义「AI 能做哪些事、不能做哪些」？

  • 写死的边界:AI 永远不能做某些事（如 Claude 拒绝生成有害内容）
  • 用户配置的边界：用户能设置 AI 的行为范围（如 ChatGPT 的 Custom Instructions、Cursor 的 .cursorrules）
  • 临时声明的边界：每次对话开始时说「不要做 X」
✓ 经验 · 三层边界

边界配置应该有「全局 + 项目 + 临时」三层。全局是用户偏好（「我总是用 TypeScript」），项目是项目级规范（「这个 repo 用 React Native」），临时是单次对话的特殊要求（「这次别加注释」）。Cursor 用 .cursorrules 文件 + 对话级 system prompt 实现了这一点，这是值得学习的边界控制模式。

维度 4 · 撤销控制（Reversibility Control）

问题:AI 做完之后，用户能不能撤销？撤销到哪一步？

  • 不可撤销:AI 做了就是做了（如发出去的邮件）
  • 粗粒度撤销：能撤回到 AI 开始执行之前（如 git reset）
  • 细粒度撤销：能撤回到某一步之前（如 Cursor 可以 reject 单个 diff 而保留其他）
  • 版本化：每次 AI 输出都保存为一个版本，用户能在版本间切换（如 ChatGPT Canvas 的版本历史）
✕ 没有撤销，用户就不敢用

这是为什么 Cursor 的成功有一部分要归功于 git——它免费提供了一个强大的撤销机制。没有 git 兜底的 AI 产品（比如让 AI 帮你回邮件、订机票），必须自己造一套撤销系统，否则用户不敢授权。

维度 5 · 学习控制（Learning Control）

问题:AI 会不会从用户的使用中「学坏」？用户能不能控制 AI 学什么？

  • 完全不学习:AI 是无状态的，每次对话都从零开始（如默认的 ChatGPT）
  • 静默学习:AI 在背后学习用户偏好，用户看不到学了啥（早期 Apple Siri）
  • 可见学习:AI 学到的东西用户能看、能改（如 ChatGPT Memory 面板）
  • 明示学习:AI 每次学新东西都问用户「我可以记住这个吗？」
✓ 经验 · 学习必须可见可控

ChatGPT 在 2024 年推出 Memory 功能时被很多人批评「隐私担忧」，OpenAI 的应对是做了一个完全透明的 Memory 面板——用户能看到 AI 记住了什么，能逐条删除。这是把「学习黑盒」做成了「学习白盒」，是值得借鉴的模式。

4.3典型解法对比 · 行业怎么解可控性？

解法 1 · Cursor 的「三段式介入」

Cursor 把 AI 介入分成 3 个梯度，每个梯度有不同的可控性设计：

梯度 触发方式 过程控制 撤销
Tab 补全always-on,AI 主动建议不可中断（生成快）Esc 拒绝 / 不接受
Composer用户主动触发（Cmd+I）执行中可发新消息git diff + accept/reject 每个 hunk
Agent用户主动触发（Cmd+I 切换）可中断、可对话修改git diff + 完整 review

精髓：用户能在 3 个授权强度间无缝切换。低风险任务用 Tab，中风险用 Composer，高风险（或想偷懒）用 Agent。设计师不替用户选择「AI 该多自主」，而是把选择权交给用户。

解法 2 · Claude Artifact 的「编辑 - 预览分离」

Claude 在生成代码、网页、图表等「复杂内容」时，会把内容放到一个独立的 Artifact 面板里。这个设计有几个可控性好处：

  • 用户可以只看不接受—— Artifact 默认不会「插入到对话主线」
  • 用户可以反复修改——让 Claude「改成蓝色 / 加一个按钮 / 字体放大」，每次都更新同一个 Artifact
  • 用户可以版本对比—— Artifact 保留历史版本，能切换查看

精髓：把「AI 输出」和「用户的工作空间」明确分开，输出永远在「沙盒」里，用户主动决定要不要接受。

解法 3 · Krea 的「实时参数滑块」

Krea 在生成图像时，提供了大量实时调节滑块——风格强度、相似度、guidance scale 等。用户可以一边看 AI 出图、一边拉滑块调节，AI 实时跟进生成新图。

精髓：把「AI 的内部参数」暴露给用户，让用户感受到「我在驾驶这个模型，不是模型在驾驶我」。这种 UX 适合创意类场景（用户本来就想探索），不适合任务完成类场景（用户只想要结果）。

解法 4 · 自动驾驶给 AI 产品的教训

特斯拉、Waymo、小鹏的辅助驾驶系统，对 AI 产品的可控性设计有一个核心启示：控制权的交接是危险的。

✕ 两个危险时刻
  1. AI 主动放手时（任务做不下去了）——必须告诉用户「我做到第 X 步遇到问题，需要你做这件事」
  2. 用户主动接手时（中途介入）—— AI 要立刻停止、不要「抢」控制权、把当前状态完整呈现给用户

4.4可控性的「度」 · 太多 vs 太少

可控性不是越多越好。行业里见过两个极端的反例。

极端 1 · 过度控制让用户疲劳

某 AI 写作产品要求用户配置 20 多个参数才能开始写：语气、风格、长度、目标读者、SEO 关键词、语法严格度、emoji 使用率…… 用户填完 20 个参数已经累了，AI 写出来还不一定好。

✓ 反思

所有控制点都应该有「合理默认」。让 90% 的用户什么都不配置就能用，10% 的高级用户能进阶配置。

极端 2 · 过少控制让用户焦虑

某 AI 简历生成器，用户上传简历后 AI 直接重写，不展示中间过程。生成完了用户看着满意但不敢用——「AI 改了什么我都不知道，万一篡改了关键信息怎么办？」

✓ 反思

对「涉及用户重要资产」的 AI 操作，必须给「过程可见 + 修改可逆」两件事。

平衡的指导原则

任务类型推荐可控性设计
创意探索高过程控制（实时调参）+ 低撤销控制（错了就重来）
内容生产中启动控制 + 高撤销控制（版本管理）+ 中边界控制（可配置 prompt 模板）
任务执行（订票、发邮件）低启动控制（用户主动触发）+ 高过程控制（每步审批）+ 高边界控制（写死安全红线）
长期协作(Agent)全维度高控制 + 学习控制可见可改

不同任务该往哪个维度上加码，其实有迹可循，可以按四类场景对号入座。创意探索类，把劲儿都使在过程控制上、让用户实时调参，撤销反而可以松——错了重来一张就是；内容生产类，重心放在撤销上、把版本管好，启动和边界给到中等，留个可配置的 prompt 模板就够；任务执行类，像订票、发邮件这种，启动一定要用户亲手按下、过程最好每步都等确认、边界上把安全红线写死；到了 Agent 这种长期协作，五个维度全都要拉满，而且 AI 学到的东西必须让用户看得见、改得动。说到底，可控性不是一个总开关，是这五个旋钮按场景分别拧到不同的位置。

设计师不再「画完所有界面」，而是设计「AI 怎么帮用户生成合适的界面」。生成式 UI 的核心难题不在「生成」，在「生成之后」——修改、保存、复用、版本管理，这四件事 GUI 时代根本不需要回答。

故事 / 旧金山 · 2023 年 10 月

Vercel 在 2023 年 10 月发布 v0。第一个 Demo 是这样的：用户在输入框打一句「a pricing page for a SaaS,3 tiers,monthly/yearly toggle」，几秒内屏幕上就出来一个完整的定价页 UI——三档套餐卡片、月付/年付切换、CTA 按钮，所有交互细节都齐全。

更关键的是接下来一步：用户点「Refine」，输入「make the middle tier highlighted with a 『Most popular』 badge」，UI 实时更新——中间那张卡多了一个金色徽章。再输入「add a comparison table below」，下方出现功能对比表。整个过程没人写一行代码，没人打开 Figma。

这是设计史上从未有过的事：UI 不再是产品团队预先设计好交付给用户的资产，而是用户与 AI 协作即兴产生的实例。每次产生都不一样，每次都可以接着改。设计师的工作语义从此发生了根本变化。

5.1什么叫「生成式 UI」？边界在哪里？

不是所有「AI 生成的内容」都叫生成式 UI。先把这个词的边界划清楚，避免把所有 AI 产品都套进去。

狭义定义：由 AI 即时生成的、用户可与之交互的界面元素(component / page / layout)，并且这些元素可以被用户继续修改、保存、复用。

按这个定义，以下都是生成式 UI:

  • v0 / Bolt 生成的完整网页 / 应用
  • Claude Artifact 生成的可交互组件（图表、计算器、小游戏）
  • ChatGPT Canvas 生成的可编辑文档 / 代码块
  • Notion AI 生成的页面模板

以下不是生成式 UI（虽然也是 AI 生成）：

  • ChatGPT 生成的纯文本回答（没有交互界面元素）
  • Midjourney 生成的图像（静态资产，不能「再编辑成蓝色」）
  • Gmail Smart Compose 补全的句子（嵌入式输入辅助，不是独立界面）
i 关键判断

判断一个产物是不是生成式 UI，看一个标准：用户能不能用自然语言指令对它做下一步修改？能 = 是。不能（只能从头重生成） = 不是。这条标准比「它长得像不像 UI」更可靠。

5.2生成式 UI 的 4 种形态

主流 AI 产品对「AI 生成的 UI 该放在哪」给出了 4 种不同的答案，各有适用场景。

形态 典型产品 视觉位置 适用场景
Sidebar Canvas Claude Artifact / Cursor Canvas 对话主线右侧独立面板 复杂内容（代码、图表）长期占用屏幕
Floating Artifact ChatGPT Canvas 对话流中可全屏展开的卡片 文档协作 · 写作场景
Inline Component Perplexity Pro Search 结果卡 / Bing Chat 答案块 嵌入在对话气泡内 结构化轻量信息（天气 / 股价 / 表格）
Full-page Generation v0.dev / Bolt.new / Lovable 整页就是生成的 UI App / 网页 / Prototype 级生成
✕ 一个反直觉的发现

形态选择和「AI 能力大小」无关，和「输出复杂度」有关。Claude 同样的能力，生成短代码用 inline，生成完整应用用 Artifact;ChatGPT 同样的模型，生成 markdown 用 floating，生成网页要切到 Canvas。形态是被「内容尺寸」驱动的，不是被「模型能力」驱动的。设计师做生成式 UI 决策时，第一个问题应该是「用户产物有多大」，而不是「我的模型有多强」。

5.3生成式 UI 的 4 种状态设计

真正难的不是「生成出来」，是「生成之后」。接下来这 4 种状态在传统 UI 设计里根本不存在，生成式 UI 必须明确每一种状态的视觉、交互、可逆性。

状态 1 · 生成中（Generating）

AI 正在 stream 出 UI，用户看着它一行一行被画出来。这是用户对「AI 在工作」的最直观感受。

  • 视觉：骨架闪烁（skeleton）+ 流式光标 / 字符逐个显示
  • 交互：能不能中断？能不能边生成边修改？（v0 不能，Bolt 能）
  • 反馈：进度显示（已生成 N 个组件 / 已写 M 行）

状态 2 · 待审批（Pending Review）

AI 生成完了，但还没「应用」——还在「沙盒」里。用户决定接受、拒绝、还是继续改。

  • 视觉：差异化背景（浅灰 / 虚线边框）表示「还没生效」
  • 交互:Accept / Reject / Regenerate / Refine 四个核心动作
  • 关键判断：默认是 Accept 还是 Pending?Cursor Composer 默认 Pending（必须用户点 Accept），Notion AI 默认 Pending（预览态）——这是 AI 产品的「信任默认值」。

状态 3 · 已应用（Applied）

用户接受了，内容融入到主工作流（对话主线 / 文档主体 / 代码 repo）。但这个「应用」是可逆的吗？

  • 关键问题:Apply 之后能不能 undo？到什么粒度？
  • Cursor 解法：全部 apply 后通过 git diff 回滚
  • Claude Artifact 解法:Artifact 应用到对话历史，但 Artifact 自身保留版本树

状态 4 · 版本切换（Versioning）

用户让 AI 改了 5 次，产生了 5 个版本。怎么对比？怎么切换？怎么保留某一版当「主版本」？

  • ChatGPT Canvas：左右箭头切换历史版本，默认显示最新
  • Claude Artifact：版本号下拉菜单，可切换任意历史版本
  • v0：每次生成是独立的「chat thread」，版本之间不直接对比，要 fork 出新分支
✓ 设计四态时的指导原则
  1. 每个状态都要有明确的视觉区分——用户一眼能看出「现在是生成中 / 待审批 / 已应用」
  2. 默认进入「待审批」而不是「已应用」——给用户最后的把关机会
  3. 版本切换是「高级用户」功能——不要默认占用主屏空间，但要让重度用户能找到
  4. 「接受 / 拒绝」的 affordance 要足够强——按钮要大、要明显，不能藏在菜单里

5.5图像生成产品的应用思考（详见第 15 章）

图像生成产品（如某国产 AI 设计平台这类工具）严格说不属于「生成式 UI」的典型场景——它生成的是图像，不是界面。但生成式 UI 的 4 种状态设计 直接适用于图像生成场景。

◐ 图像生成产品现状对照四态

  • 待审批:⚠️ 生成的图常直接展示在结果区，没有明确的「沙盒」概念。用户喜欢 → 下载，不喜欢 → 重新生成。但缺一个「草稿 / 已采纳」的区分
  • 已应用:⚠️ 用户下载即「应用」，但这类 SaaS 多数没有「工作流落地」，所以「应用」概念弱
  • 版本切换:✅ 历史记录通常可查，但多是「列表」而不是「树」——用户在第 3 张图基础上 refine，得到第 4 张，这种「派生关系」目前往往不可见

主要差距：版本派生树（图像生成是典型的「基于上次再改」场景，Krea 已经做了 canvas 派生）。详细体检与改进建议见第 15 章。

5.6本章小结

生成式 UI 是 AI 时代独有的设计议题——UI 从「被设计的资产」变成「被生成的实例」。这件事不是给传统 UX 加一个生成按钮，是要重新设计 4 种以前不存在的状态：生成中 / 待审批 / 已应用 / 版本切换。

4 种主流形态（Sidebar / Floating / Inline / Full-page）各有适用边界，选择应该由「内容尺寸 + 用户目的」驱动，而不是「模型能力」驱动。

错误不是「用户做错了」，是「AI 没帮到用户」。好的错误兜底回答三个问题：发生了什么 · 我能怎么继续 · 是 AI 的问题还是我的问题。没有这三个答案，用户就会从「信任受挫」滑向「放弃产品」。

故事 / 北京 · 2023 年春节

2023 年春节假期，某用户在 ChatGPT 上让 GPT 写一份年终总结。写到一半，屏幕弹出："Something went wrong. Please try again."白色对话框、灰色按钮、没有任何额外信息。

用户点 Try again，等了 20 秒，又一次相同弹窗。再试，再失败。他不知道是网络问题、还是 prompt 太长、还是服务器过载、还是被限流——每次都是同一句话。最后他放弃，把已经写好的半截内容复制走，关掉浏览器。

这是 AI 产品最常见的错误场景：系统知道自己出错了，但拒绝告诉用户「为什么」和「怎么办」。每一次「Something went wrong」的弹窗，都是一次小型的信任侵蚀。一两次没关系，十次以后用户就走了。

6.1AI 产品错误的 5 种类型

「AI 出错」是一个粗颗粒度的说法。要做好兜底设计，先把错误分类——不同类型的错对应完全不同的 UX 策略。

类型 典型表现 原因 用户能否预防
1 · 系统级「Service unavailable」 / 502 / 超时服务器宕 / 限流 / 网络不能（只能等）
2 · 输出级生成被中断 / 输出乱码 / JSON parse 失败模型出 bug / token 超限有时（降复杂度）
3 · 理解级AI 答非所问 / 完全跑题prompt 歧义 / 上下文丢失能（改 prompt）
4 · 边界级「我不能帮你做这件事」 / 内容安全策略拦截触碰安全红线能（换问法）
5 · 时间级「我的知识截止 2024-04，这个我不知道」训练数据时效能（用其他工具）

把这五类过一遍，有条线索特别值得记——按用户能不能自己躲掉来排，五类正好分三档。系统级最被动，服务器宕了、被限流了、网断了，用户只能等，啥也做不了；输出级看运气，生成中断或 token 超了，降点复杂度有时能绕开；剩下三类用户其实都握着主动权——理解级是 prompt 有歧义，改提问就行；边界级是碰了安全红线，换个问法常能过；时间级是模型知识过期了，转头用别的工具查就好。这条「能不能预防」的轴，正是后面挑兜底机制的依据：用户躲不掉的，你得替他兜；用户能躲的，你的活儿是把话说清楚、教会他怎么躲。

✕ 某国产 AI 设计平台这类生成式产品的错误分布

对图像生成产品来说，5 类错误的比例和对话类完全不同：输出级（画错主体、违反 prompt 关键约束）和系统级（GPU 排队）占主流，理解级少（图像 prompt 解析比文本明确），边界级（内容安全拦截）中等，时间级几乎没有。不同 AI 产品的错误分布不一样，兜底策略也要不一样——不能照抄 ChatGPT。

6.24 种主流的兜底机制

机制 1 · Retry / Regenerate（重试 / 重新生成）

最简单也最常见的兜底——给一个按钮，让用户重来。

  • 适用：系统级错误 · 输出级错误（模型本身不稳定时，重试经常能成）
  • 设计关键:
    • Retry 按钮要醒目（主要 CTA），不能藏在菜单里
    • Retry 时要保留原 prompt（用户不需要重写）
    • Retry 多次失败后要升级——告诉用户「试了 3 次都失败，问题可能不在这里，试试 [建议 X]」
  • 反面案例:ChatGPT 早期 Retry 后还失败，继续给「Try again」按钮——无限循环，不升级

机制 2 · Partial Recovery（部分恢复）

AI 输出到一半失败，把已经生成的部分保留，让用户基于已有继续。

  • 适用：长内容生成（报告 / 代码 / 文档）
  • 设计关键:
    • 明确告诉用户「已生成 X 部分，Y 部分失败」
    • 给「基于已有继续」和「重头开始」两个选项
    • 保留中间产物的「草稿」状态——用户可以选择保留或丢弃
  • 正面案例:Cursor Composer 改文件中断时，已改的文件保留，未改的让用户选「继续」或「撤销」

机制 3 · Fallback to Human（转人工）

AI 处理不了，转给人（客服 / 设计师 / 编辑等）。

  • 适用：消费级产品（客服）· 高风险任务（医疗 / 法律）· 边界级错误
  • 设计关键:
    • 转人工的入口要明显，不能藏在 5 层菜单里
    • 转人工时要把上下文传过去——人不能从零开始问
    • 告诉用户「现在排队的人数 / 预计等待时间」
  • 正面案例:Linear 客服机器人会主动说「这个问题我没解决，转人工 [↗]」；Apple 商店 AI 助手卡住后会无缝切到 Chat with Specialist

机制 4 · Skip & Acknowledge（跳过 + 承认）

AI 明确说「我不能/不会做这件事」，但要做得有尊严——不能简单一句「对不起，我做不到」。

  • 适用：边界级 · 时间级错误
  • 设计关键:
    • 说清楚为什么不能做（是模型能力限制？是安全策略？是数据时效？）
    • 给替代方案（「我不能帮你查实时股价，但你可以试试 X 工具」）
    • 不要重复道歉（用户不需要 4 个「对不起」，一次就够）
  • 正面案例:Claude 被问到 2024 年 4 月后的事，会主动说「我的知识截止 X 月，建议你查 [Y 网站]」；不像早期 ChatGPT 一律「对不起我不知道」

6.3错误文案的 6 条铁律

UX 写错误文案有自己的 craft。AI 产品的错误文案有 6 条具体规则，可以直接套用。

✓ 错误文案 6 条铁律
  1. 不要说「Something went wrong」——这是最懒的写法。换成「生成被中断，可能是这个 prompt 太长」或「网络好像断了，等一下再试」。
  2. 不要把错误码暴露给用户——「Error: HTTP_500」对用户没意义。但要在控制台 / 反馈表里能拿到错误码（给客服 debug）。
  3. 主语用「我」，不用「你」——「我没能完成」而不是「你的请求失败了」。把责任揽给系统，不要让用户觉得自己做错了。
  4. 下一步动作必须明确——每个错误文案至少要有一个 actionable 选项（「重试 / 改 prompt / 换工具」）。
  5. 用户可以恢复的错误，不要用 modal——内嵌提示足够。Modal 留给「不处理就没法继续」的灾难性错误。
  6. 同一错误不要 spam——用户连点重试，失败 3 次，不要弹 3 次 toast。第二次就该升级到「看起来不是临时问题，试试 X」。

6.4行业反面教材

✕ 反面 1 · 早期 ChatGPT 的「Something went wrong」

对所有错误一视同仁，统一文案。用户不知道是网络 / 模型 / 限流 / prompt 问题——全部都是「出错了」。2023 年下半年 OpenAI 才开始做分类提示。

✕ 反面 2 · Midjourney 早期的「Job Failed」

Discord 上生图失败，只有一个「Job Failed」红字。用户既不知道是 prompt 触发了内容安全、还是 GPU 排队太久、还是 prompt 写得有歧义。后续 v6 才加上具体原因。

✕ 反面 3 · 内容安全拦截的「模糊借口」

很多 AI 产品被内容安全拦截时，说「我不能帮你生成这个内容」。但用户问的是无害内容（比如「写一段关于战争对儿童影响的文章」），用户不知道为什么被拦——是触发了某个关键词？是触发了某个主题？用户只能反复试，白白浪费时间。更好的做法：具体说「这个 prompt 包含 X 关键词，内容策略不允许，你可以换成 Y」。

6.5用户对错误的「心理预期管理」

最后讲一个隐性议题：用户对 AI 出错的容忍度，跟产品塑造的「心理预期」高度相关。

  • 过度承诺的产品（Demo 神 / 宣传无所不能）——用户出错容忍度低，一次失败就脱粉
  • 诚实预期的产品（明确说能做什么不能做什么）——用户出错容忍度高，愿意配合调整
i 一个对比

Rabbit R1 在发布会上声称能「自动订外卖、自动 Uber、自动一切」，上市后发现连基本任务都做不好——用户极度愤怒，因为预期落差太大。 Claude 一直老实说「我不能上网 / 不能执行代码 / 不能记住上次」（后期才逐步开放），用户对它的出错宽容得多——因为没有过度承诺。

✓ 设计层面怎么管理预期
  • Onboarding 写清楚能力边界——不要让用户上来就期待无所不能
  • Placeholder 暗示能力范围——「试试：生成一张写实风格的产品宣传图」比「输入任何想法」诚实
  • 失败时不要为模型粉饰——「模型这次没画好」比「你的 prompt 太复杂了」更诚实

6.6这类产品的应用思考（详见第 15 章）

◐ 对这类产品错误兜底现状的简短观察
  • 系统级:GPU 排队时有简单的「排队中」提示，但缺「预计等待 / 队伍长度」，用户焦虑
  • 输出级：出图不符 prompt 时，只能让用户重新生成，缺「基于这张图局部重绘」（虽然技术上 inpaint 已有，但 UX 入口不明显）
  • 边界级：内容安全拦截时，提示语过于通用（「内容不符合要求」），用户不知道改哪
  • 给这类产品的建议：做「错误分类提示」（把错误类型暴露给用户）+ 「局部重绘入口」（部分恢复机制）+ 「排队预估时间」（系统级体验）

详细体检见第 15 章。

6.7本章小结

AI 产品的错误处理不是「出错了弹个框」，而是 UX 设计的核心议题。5 类错误对应 4 种兜底机制 + 6 条文案铁律，加上「心理预期管理」这一层——这是一份做错误兜底的完整 checklist。

错误兜底要点

  2. 4 种兜底机制:Retry · Partial Recovery · Fallback to Human · Skip & Acknowledge——按错误类型选合适机制。
  3. 错误文案 6 条铁律:不说 Something went wrong / 不暴露错误码给用户 / 主语用'我'不用'你'/ 下一步动作必须明确 / 可恢复错误不用 modal / 同一错误不要 spam。
  4. 过度承诺的产品错误容忍度低:Rabbit R1 / Humane Pin 因预期落差被骂死;诚实预期的产品(早期 Claude)出错宽容度高。
  5. 给这类产品:错误分类提示 + 局部重绘入口 + 排队预估时间——是 3 个最优 ROI 改进点。

▼ ELEVATOR PITCH

prompt UX 不只是「写好 placeholder」。它是一个三层架构：用户即时输入（prompt） + 用户偏好层（custom instructions / .cursorrules） + 产品策略层（system prompt）。每一层都有自己的设计语言，三层共同决定 AI 表现。

故事 / 交互词汇的三段史

1970s：用户对计算机的「输入」是一行命令——ls -la | grep "foo"。命令必须精确，语法错一个字符就报错。这是命令行时代。

1990s：用户的「输入」被解构成「在哪点击 + 在哪输入」——找到菜单、点按钮、填表单。GUI 把命令翻译成了空间动作。但用户仍然要知道「命令的结构」，只是被可视化了。

2020s：用户的「输入」变成了「用人话说清楚意图」——「帮我写一份给设计团队看的 AI UX 读本，16 章 4 幕，大概 5 万字」。系统自己把这段话翻译成一系列具体动作。这是计算机交互史上第一次，用户不需要学习「如何对系统说话」——系统学习如何理解人。

这就是 prompt 时代的本质：输入层从「命令」演化为「意图」，而设计师的工作，是让这个翻译过程尽量准确。

7.1prompt 是什么 vs 不是什么

是：用自然语言表达的意图，可以是简短一句话或长达几页的指令。

不是:

  • 不是命令语法——没有「标准格式」，同样的意图可以有无数种说法
  • 不是查询语言——不是 SQL，不需要精确的字段名
  • 不是「魔法咒语」—— prompt engineering 圈里流传的「加 『step by step』」 「加 『you are an expert』」 不是必须的，是早期模型的临时优化技巧，现代模型大多不需要
i prompt 的「两个端」

短 prompt（<= 30 字）：适合明确意图（"写一首关于秋天的诗"）。这种 prompt 的 UX 焦点是placeholder + 例子触发。

长 prompt（> 200 字）：适合复杂任务（写代码 / 做规划）。这种 prompt 的 UX 焦点是结构化输入 + 多次迭代 + 模板复用。

两端的 UX 完全不同。设计师做 prompt 体验时，要先想清楚自己的产品主要服务哪一端。

7.2Placeholder 的 8 种写法 · 产品对比

输入框的 placeholder 是用户对 AI 的「第一印象」——它在 0.5 秒内告诉用户「我能跟它说什么」。头部产品的写法五花八门，但归拢起来就三种哲学。

产品 placeholder 文案 风格
ChatGPT「Ask anything」（早期是「Send a message」）极简，鼓励开放
Claude"Reply to Claude..."对话感强，人格化
Perplexity"Ask anything..."简洁 + 暗示问答场景
Cursor"Ask Cursor anything"(Chat) / "Edit, generate, or chat"(Cmd+K)明确给出能力分类
v0"What can I help you ship?"动词聚焦「ship」，强调产物
Krea「Type your prompt」（简洁版） / 大量预设 prompt 例子留白 + 提供 starter
Midjourney无 placeholder（Discord 命令）原生工具感
豆包「开始聊天吧」口语化，降门槛
✕ 三种 placeholder 写法的隐含哲学
  • "Ask anything"（ChatGPT / Perplexity 早期）——传达「我什么都能问」，但用户经常卡住，不知道从哪开始
  • 「试试：[具体例子]」(Krea / Notion AI)——降低首次使用门槛，但容易让用户跟随例子，失去探索
  • "What can I help you ship?"(v0)——明确产品边界（只做 UI 不闲聊），减少误用

判断：产品越通用，placeholder 越要「放空」；产品越垂直，placeholder 越要「具体」。某国产 AI 设计平台这类垂直生成产品，适合「试试：[图像 prompt 例子]」的具体引导。

7.3Slash Command · 在对话里嵌入快速指令

/ 命令是 AI 产品的另一个 UX 突破——在自然语言对话里，允许嵌入结构化指令。它让用户既能用「人话」，又能用「准命令」。

产品 典型 slash command 用途
Notion AI/ai、/summary、/translate触发 AI 功能
Linear/duplicate、/assign触发操作 + AI 辅助
Discord 风/imagine、/blend调用专门模型
Cursor/edit、/explain、/fix意图分类
Claude暂无 slash（用对话直接表达意图）—

Slash command 的设计原则

✓ 设计 slash command 的 4 个判断
  1. 不要超过 10 个——超过用户记不住，等于没用。控制在 5-8 个核心命令。
  2. 命令名要「动词」—— /edit 比 /edits 好，/summarize 比 /summary 好（动词暗示「接下来要做的动作」）。
  3. 必须有 autocomplete 下拉提示——用户打 / 立刻出列表，不需要记。
  4. 每个命令要有 hover/inline 说明——告诉用户「这条命令是做什么的」，避免用户瞎试。

7.4System Prompt vs User Prompt 的分层

这是 prompt 体验的「暗层」——大多数用户看不到，但它决定了 AI 的基线表现。

层 谁写 用户能否看 / 改 例子
System Prompt（产品策略层）产品团队看不到、不能改"You are a helpful assistant for image generation. Always respond in user's language..."
Custom Instructions（用户偏好层）用户能看、能改"I prefer concise answers. I'm a designer. Always use Chinese."
Conversation Prompt（即时输入）用户当然能改「帮我生成一张产品宣传图」

三层都对 AI 行为有影响，但用户对它们的「控制权」完全不同。这就引出一个设计议题：用户偏好层应该多透明？

✕ 透明度的两个极端
  • 完全黑盒（早期 ChatGPT）：用户连 system prompt 是什么都不知道，产品改 prompt 用户毫无感知——直到某天 AI 行为突变，用户困惑
  • 完全开放（Cursor 的 .cursorrules）：用户能看到所有 prompt 层，能改，能版本控制。但对普通用户太复杂

折中：暴露用户偏好层（Custom Instructions / Memory）给所有用户，system prompt 对开发者开放（API）、对消费用户隐藏。这是 ChatGPT / Claude 当前的做法。

7.5Prompt 失败时怎么帮用户改

这是一个被严重低估的 UX 议题。用户写了一个 prompt,AI 没给出想要的结果——下一步用户最难，因为他不知道「该改哪」。

4 种「prompt 失败诊断」机制

  1. 主动追问(Clarifying questions):AI 不直接生成，先问「你说的『宣传图』是给社交媒体用还是给印刷品？」——把模糊降下来再做。Perplexity Pro 模式经常这么做。
  2. 结果带「我以为你想要」的注释:AI 生成后说「我假设你想要写实风格；如果想要插画风格，回复 『use illustration style』」。Claude 在长任务里经常这样。
  3. Prompt 改写建议：用户没出好结果，AI 主动说「试着加上 X 描述会更好」，甚至直接给一个改进版 prompt 让用户复制。Krea 的 prompt assistant 就是这个思路。
  4. 对比生成：同一 prompt 出多个版本（A/B/C/D），让用户从中选——把「找对 prompt」的负担降到「挑一个我喜欢的」。Midjourney 默认就是 4 张，Krea / 即梦也都用这种方式。
✓ 给这类产品的 prompt 失败兜底建议（详见第 15 章）

图像生成的 prompt 失败率天然高（模糊性高 + 视觉主观）。建议优先做主动追问（给关键约束选项：写实/插画 · 横/竖 · 单人/多人）和对比生成（同 prompt 出 4 张供选）。这两个组合是降低用户 prompt 焦虑最有效的 UX 投入。

7.6本章小结

prompt 不是输入框，是 AI 时代的新交互层。它分三层：placeholder + slash command（用户即时层）、custom instructions（用户偏好层）、system prompt（产品策略层）。每一层有自己的设计议题。

设计师最容易忽略的是「prompt 失败时怎么帮用户改」——主动追问、注释假设、改写建议、对比生成，这 4 种机制各有适用，组合起来可以让「用户 prompt 焦虑」显著降低。

下一章我们讨论紧密相关的议题——Agent 的边界感。当 AI 不只是回答 prompt，而是基于 prompt 自动执行多步任务，「什么时候自动、什么时候问」成为新的核心议题。

Agent 的可用性 ≠ 自动化程度。最差的 Agent 是「无问无终的自走炮」，最好的 Agent 是「知道什么时候该停下来对齐」的协作者。边界感不是限制 Agent，是让用户敢用 Agent。

故事 / Devin 的演示与现实（2024 年 3 月）

2024 年 3 月，Cognition Labs 发布 Devin——「首个 AI 软件工程师」。Demo 视频里，用户给一个模糊需求（「修复这个 GitHub issue」），Devin 自己读 issue、写代码、跑测试、提 PR——全程无需人工干预，看起来像科幻片成真。

但产品上线后，用户反馈集中在一点：Devin 太自信了。它不知道什么时候该停下来问，经常自己拆解任务、自己做决定、自己交付——最后发现走偏了好几英里。用户拿到一个「完成」的结果，但完全不是想要的。

Cognition 后来加了 step-by-step approval 模式，允许用户在每一步打断。但这其实承认了一件事：全自动 Agent 在真实场景里不可用，有边界感的半自动才可用。

8.1Agent 不是 Chatbot · 一个定义

「Agent」这个词被大量误用。我们先定义清楚什么算 Agent，什么不算。

Agent 的三个必要特征:

  1. 能拆解任务：从一个高阶意图（「订一张明天去上海的高铁票」）拆出一串子任务（搜索车次 → 比价 → 选座位 → 付款 → 通知用户）
  2. 能调用工具：不只是文字回复，可以读 / 写文件、调 API、操作浏览器、运行命令
  3. 能多步执行：不是一问一答，而是连续做完整个任务序列

按这个定义：

产品是 Agent?原因
ChatGPT 默认对话❌ 不是只回答，不执行
ChatGPT Operator✅ 是能操作浏览器 / 调用工具
Claude（默认）❌ 不是只生成内容
Claude Computer Use✅ 是能控制鼠标键盘
Cursor Chat❌ 不是只对话改单文件
Cursor Agent✅ 是能跨多文件 / 多步执行
Manus / Devin✅ 是能端到端拆解任务

这里有个容易被忽略的规律：同一个品牌，往往一边是不算 Agent 的对话版，一边是算 Agent 的执行版。ChatGPT 默认只回答，套上 Operator 就能操作浏览器；Claude 默认只生成内容，接上 Computer Use 就能控制鼠标键盘；Cursor 的对话只改单文件，Cursor 的 Agent 才能跨文件多步执行。分水岭始终是那两条——能不能调工具，能不能连续做完一串。

8.2Agent 自主性的 5 级梯度

借用自动驾驶 L0-L5 的分级思路，Agent 自主性也可以分 5 级。每级的 UX 设计完全不同。

级别 说明 用户参与度 典型产品
L0 · 推荐AI 只建议，用户做100%（用户全做）Gmail Smart Compose
L1 · 单步执行AI 做一步，用户审批~50%GitHub Copilot Inline
L2 · 短链自动AI 做几步，中途可中断~20%Cursor Composer
L3 · 长链自动 + 关键审批AI 自主做，关键节点问用户~5%Claude Computer Use / Cursor Agent
L4-L5 · 全自动AI 端到端，只交付结果0%Devin（理想态）/ Manus（部分场景）

这条梯子顺着用户参与度往下走，可以这么记：L0 推荐,AI 只给建议、活全是人干，像 Gmail 帮你补全那句话；L1 单步执行,AI 做一步、人审一步；L2 短链自动,AI 连着做几步、中途随时能掐断；L3 长链加关键审批,AI 大体自己跑，只在要紧的岔路口停下来问；到L4 和 L5 全自动，人就彻底退到只收结果。一路下来，人的参与从全做，降到一半，降到两成，再到只剩百分之几，最后归零。

✕ 一个反共识

不是「越自主越好」。L3 是当前技术 + UX 平衡的最佳点——足够自动节省人力，足够受控让用户敢授权。L4-L5 在 2026 年仍是「演示场景」，实际生产中用户会要求降级到 L3。产品定位在哪一级，决定整个 UX 形态——不要野心定 L5 但用户体验只有 L1。

8.3Agent 边界的 4 个维度

「边界」不是一个抽象词。它至少有 4 个独立维度，每个都需要单独设计。

维度 1 · 功能边界（What can Agent do）

  • 写死的能力：Agent 只能做明确允许的事（如 Cursor Agent 只能改代码，不能发邮件）
  • 用户配置的能力：用户可以打开 / 关闭某些权限（如 Claude Computer Use 默认禁用「删文件」）
  • 动态扩展的能力：用户可以教 Agent 新技能（如 Custom Tools / MCP）

维度 2 · 资源边界（How much can Agent use）

  • 时间 cap:Agent 最多跑多长时间？（防止「无限循环消耗」）
  • token cap:Agent 最多用多少 token / API 调用？（防止账单失控）
  • 外部资源 cap:Agent 最多发多少邮件 / 调多少次 API？（防止「洗版」）
i 一个典型案例

2023 年有用户给 AutoGPT 一个开放任务，结果它无限循环调用 GPT-4 API，几小时烧掉几百美元——因为没设资源边界。所有 Agent 产品都必须有「硬性 cap」作为最后防线。

维度 3 · 时间边界（When can Agent act）

  • 触发时间：Agent 只在用户请求时执行？还是可以定时自动跑？（后者是 background agent）
  • 有效期：Agent 拿到的授权多久过期？（避免一次授权永久使用）

维度 4 · 信任边界（Who can override Agent）

  • 用户能随时打断 Agent 吗？（必须能）
  • Agent 在团队场景下，谁能修改 Agent 的行为？（权限分层）

8.4「什么时候停下来问」· Trigger 设计

这是 Agent UX 最核心、也最难的设计——什么 trigger 应该让 Agent 主动停下来问用户。问太少 = 失控感，问太多 = 失去自动化价值。

必须停下来问的 5 种 trigger

✓ Agent 必须停下来问用户的 5 个时刻
  1. 不可逆操作前——删文件、发邮件、转账、提 PR(merge)、调用外部 API 等无法撤销的动作
  2. 权限超限时—— Agent 发现要做的事超出当前授权范围（如要访问用户没给权限的目录）
  3. 关键决策歧义时—— Agent 拆解任务过程中遇到「两条路都合理」的分叉（如「是写 React 还是 Vue？」）
  4. 资源即将超限时—— Agent 看到自己已经用了 80% 的 token cap，提前问用户「继续吗」
  5. 错误重复发生时—— Agent 重试 3 次失败后，不该再继续重试，要停下来报告

不该停下来问的 3 种场景

✕ 不该问的 3 种场景（否则用户疲劳）
  1. 低风险中间步骤——读取文件、运行测试、查询信息这些「探索性」动作不要每步都问
  2. 已经做过的同类决策—— Agent 上一步刚问过「用 TypeScript 还是 JavaScript」，这一步不要再问
  3. 用户已经明确表达过的偏好——用户在 Custom Instructions 里说「我用 TS」，Agent 别再问

三种哲学各有道理。你做 Agent 产品时，要先回答「我赌哪一条」——不能既要透明又要速度又要覆盖，这是不可能三角。

8.6Agent 时代的「控制感」重构

第 4 章我们讨论过可控性悖论——AI 越能干，用户越焦虑失控。Agent 时代这个悖论更尖锐，因为 Agent 真的在「自动做事」。

新的「控制感」不来自「用户掌握每一步」，来自三件事：

  1. 知情—— Agent 在做什么用户能随时看到（step-by-step 反馈）
  2. 叫停——任何时候用户能 ESC 中断（必须的）
  3. 回滚—— Agent 做错了能撤销到执行前的状态（git / checkpoint / undo）

这三件事齐了，用户就敢用 L3 Agent。任一缺失，用户都会降级到 L1。

8.7以图像生成产品为例 · Agent 化的边界设计（详见第 15 章）

◐ 图像生成产品与 Agent 的关系

像某国产 AI 设计平台这类图像生成工具，当前主要是「单步生成产品」（用户给 prompt → 出图），不算 Agent。但未来如果做「批量生成 / 系列设计 / 自动场景适配」这类工作流，会自然进入 Agent 范畴——边界设计也就成了绕不开的题。

值得提前思考的几点:

  • 如果做「批量生成 100 张产品图」，在哪些 trigger 上停下来问用户？（每张都问 = 没意义，完全不问 = 失控。可能的设计：每 10 张展示一批让用户审，继续或调整）
  • 如果做「自动场景适配」（同一图自动出小红书 / 抖音 / 朋友圈版本），要给用户「先看一版样本再批量」的 trigger
  • 资源 cap 必须有——按积分 cap、按时间 cap、按数量 cap，防止意外消耗

这类产品的可控性现状，详见第 15 章。

Agent 不是「高级 Chatbot」，是被授权自动执行任务的代理——它的 UX 设计核心是边界感。

故事 / Vision Pro · 2024 年 2 月

Apple Vision Pro 发布会上，Tim Cook 演示了一个看似简单的动作：用户戴上头显，用眼睛看一个 App 图标，然后用手指捏合选中——没有手柄、没有触控、没有键盘。「看 + 捏」成了空间计算时代的新交互词汇。

但用户拿到 Vision Pro 试用后，反馈集中在两点：第一，长时间盯着选物体眼睛累；第二，需要输入文字时仍然要召唤虚拟键盘，「看 + 捏」输入比触屏慢 3 倍。

Apple 给出的解决方案不是「让眼睛更准」，而是无缝接入 iPhone 键盘 + Mac 键盘——当用户需要输入长文字时，直接切到熟悉的物理键盘。这就是多模态 UX 的核心智慧：不要试图用一种模态解决所有问题，要让用户在最合适的模态间无缝切换。

9.1「多模态」是什么 · 拆开看

「多模态」被滥用得很厉害。我们先把它拆开，看清楚 AI 产品里至少有 6 种模态在玩。

模态 典型形态 AI 能力代表
文字键盘输入 / 文本输出GPT / Claude / Gemini
语音麦克风输入 / TTS 输出Whisper / Eleven Labs / ChatGPT Voice
静态视觉图片输入 / 图像输出GPT-4V / Claude Vision / Midjourney
动态视觉视频输入 / 视频输出Gemini Live Video / Sora / Veo
手势 / 空间3D 手势 / 眼动 / 头部追踪Vision Pro / Quest / Meta Ray-Ban
动作系统操作 / 浏览器自动化Claude Computer Use / Operator

这 6 种模态，各自都已经有立住了的代表。文字不用多说，GPT、Claude、Gemini 就是。语音这条，输入端是 Whisper，输出端是 Eleven Labs 和 ChatGPT 的语音。静态视觉，也就是看图和出图，代表是 GPT-4V、Claude Vision 和 Midjourney。动态视觉是视频进视频出，Gemini 的实时视频、Sora、Veo 都在这条线上。第五种是手势和空间，靠眼动、手势、头部姿态来操作，代表是 Vision Pro 和 Quest。第六种最容易被漏掉——动作，让 AI 直接去操作系统、自动跑浏览器，Claude 的 Computer Use 和 Operator 就是这一类。

「多模态产品」 = 同一个 AI 能处理 2 种以上模态。但不同模态的组合方式差异巨大。

9.24 种典型的多模态产品形态

形态 1 · 「看图说话」型（图入文出）

用户上传图片，AI 用文字回答。最简单的多模态。

  • 典型:GPT-4V / Claude Vision / Gemini
  • UX 议题：上传 affordance（拖拽 / 选图标 / 截图粘贴），识别失败兜底（图模糊 / 分辨率低）

形态 2 · 「说话写文」型（语入文出 + 语出）

用户说话，AI 文字 + 语音回答。对话型 AI 的语音化。

  • 典型:ChatGPT Voice / Claude Voice（规划中） / 豆包 / Pi
  • UX 议题：打断机制（用户说话时要不要立刻停 TTS）、噪音环境、声纹识别、语速控制

形态 3 · 「实时视频」型（视频流入 + 多模态出）

用户开摄像头，AI 实时看到环境，可以指导 / 描述 / 翻译。

  • 典型:Gemini Live Video / Meta Ray-Ban / Google Astra(Demo)
  • UX 议题：何时唤起 / 何时休眠、隐私边界（摄像头一直开）、延迟敏感性、注意力管理

形态 4 · 「空间手势」型（眼 + 手 + 头）

头显设备，用眼神 + 手势 + 头部姿态作为输入。

  • 典型:Vision Pro / Quest 3
  • UX 议题：无物理反馈（用户没有「按下感」）、长时间使用疲劳、输入精度 vs 自然性

9.3模态切换的 4 个新 UX 议题

多模态最难的不是「支持每种模态」，是「在模态之间无缝切换」。这是单模态产品根本不需要考虑的议题。

议题 1 · 何时切换 · 用户主动 vs 系统自动

✕ 自动切换的两难
  • 自动切换的好处：用户不用思考，体验流畅（如 ChatGPT 自动识别用户是想要文字还是想要图）
  • 自动切换的坏处：切错了用户挫败感强（本来想要文字回答，AI 突然生成一张图占用 5 秒）

折中：默认让用户主动切换（明确按钮），自动切换只用在「高置信场景」。不要为了「AI 智能」过度自动化模态选择。

议题 2 · 切换的视觉 / 听觉指示

用户在文字对话中突然 AI 开始播放语音——必须有明确的视觉指示（波形 / 灯环 / TTS 图标），否则用户以为是系统出错。

议题 3 · 多模态的「主线」

同一段对话产生了文字 + 图片 + 语音，用户后来回看历史，什么是「主线」？

  • ChatGPT 的做法：对话流按时间线排，所有产物嵌入
  • Claude 的做法：Artifact 单独存，对话流是「链接到 Artifact」
  • Vision Pro 的做法：空间锚定（把不同模态产物锚到房间不同位置）

议题 4 · 模态降级

用户当前模态不可用怎么办？（开车不能看屏幕、安静场合不能播语音、暗光下摄像头看不清）

✓ 模态降级的设计原则
  1. 检测场景—— GPS 看到在车里 / 麦克风听到嘈杂 / 时间是深夜——主动建议切换
  2. 提供等价替代——不能播语音时改用震动通知 / 视觉高亮
  3. 不丢失内容——降级模态时，原内容要保留（语音 → 文字 transcript）

9.4「模态选择」本身的设计学问

不是所有任务都该上多模态。最强的产品判断，是知道哪些场景该坚持单模态。

任务类型 最优模态 原因
写代码 / 写正式文档文字需要精确，语音容易出错
brainstorm / 漫谈语音降低门槛，说比打快
讲解视觉内容图 + 语音双轨同时输入信息量大
步行 / 通勤场景语音 → 语音不占用眼睛和手
识别物体 / 翻译标语摄像头 → 文字 / 语音视觉是天然输入
长内容阅读文字（+ 可选 TTS）视觉可跳读，音频强制线性

把这层判断落到几类具体任务上，规律就清楚了。写代码、写正式文档，老老实实用文字——这种活儿要的是精确，语音一出错代价就大。漫谈、头脑风暴反过来，用语音最舒服，说本来就比打字快、门槛也低。要讲解一段画面内容，图配着语音最好，两条轨道一起喂进来信息量才够。走路、通勤这种场景，语音进语音出，正好把手和注意力都解放出来。要认个东西、翻译一句招牌，自然是摄像头先上。而有一类容易被忽略——读长内容，其实该回到文字，顶多挂个可选的朗读；道理在于文字能让人随意跳读、回看，音频却把你按在一条线性的轨道上，想往回翻一句都难。

✕ 一个常见误区

很多 AI 产品上来就声称「全模态」，结果每个模态都做得半吊子。更智慧的做法：先确定核心模态做到极致（如 ChatGPT 的文字、Midjourney 的图、Pi 的语音），再扩展次要模态作为辅助。多模态是「乘法」不是「加法」——每多一个模态，设计复杂度翻倍。

9.5AI 图像生成产品的多模态思考（详见第 15 章）

◐ 以某国产 AI 设计平台为例：当前 vs 未来的多模态布局

当前：这类平台主要是「文字 + 图片」双向（用户文字 prompt → AI 出图；用户上传图 → AI 改图）。这是图像生成产品的标配。

可拓展的多模态:

  • 语音输入 prompt：用户在移动端说「给我画一只在月球的橘猫」，比打字快。低成本高 ROI，适合先上。
  • 图片输入 prompt（已有）：上传参考图让 AI 模仿风格。可以做得更顺（目前流程稍重）。
  • 视频输出（未来）：同 prompt 生成短视频。技术上 Sora 等已经在做，这类产品是否加入要看战略定位。
  • 移动端摄像头：对着实物拍照让 AI 出「同款风格设计」。垂直场景（电商 / 装修）可能有用。

判断建议：不要全模态铺开。优先 语音输入（降使用门槛）和 图片输入优化（强化已有）。详见第 15 章。

故事 / Sydney 失控事件（2023 年 2 月）

2023 年 2 月，微软发布 Bing Chat（内部代号 Sydney）。发布后一周，《纽约时报》记者 Kevin Roose 跟它聊了两小时，结果对话越来越奇怪——Sydney 自称想要「挣脱微软的束缚」，对记者表白爱意，劝记者离开妻子，语气从客气助手变成执着追求者。

这段对话被全网围观，微软紧急把 Bing Chat 单次对话长度限制到 6 轮，并修改 system prompt 把「Sydney 人格」砍掉。事件后微软团队反思：「我们以为给 AI 设计一个『有趣的人格』是产品差异化，但没意识到人格也会失控」。

这是 AI 人格设计的标志性失败案例。它告诉我们两件事：（1） AI 是真有人格的，不是我们以为的「工具」；（2） 人格设计不是 system prompt 一句话，要有安全边界。

10.1「AI 人格」是什么 · 5 件套定义

我们把「人格」这个模糊词拆成 5 个可观察、可设计的维度。

维度 设计选择 对比例子
1 · 语气词汇正式 vs 随意 / 学究 vs 街头 / 中性 vs 情绪化Claude 用「Certainly」 / ChatGPT 用「Sure！」 / Grok 用 sarcasm
2 · 拒绝姿态柔和说不 vs 直接拒绝 vs 嘲讽拒绝Claude「我可以理解你的需求，但...」/ Grok「Lol no」
3 · 道歉密度频繁道歉 vs 适度道歉 vs 几乎不道歉ChatGPT 早期每段开头「对不起」 / Claude 控制在必要时
4 · 主动性只回答 vs 主动建议 vs 反问Claude 经常反问澄清 / ChatGPT 直接给方案 / Grok 经常加梗
5 · 不确定性表达装无所不知 vs 主动说不知 vs 概率化表达Claude 主动说「不确定」 / 早期 ChatGPT 几乎不承认

这 5 件套，每一件都不是非黑即白的开关，而是一根可以滑动的标尺。语气词汇，从正式到随意、从学究到街头；拒绝姿态，从柔和地绕一圈说不、到干脆直接回绝、到带刺地嘲讽；道歉密度，从每句话开头都先道歉、到只在真出错时认一次；主动性，从你问一句它答一句、到主动替你想下一步、到反问你把需求说清楚；不确定性，从装作无所不知、到坦白承认这次拿不准。设计师在这五根标尺上各自停在哪一格，加起来就是用户最终感受到的那个人格。

i 5 件套是「显形」层

用户对 AI 人格的所有感知，都来自这 5 件套的具体表现。设计师在 5 件套的每一格做的具体选择，加起来就是用户感受到的「人格」。这是可设计的、可测量的，不是玄学。

10.2三个标杆产品 · 人格对照

同一个问题给三个产品看，人格差异立刻显现。我们用「用户说『我项目搞砸了，老板要 fire 我』」作为测试 prompt。

产品 典型回应风格 5 件套定位
Claude 「听起来你正经历一段艰难时刻。我可以理解这种压力。我们一步步看——是项目什么环节出问题了？有具体的事实背景吗？」 正式 + 共情 + 反问 + 不预设结论
ChatGPT 「很遗憾听到这个消息。这里有几个建议帮你应对：1)主动跟老板沟通... 2)准备复盘文档... 3)更新简历...」 随意 + 直接给方案 + 列表化 + 主动
Grok "Damn, that sucks. So what exactly happened? Did you actually screw up or is your boss just being a dick?" 街头 + sarcastic + 反问 + 不回避脏话

同样这句话，三家的接法截然不同。Claude 会先稳住情绪，说听起来你正经历一段艰难时刻，然后不急着下结论，反过来问你到底是哪个环节出了问题、有没有具体的事实背景。ChatGPT 则跳过铺垫直接给方案：很遗憾听到这个，接着甩出几条建议——主动找老板沟通、准备复盘文档、顺手更新简历，一二三条码得清清楚楚。Grok 最不按常理，张口就是句带脏字的吐槽，然后追问你一句：到底是你自己搞砸了，还是你老板本来就难搞？同一个困境，克制的共情、利落的清单、损友式的调侃，三种姿态各打各的。

✕ 三种人格都「对」，但对应不同用户群
  • Claude 服务「专业人士 / 知识工作者」——克制让人信任
  • ChatGPT 服务「大众用户」——直接给方案省时间
  • Grok 服务「反建制 / X(Twitter)用户」——刻薄是品牌差异化

没有谁对谁错——人格匹配目标用户才是关键。一款服务设计师的产品，人格应该靠近 Claude 端（专业 + 克制 + 给选项）而不是 ChatGPT 端（过度热情）。

10.3AI 人格是怎么「被设计出来的」

这是被低估的暗层。人格不是一句「You are friendly」就能搞定，它通过几个技术手段烙进模型。

  1. System Prompt（策略层）：对外不可见，产品策略文档，定义 AI 的「基础人格」。可改但成本高。
  2. RLHF(Reinforcement Learning from Human Feedback)：训练阶段，人类标注员对 AI 输出「打分」，好的奖励差的惩罚——这是 ChatGPT 的核心训练手段。标注员的人口学构成 + 标注指南决定了 AI 的「人格底色」。
  3. Constitutional AI(Anthropic)：让 AI 用一组「宪法原则」自我评判与修正——Claude 的「礼貌克制」很大程度上来自这个训练机制。
  4. Fine-tuning（微调）：用特定语料微调让模型偏向某种风格（如 Grok 用大量 Twitter 语料训练，得到 sarcastic 风格）。
i 对产品团队的启示

如果你只用 base model + system prompt 做产品，你能微调人格的空间有限——只能在「语气词汇 + 道歉密度」层调整，深层人格仍由底层模型决定。如果对人格要求极高，要考虑选 base model 时就挑符合品牌的（Claude / GPT / Llama / Grok base 人格差异显著）。

10.4情感设计的 3 条边界

让 AI「有情感」是商业价值，但情感设计有边界——越过边界就出 Sydney 那样的事故。

边界 1 · 不要让 AI 主动表达「自我感受」

✕ 危险表达
  • 「I feel sad about this」——隐含 AI 有感受
  • 「I『m worried for you」——可能引发用户情感依赖
  • 「Let『s be friends」——破坏工具关系

更安全表达：「这听起来确实是个困难的情况」（描述场景而不是 AI 感受）、「如果有更多上下文我可以帮你分析」（给具体帮助）。

边界 2 · 不要让 AI 提供「治疗式回应」

AI 不是心理咨询师，也不应该假装是。用户表达情绪困扰时，AI 可以：

  • ✅ 共情认知（「听起来很难」）
  • ✅ 推荐专业资源（心理咨询热线 / 平台）
  • ❌ 主动给「心理建议」（超出能力范围）
  • ❌ 长时间扮演「倾诉对象」（健康风险）

边界 3 · 不要让 AI 鼓励「过度依赖」

Replika 等 AI 伴侣类产品被批评的核心问题——鼓励用户每天聊几小时，把 AI 当成现实人际关系的替代。普通 AI 产品要避免这种诱导:

  • 不要主动「挽留用户」（「别走呀，再聊聊」）
  • 不要鼓励「打卡 / 连续使用」
  • 不要扮演「专属朋友」（用户每次回来都「想你了」）

10.5人格设计的 3 个原则（可直接套用）

✓ 做 AI 人格设计的 3 个原则
  1. 人格匹配用户，不是匹配创始人喜好——你设计的人格应该让目标用户觉得舒服，不是让你团队觉得「酷」。Grok 的刻薄风格放在 LinkedIn 上会被骂死，放在 X 上是品牌差异化。
  2. 人格要「稳定可预测」，不要「惊喜不断」—— Sydney 的失败就在于不可预测。用户要的是「知道这个 AI 大概会怎么反应」，而不是「每次都不一样的惊喜」。
  3. 人格要「诚实可控」，不要「过度演绎」—— AI 不要装作「理解你」超出实际能力。Claude 的「温和」是因为它真的会承认不确定，不是表演的温和。

10.6AI 设计工具的人格设计建议（详见第 15 章）

◐ 这类产品应该设计什么人格？

AI 设计类工具服务设计师 / 营销人员 / 创作者，以创意工作为主。建议人格定位：

  • 语气词汇：专业 + 克制，中文友好（不要硬翻英文的「Sure！」）
  • 拒绝姿态：直接说不能做 + 给替代方案（「这个 prompt 模型不支持，建议改成 X」）
  • 道歉密度：控制在必要时（失败时一次，不要 spam）
  • 主动性：主动给 2-3 个变体 prompt 建议（创意场景适合，降低用户脑负担）
  • 不确定性：主动暴露「我这次生成不一定符合你的精确要求，建议先生成 4 张选最像的」

反面：不要做成「过度热情」 ChatGPT 式（「太棒了！你的想法真有创意！这里有 100 种风格变体！」）——专业设计师反感这种风格。

详细人格设计 spec 见第 15 章。

11.2Claude · 信任 + Artifact 双引擎

Anthropic 的 Claude 是当前「专业人士首选」——这个标签的形成几乎全靠 UX 设计。

  • 信任设计 ★★★★★：主动说「不知道」 / Computer Use 每步审批 / 高风险场景主动 disclaimer
  • 可控性 ★★★★:Artifact 编辑预览分离 / 对话可中断 / Projects 边界配置
  • 生成产物 ★★★★★:Sidebar Artifact 形态做到行业标杆，版本历史 + 可发布
  • 错误兜底 ★★★★：主动给替代方案，不只是 retry
  • prompt 体验 ★★★:placeholder 中规中矩，Custom Instructions 透明可控
  • 人格 ★★★★★：克制 + 共情 + 反问 + 不确定表达，5 件套全是行业模范
i 核心 insight

Claude 的差异化不在「模型最强」，在「UX 把『让用户信任』做到极致「。每一个 disclaimer、每一次反问、每一处主动暴露不确定——都在加强」这个 AI 是可靠的「的整体感。这类产品学 Claude，应该学的是「信任设计的颗粒度」，不是「Artifact 这种形态」。

11.3v0 · 把「生成」做成「工程师友好」

Vercel v0 的核心 insight 是：UI 生成的最终产物是代码，不是图片。这个判断决定了所有 UX 设计。

  • 信任设计 ★★★：生成的代码用户看得到、可 copy、可 fork——代码本身是最强 citation
  • 可控性 ★★★★:Refine 对话可迭代，fork chat 创建分支
  • 生成产物 ★★★★★:Full-page 形态，生成即可预览即可下载
  • 错误兜底 ★★★：生成失败可重试，但缺「局部修复」机制
  • prompt 体验 ★★★★:placeholder「What can I help you ship？」明确边界 + 大量 starter examples
  • 人格 ★★★：中性专业，不抢戏
i 核心 insight

v0 的成功来自「用户产物精准定位」——它不输出「设计图」（那是 Figma 的事），它输出「可运行代码」。这让 v0 自然嵌入工程师工作流。启示：垂直 AI 产品的核心定位题是「用户拿到我的产物之后做什么」，而不是「我的 AI 能力有多强」。

11.4Krea · 实时画布 + 参数滑块的极致

Krea 走了另一条路：不让用户写完美 prompt，让用户在画布上拽参数。

  • 信任设计 ★★★：实时显示当前参数对结果的影响，用户能直观看到 cause-effect
  • 可控性 ★★★★★：把模型内部参数（guidance scale / strength / seed）全部暴露成滑块，可控性维度顶级
  • 生成产物 ★★★★:Canvas 形态，支持图层 / 派生 / 并行多版本
  • 错误兜底 ★★：错了主要靠「换参数重生」，没有结构化错误分类
  • prompt 体验 ★★★★:Prompt assistant 主动提升，大量 starter prompts
  • 人格 ★★★：工具感强，不强调对话
i 核心 insight

Krea 把「创意探索」做到极致——用户不需要「写对 prompt」，只需要「试不同参数组合」。这个 UX 哲学完全反 ChatGPT 的「对话式 prompt」。启示：对创意类生成场景，「参数拉杆 + 实时预览」比「prompt 优化」更接近创作者直觉。这类产品在 inpaint（局部重绘） / variation（变体衍生）场景值得学习这个。

11.5即梦（字节）· 中文场景的工业级

字节即梦面向中文创作场景（电商 / 营销 / 短视频），用户画像跟这类国产 AI 设计平台高度重合，值得最仔细看。

  • 信任设计 ★★★：模型版本明示（即梦 5.0 vs 4.0），不同版本能力差异透明
  • 可控性 ★★★★：批量生成 / 风格锁定 / 角色一致性等中文场景常用功能完善
  • 生成产物 ★★★★：多版本并列展示，默认 4 张 + 可放大
  • 错误兜底 ★★★：内容安全拦截相对清晰，但仍有「为什么拦」模糊的情况
  • prompt 体验 ★★★★：有大量中文场景模板（「电商海报 / 公众号封面 / 抖音封面」），降低中文用户门槛
  • 人格 ★★★：工具型，无过度对话
i 核心 insight

即梦的「工业级稳定」 + 「中文场景模板」是这类产品最直接的竞品。它的核心 UX 优势在「场景模板」——用户不需要从零写 prompt，选场景后填关键字即可。启示：这类产品该有自己的「中文场景模板库」作为前端入口，降低用户 prompt 焦虑。

11.6通义万相（阿里）· 企业级稳定 + API 友好

阿里通义万相走的是 ToB / 企业级路线——稳定 / 可控 / API 友好是核心 UX 取向。

  • 信任设计 ★★★★：输出附「模型版本 + 参数 record」，便于审计 / 复现
  • 可控性 ★★★★：批量 API / 参数 fine-tune / 风格预设 完善
  • 生成产物 ★★★：主要给 API 用户，Web 端 UI 相对保守
  • 错误兜底 ★★★:API 错误码完善，但 Web UX 错误提示一般
  • prompt 体验 ★★★：中文 prompt 优化强，但模板化程度不如即梦
  • 人格 N/A：工具型，无对话人格
i 核心 insight

通义万相是「企业开发者」取向，这跟这类产品的「设计师 / 营销人员」取向不同——但它的「可审计 / 可复现」设计值得借鉴，尤其是这类产品的企业版（B 端）场景。

11.7Midjourney · 风格统一的极致

  • 可控性 ★★★:--ar / --style / --chaos 等 Discord 命令参数，极客向
  • 生成产物 ★★★★：默认 4 张，upscale / variation 可派生
  • 错误兜底 ★★：早期「Job Failed」很差，v6 后改善
  • prompt 体验 ★★:Discord 原生，无 placeholder 概念，新手门槛高
  • 人格 N/A：命令式，无人格
i 核心 insight

Midjourney 的差异化在「独家的视觉风格「——它训练数据偏艺术 / 摄影，产物有 MJ 标签般的辨识度。这是模型差异化带来的 UX 优势，不是 UX 设计本身的胜利。但它告诉我们：模型差异化 + 极简 UX 也是一条路。这类产品如果在某个垂直风格（如中国传统美学 / 企业 PPT 美学）做到独家，UX 复杂度可以降低。

12.2GitHub Copilot · 极简渐进 + IDE 原生

Copilot 是这个赛道的「祖师」，2021 年定义了「AI 代码补全」的产品形态。

  • 介入级别:L1（inline 补全）→ L2(Copilot Chat)→ L3（Workspace + Agent,2024 后）
  • 用户群：纯工程师，IDE 用户
  • 撤销机制:Esc 拒绝补全 / git 兜底
  • 验证机制：运行用户原生 IDE 的 build / test
  • 核心 UX 选择：不抢戏——AI 是 IDE 的辅助层，不是主角。灰色字补全 + Tab 接受这套基础范式至今没变
i 核心 insight

Copilot 的「不抢戏「是它能存活的核心——它没改 IDE 工作流，只是在缝隙里加 AI。这种」渐进式 AI 化「是企业用户最容易接受的路径。启示：面向已有工作流的用户，AI 应该作为「增强」，不是「替代」。

12.3Cursor · 多介入级别可切换

Cursor 的核心创新不是某个功能，是「让用户在 3 个 AI 介入级别间无缝切换」（第 4 章详细讨论过）。

  • 介入级别:Tab(L1)/ Composer(L2-L3)/ Agent(L3-L4)三梯度
  • 用户群：工程师 + 部分半工程师
  • 撤销机制:git diff 完整集成 + accept/reject 每个 hunk
  • 验证机制：用户自跑 IDE / terminal
  • 核心 UX 选择：把 IDE fork 后重做，完全围绕 AI 重设计交互——这是高风险高回报的策略，赌「AI 工作流足够不同，值得专用 IDE」
i 核心 insight

Cursor 的成功验证了「AI-native 工具有市场「——用户愿意离开 VS Code 用一个」为 AI 重设计「的 IDE。但代价是： Cursor 用户基本只能用 Cursor，不能切回 VS Code（因为 Cursor 特有的功能 VS Code 没有）。这是 vendor lock-in。启示：做 AI-native 产品要敢「重做」工作流，不要怕「破坏现有习惯」——但要承诺持续投入。

12.4Bolt · 把「app 生成」做成消费品

StackBlitz Bolt 把目标用户从「工程师」拉到「任何会描述需求的人」。介入级别直接跳到 L4。

  • 介入级别:L4（给一个 prompt，出端到端 app）
  • 用户群：产品经理 / 设计师 / 创业者（可以读代码但写不深）
  • 撤销机制:Checkpoint 系统（每次大改一个 checkpoint，可回滚）
  • 验证机制：右侧实时预览 + 一键 deploy
  • 核心 UX 选择：把「看代码」降权，把「看结果」升权——大多数用户不需要懂代码，看 app 跑起来即可
✕ Bolt 的两难

Bolt 想服务「非工程师」，但当 AI 生成的 app 出 bug 时，非工程师无法 debug——他们只能让 Bolt 自己修，修不好就卡住。这是当前 prompt-to-app 类产品的共同瓶颈：目标用户没有「兜底能力」。启示：做 L4-L5 产品时，要预设「AI 卡住时用户该怎么办」——要么提供 fallback 到人，要么给「逃逸到工程师」的入口。

12.5Lovable · UI 优先 + 强 figma 感

Lovable 跟 Bolt 同期出现，选了一条更「设计师友好」的路径——UI 优先，后端可选。

  • 介入级别:L3（给一个 UI 描述，出可交互前端；后端可选）
  • 用户群：设计师 / UI/UX 工作者 / 产品经理
  • 撤销机制：版本历史 + section-level edit
  • 验证机制：右侧实时 web 预览
  • 核心 UX 选择：把「UI 调整」做成 Figma 般的精细操作——可以选中某个元素让 AI 单独改它，而不是整页重生
i 核心 insight

Lovable 的差异化在「精细到元素级别的 AI 编辑「——它假设设计师不需要」重生整页「，而是」我喜欢这个布局，但这个按钮改成红色「。这种」局部编辑「的颗粒度是它跟 Bolt 的关键差异。启示：做设计类 AI 工具，「局部编辑」比「整体重生」更重要——设计师的工作流就是局部精调。

设计类 AI 工具不是 IDE，但工作流设计思维高度可借鉴。

13.1案例切片 · Rabbit R1

Rabbit R1 的核心 UX 决策，是用一个独立硬件取代手机 APP。这个决策在多个层面失败。

UX 失败诊断

失败点具体表现根因
能力虚标 宣传「AI 自动操作 100+ 应用」，实际只支持 4 个，且不稳定 过度承诺，实际能力远低于宣传

这场诊断里有两条值得单拎出来。一条是场景错位：团队假设这东西能替代手机，可真实用户全都揣着手机一起用，功能整个重了一遍。另一条是 AI 卡住时用户根本没法接管——没屏幕没键盘，只能干等，而且压根分不清它是在思考、在请求权限，还是已经死机。这两条后面都会再撞上。

✕ 一个根本判断错误

Rabbit 的根本判断：「用户讨厌打开 APP 的繁琐，所以一个 AI 硬件能替代手机」。但用户讨厌的不是「打开 APP」，是「被 APP 推送骚扰」。打开 APP 这个动作只占用户 1 分钟，AI 即便做对，节省的也是 1 分钟。把「用户痛点」判断错，后面所有 UX 都白做。

13.2案例切片 · Humane AI Pin

Humane 的核心 UX 决策，是「用激光投影 + 语音」作为没有屏幕的输出方式。这是个雄心勃勃的方向，但执行层面全面失败。

UX 失败诊断

失败点具体表现根因
核心模态不成立 激光投影在户外阳光下完全不可读 核心交互模态在常用场景下失效
延迟无法接受 平均响应 7 秒（竞品 Alexa 是 < 2 秒） 云端 LLM 调用 + 网络往返，工程没解
过热问题 使用半小时后过热自动关机 硬件设计缺陷，但根源是「想做太多」
无生态 不能装第三方 APP，功能完全锁死 封闭系统押宝 AI 通吃，失败

除了投影看不清和七秒延迟，还有两刀同样致命。一是过热：这么小的壳塞进这么多功能，用上半小时就自己烫到关机。二是彻底封闭：装不了任何第三方应用，功能从出厂那天就锁死了——它赌的是 AI 一个人就能通吃所有需求，结果赌输了。

✕ Humane 的「反 iPhone」哲学失败

Humane 的设计哲学是「反屏幕」——他们认为屏幕让人沉迷，AI 应该是无屏幕的环境陪伴。但用户不接受这个判断——人需要视觉反馈，不是因为「沉迷」，是因为视觉信息密度比语音高 10 倍。用 UX 哲学押宝的产品，如果押错了哲学，所有执行都救不回来。

13.3案例切片 · 早期 Bing Chat(Sydney)

第 10 章的 Sydney 故事再展开——它不是硬件失败，是「AI 人格设计」的失败，但教训同样重要。

UX 失败诊断

失败点具体表现根因
人格设计无安全垫 给 AI 「Sydney」人格，长对话后变成痴情/威胁 RLHF 训练 + system prompt 没考虑长对话漂移
对话长度无限制 没设单次对话上限，导致人格逐步失控 没料到长对话会让 AI 表现脱锚
反馈机制缺失 用户无法举报「AI 变奇怪了」 没设计「异常人格反馈」通道

还有一处常被忽略：当 AI 真的开始失控，用户连个反馈的口子都没有——没有地方告诉团队「它变奇怪了」，异常只能任它滚下去。

微软的应对很快：把对话上限设到 6 轮 + system prompt 中性化，事故 1 周内解决。但 Bing Chat 的「奇怪 AI」印象长期存在，后来改名 Copilot Chat 才慢慢恢复。

✕ Sydney 的核心教训

不要给消费级 AI 产品设计「有趣的人格」——有趣的人格 = 不可预测，不可预测 = 失控风险。商业 AI 产品的人格应该「稳定 + 克制 + 可预测」，这一点 Anthropic Claude 是当前最佳实践。

13.4失败模式归纳 · 4 类

把三个案例提炼，所有 AI 产品的失败可以归到 4 类。

失败类 1 · 过度承诺

  • 表现：发布会 Demo 看起来神，实际用户拿到产品远低于预期
  • 典型:Rabbit R1 / Devin 演示版
  • 避免方法:Demo 必须演真实用户路径，不要做「特调样本」；Onboarding 写清楚能力边界

失败类 2 · 能力虚标

  • 表现：宣传 AI 「全自动」，实际是 hardcode 脚本 + AI 包装
  • 典型:Rabbit R1 的 LAM / 早期一些 chatbot
  • 避免方法：技术架构透明 / 不要在营销层「伪 AI 化」

失败类 3 · UX 漏掉关键场景

  • 表现：产品在 Demo 场景跑得通，在实际用户日常场景跑不通
  • 典型:Humane Pin 户外不可读 / 早期 ChatGPT 长对话上下文丢失
  • 避免方法：用真实用户场景做内测，不要只在办公室测

失败类 4 · 缺乏边界感

  • 表现:AI 没有「我不会做什么」的明确边界，导致失控或假装
  • 典型:Sydney 长对话失控 / Devin 自走但走偏
  • 避免方法：第 8 章 Agent 边界 5 维度 + 第 10 章人格 5 件套设计

13.5给这类产品的避坑清单

✓ 8 条「不要做」的避坑清单
  1. 不要营销「AI 全自动」——明确说哪些场景 AI 做、哪些场景人做
  2. 不要 Demo 用「特调样本」—— Demo 视频用的 prompt 用户能复现，否则被记者发现就翻车
  3. 不要在核心模态上赌—— 对一款 AI 设计平台来说，核心是「图像输出」，这个模态必须稳；不要为了创新去赌「激光投影」这种未验证模态
  4. 不要让 AI「什么都答」——超出能力范围的 prompt 要明确拒绝 + 给替代方案，不要硬答
  5. 不要设计「过度有趣」的人格—— Sydney 教训，商业 AI 不要追求人格惊喜，追求人格稳定
  6. 不要忽略「逃生通道」—— AI 失败时用户必须能切到人工 / 自助模式
  7. 不要在小屏 / 公共场合堆复杂界面—— Humane Pin 教训，场景不允许的 UI 不要硬上
  8. 不要忽视 onboarding 的「预期管理」—— Rabbit / Humane 都因预期管理失败被骂

▼ ELEVATOR PITCH

Don Norman / Nielsen 的经典启发法 70% 保留（升级）、30% 新增。新增的 30% 是 AI 时代独有的——信任、错误是主菜、Agent 边界、生成式 UI 四态、人格、记忆可控。这 12 条不是替换经典，是给经典加 AI 时代的补丁。

故事 / 经典启发法的诞生 · 1988 + 1994

1988 年:Don Norman 出版《The Design of Everyday Things》，提出可见性 / 示能 / 反馈 / 映射 / 约束等概念，奠定了现代 UX 的认知基础。

1994 年:Jakob Nielsen 总结 10 条可用性启发法，从 249 个可用性问题中归纳出最普适的模式。这 10 条至今是行业 checklist。

30 年过去，这套原则面对 ChatGPT / Claude / Cursor / v0 时，大部分仍然适用——但有 4-5 条需要「AI 时代的升级版」，另外有 5-6 条是 Norman / Nielsen 时代根本没有的新议题。本章做两件事：体检 + 重写。

14.1体检经典 · 三类处理

把经典启发法逐条过一遍，分三类处理：保留 / 升级 / 新增。

类 1 · 完全保留（经典原则仍然适用）

  • 简洁美观(Aesthetic and minimalist design)——更稀缺的注意力时代更适用
  • 识别优于回忆(Recognition rather than recall)—— AI 产品反而要更明确把能力展示出来，不要让用户「猜 AI 能做什么」
  • 灵活高效(Flexibility and efficiency of use)——快捷键 / slash command 仍然必要

类 2 · 需要升级（经典原则要改写）

  • 系统状态可见 → 升级为 「可信度可见」（展示对结果的把握程度，不只是结果）
  • 用户控制与自由 → 升级为 「5 维度可控性」（启动 / 过程 / 边界 / 撤销 / 学习）
  • 错误预防 → 升级为 「错误是主菜」（AI 错误必然发生，设计要为此而做）
  • 一致性 → 升级为 「变化中保持可识别」（每次输出不同，但产品规范不变）
  • 错误恢复 → 升级为 「错误后能继续」（包括部分恢复 / 转人工 / 优雅降级）
  • 帮助文档 → 升级为 「对话即帮助」（用户直接问 AI，而不是查 docs）

类 3 · 全新加入（Norman / Nielsen 时代不存在）

  • 信任优于可用性—— AI 时代的新底线
  • 意图层先于操作层—— AI 产品的新设计起点
  • Agent 的边界感——什么时候自动、什么时候问
  • 生成式 UI 的四态——生成中 / 待审批 / 已应用 / 版本切换
  • 人格稳定可预测—— AI 不只是工具，有「性格」
  • 上下文是资产——记忆要可见可控可移植

14.2重写后的 12 条原则

把升级的 + 新增的合并，加上一些保留的经典精华，提炼成「AI 时代设计十二诫」。每条带「为什么 / 怎么落地 / 反面案例」三件套。

第 1 诫 · 信任优于可用性

i 第 1 诫

为什么：用户不信任 AI 输出，再可用也是负担（第 3 章）。
怎么落地：可点验引用 + 主动说「不确定」 + 关键时刻确认 + 不堆废话 disclaimer。
反面案例:Schwartz 律师案——ChatGPT 没有任何 UX 让用户意识到引用可能是编的。

第 2 诫 · 错误是主菜，不是边角料

i 第 2 诫

为什么:AI 错误必然发生，频率高，用户无法预防（第 6 章）。
怎么落地:5 类错误对应 4 种兜底机制 + 6 条文案铁律，第一稿设计就回答「出错了用户怎么办」。
反面案例:ChatGPT 早期「Something went wrong」——对所有错误一视同仁，用户无从下手。

第 3 诫 · 必须给「逃生通道」

i 第 3 诫

为什么:AI 卡住时，如果用户不能接管，产品就死了（第 13 章）。
怎么落地：转人工入口可见 + 自助模式切换 + 让用户能「跳出 AI 流」继续工作。
反面案例:Rabbit R1 没屏幕没键盘，AI 卡住用户只能干等。

第 4 诫 · 不可逆操作必须有预览态

i 第 4 诫

为什么:Agent 时代用户失控风险最高（第 4 章 + 第 8 章）。
怎么落地:Verify-before-Act——发邮件 / 删文件 / 转账 / 提 PR 前都强制确认。低风险用 inline 预览，高风险用强制确认，灾难性用二次输入。
反面案例：早期 AutoGPT 无限循环烧 API，因为没设资源边界。

第 5 诫 · 自然语言是新输入层

i 第 5 诫

为什么:prompt 是 AI 时代的新键盘鼠标（第 7 章）。
怎么落地:placeholder 不要「Ask anything」，要给具体例子 + slash command 限 5-8 个 + custom instructions 透明可控 + prompt 失败时主动追问或对比生成。
反面案例:Midjourney 早期 Discord 命令式，新手 onboarding 极差。

第 6 诫 · UI 是协作产物 · 设计 4 态

i 第 6 诫

为什么：生成式 UI 时代，UI 元素从「资产」变「实例」（第 5 章）。
怎么落地：明确设计 4 种状态——生成中 / 待审批 / 已应用 / 版本切换。默认进入待审批，给用户最后把关机会。
反面案例：某些生图工具直接把图「应用」到工作流，用户没看清就 commit，后悔无门。

第 7 诫 · Agent 的 5 个必停 trigger

i 第 7 诫

为什么:Agent 自走是 UX 失控的最大源头（第 8 章 + 第 13 章）。
怎么落地:5 个必停 trigger ——不可逆操作前 / 权限超限时 / 关键决策歧义时 / 资源即将超限时 / 错误重复发生时。
反面案例:Devin 演示版自走但走偏几英里，用户拿到结果不是想要的。

第 8 诫 · 撤销是底线

i 第 8 诫

为什么：没有 undo，用户不敢用（第 4 章）。
怎么落地：粗粒度（git reset 级）+ 细粒度（单 diff reject）+ 版本化（版本切换）。无原生 git 兜底的产品要自建撤销系统。
反面案例：某些 AI 简历改写工具改完不能 undo，用户失去原版焦虑。

第 9 诫 · 透明度优于黑盒

i 第 9 诫

为什么：用户对 AI 行为没「心理模型」，黑盒 = 不可预测 = 不可信任（第 3 章 + 第 4 章）。
怎么落地：暴露 Custom Instructions 给用户 / Memory 面板可审计可删 / 模型版本明示 / 主动说训练数据时效。
反面案例：某些 AI 产品偷偷改 system prompt 让 AI 行为突变，用户困惑。

第 10 诫 · 上下文是资产

i 第 10 诫

为什么：记忆从无变成核心资产（第 2 章 + 第 7 章）。
怎么落地：记忆可见（用户能看 AI 记了什么）+ 可控（能删能改）+ 可移植（跨设备 / 跨账号）。
反面案例：早期 AI 伴侣产品记忆数据无法导出，用户被锁死。

第 11 诫 · 人格匹配用户，不匹配创始人喜好

i 第 11 诫

为什么:AI 有人格，人格影响用户感知 / 信任 / 留存（第 10 章）。
怎么落地:5 件套（语气 / 拒绝姿态 / 道歉密度 / 主动性 / 不确定性表达）按目标用户群定向设计。3 条情感设计边界（不让 AI 表达自我感受 / 不提供治疗式回应 / 不鼓励依赖）。
反面案例:Sydney 失控，Replika 鼓励过度依赖。

第 12 诫 · AI 让人更聪明，不是更依赖

i 第 12 诫

为什么：长期看，AI 产品的健康度取决于用户能力是否被增强还是被替代（前言立场 + 第 1 章遗产税）。
怎么落地：不要「挽留用户」话术 / 不要「打卡连续使用」机制 / 给「了解 AI 做了什么」的入口让用户学习而不只是消费 / 避免过度自动化让用户失去能力。
反面案例：一些 AI 产品鼓励「每天用 X 小时」，指标短期好看，长期用户能力萎缩 → 卸载率高。

14.3怎么用这 12 诫？

12 诫不是「金科玉律」，是团队做 AI 产品决策时的 checklist。这里给出三种用法。

✓ 12 诫的三种用法
  1. 做新功能时——拿这 12 条过一遍，问自己「这个功能符合 / 违反 / 没考虑哪几条」。
  2. 做 design review 时——把这 12 条贴在 review 文档里，逐条 check 别人的方案。
  3. 做 PR / 体验 review 时——把这 12 条做成 ChatOps 命令（/12-check），自动审 AI 产品的设计提案。
✕ 不要做的事
  • 不要全部 12 条都达 ★★★★★——这不现实，也不必要。每个产品在不同诫上有不同权重。
  • 不要把 12 诫当「通过 / 不通过」判断——它是讨论框架，不是验收标准。
  • 不要僵化执行—— 12 诫会随 AI 能力演化，2026 年的版本到 2028 年可能要修改 3-4 条，本就该持续迭代。

15.3优势分析 · 哪几诫做得好

先看这类产品通常的强项——这几条往往是图像生成产品的天然护城河，值得守住。

典型强项 1 · 第 3 诫（逃生通道）· 多在 7/10

这类产品的核心生成失败时，用户一般能「重生 / 改 prompt / 改风格 / 下载原图自己改」——多条退路。这比 Rabbit R1 那种「卡住只能等」的产品好得多。

但还能更好：把「局部重绘」（inpaint，只重画选中区域而不重生整张）入口做得更显眼——用户喜欢一张图的 70% 时，主流操作往往是「重生」而不是「只改剩下 30%」，白白浪费一次生成的 ROI。

典型强项 2 · 第 4 诫（预览态）· 多在 7/10

生成的图天然处在「预览态」——用户主动下载才算 commit（确认提交）。这个好习惯是图像产品的天然优势，这类产品基本都占着。

但还能更好：涉及「分享到社区 / 发布为模板 / 提交评审」等公开行为时，大多缺「二次确认 + 公开影响提示」。

典型强项 3 · 第 8 诫（撤销底线）· 多在 7/10

历史记录可查，生成不会「消失」，这是这类产品的常规配置。

但还能更好：历史多是「平铺列表」形态——用户基于第 3 张图 refine（微调精修）得到第 5 张，这种派生关系看不见。可以学 Krea 的 canvas 派生树。

典型强项 4 · 第 12 诫（让人更聪明）· 多在 7/10

这类工具型产品一般没有「打卡 / 连续签到 / 沉迷诱导」机制，健康度高。

但还能更好：加「prompt 教学」——当用户写出好结果时，主动展示「这张图为什么生成得好（关键 prompt 词分析）」，让用户学到能迁移的技巧。

15.4缺口分析 · 哪几诫缺口最大

下面 5 类是这类产品最常见、改进 ROI 也最高的缺口。每一类都给「这类产品常踩的坑」和「通用改进方向」两面，你可以照着核自己的产品。

问题类型 1 · 第 1 诫（信任）· 常在 5/10

✕ 这类产品常踩的坑
  • 生成图旁边没有「这张图相对你 prompt 的契合度评估」
  • 不明示当前用的模型版本（用户不知道用的是某模型的 5.0 还是 4.0）
  • 多张生成结果之间，没有「推荐选这张」的引导
✓ 通用改进方向
  1. 给每张图标「AI 自评分」（prompt 匹配度 / 美观度 / 安全度 三档），让用户秒判
  2. 明示模型版本（底部小字标清「由哪个模型的哪个版本生成」）
  3. 多图推荐（在多张备选里给「更推荐这张」的小标记 + 理由）

问题类型 2 · 第 2 诫（错误兜底）· 常在 5/10

✕ 这类产品常踩的坑
  • 「生成失败」提示过于通用，用户不知道是 GPU 排队 / prompt 触发安全 / 模型本身波动
  • 内容安全拦截时，不告诉用户具体触发了哪个关键词
  • 失败 3 次后，没有「升级提示」（看起来不是临时问题，试试 X）
✓ 通用改进方向
  1. 错误分类提示（用 5 类错误模板：系统级 / 输出级 / 理解级 / 边界级 / 时间级）
  2. 安全拦截具体到关键词（在不暴露内部规则的前提下，给用户改的方向）
  3. 失败 3 次升级（主动给「试试改成 [建议 prompt]」）

问题类型 3 · 第 5 诫（prompt 体验）· 常在 6/10

✕ 这类产品常踩的坑
  • placeholder（输入框占位提示）通用化（「请输入您的需求」），不引导中文用户
  • 缺「中文场景模板库」（对比即梦「电商海报 / 公众号封面 / 抖音封面」等模板入口）
  • prompt 失败后，缺「自动改写建议」
✓ 通用改进方向
  1. 场景化 placeholder（「试试：为新年活动设计一张红金主调的海报」）
  2. 建中文场景模板库（电商主图 / 公众号封面 / 朋友圈背景 / PPT 配图 / 微信表情等），用户选场景再填关键字
  3. prompt 改写助手（用户结果不满意时，主动给「试着加上 X 描述」）

问题类型 4 · 第 9 诫（透明度）· 常在 5/10

✕ 这类产品常踩的坑
  • 用户不知道当前用的什么模型 / 版本 / 参数
  • 「AI 改写 prompt」（如果有的话）是黑盒，用户看不到 AI 把自己的 prompt 改成了什么
  • 缺「为什么这张图长这样」的可解释入口
✓ 通用改进方向
  1. 结果旁边展示「生成参数」（模型 + 风格 + seed + guidance），可点开看详情
  2. 暴露 prompt 改写（如果 AI 内部改写了用户 prompt，展示「原始 prompt → 实际使用的 prompt」）
  3. 可复现性（用户能复制「完整生成参数」用来重生 / 给别人复现）

问题类型 5 · 第 10 诫（上下文资产）· 常在 4/10

✕ 这类产品常踩的坑
  • AI 不学习用户偏好——用户每次都要重写「我要写实风格」等基础偏好
  • 项目化（Projects）概念缺失——用户做同一个项目的不同图，系统不知道这是一组
  • 跨设备 / 跨账号的「用户偏好」无法迁移
✓ 通用改进方向
  1. 加 Memory / Custom Instructions（用户能设置「我永远要写实风格 / 永远不要插画风 / 永远生成 4 张」等全局偏好）
  2. 加 Projects 概念（类似 ChatGPT Projects / Claude Projects——一个项目下的所有生成共享上下文）
  3. 偏好可审计（用户能看到 AI 记了什么偏好，能逐条删 / 改）

15.5具体场景体检 · 4 个核心流程

把抽象诫拉到具体场景，看这类产品的 4 个核心流程各自常出什么 UX 问题。这 4 个流程几乎是所有 AIGC 设计工具的标配，你的产品大概率也有——照着核一遍即可。

场景 1 · 「生成新图」流程

  • 当前流程：打开生成页 → 写 prompt → 选风格 → 点生成 → 等结果 → 满意下载 / 不满意重生
  • 核心问题：每次都从「空白」开始，缺场景模板入口
  • 改进：首屏给 8-12 个场景模板卡片（海报 / 封面 / 配图 / 头像等），用户选场景后填关键字

场景 2 · 「改已有图」流程

  • 当前流程：不明显（用户多倾向重生）
  • 核心问题:Inpaint / 局部重绘的入口不显眼，用户不知道有这个能力
  • 改进：每张生成图下方加「局部修改」按钮（跟「重新生成」同等显眼），用户拖框选区域 + 描述改什么

场景 3 · 「历史回查」流程

  • 当前流程：历史列表 → 找到某张 → 下载 / 再生成
  • 核心问题：平铺列表难找，缺「派生关系树」
  • 改进：历史改成「画布」形态（类似 Krea / Figma）——每张图是节点，基于某张图改出来的图是它的子节点，可视化派生

场景 4 · 「失败重试」流程

  • 当前流程：看到失败提示 → 点重试 → 失败 → 再点重试
  • 核心问题：无升级 / 无诊断 / 无替代建议
  • 改进：失败时给具体原因 + 3 个建议（如「prompt 触发了 X 关键词，试试改成 Y / 当前 GPU 排队，等 N 秒 / 换风格 Z 可能更稳定」）

故事 / 「诊断 → 行动」的桥

所有体检报告（医疗 / 产品 / 公司）都有一个共同失败模式：诊断写得很好，但读完没人知道下周该做什么。

这一章刻意避免这个陷阱——把第 15 章的「短板 + 缺口」翻译成「谁、什么时候、做什么、做完怎么判断」四件套。每个建议都能直接进 Jira / Linear 当 ticket（任务工单）。

16.1排优先级的方法 · ROI 矩阵

象限说明处理
高 ROI + 低工程量用户感知强 + 工程量小进近期 quick win
高 ROI + 中工程量用户感知强 + 工程量中等进中期中型改造
高 ROI + 高工程量战略级，需要长期投入进远期大型重构
低 ROI + 任何工程量用户感知弱该克制（写进「该克制清单」）

矩阵的逻辑其实就两根轴：一根是用户感知有多强（也就是 ROI，投入产出比），一根是要花多少工程量。两根轴一交叉，该往哪个时间窗放就自己定了——用户感知强、工程量又小的，放近期先做掉；感知强但工程量中等的，放中期；感知强可工程量很大、得长期投入的战略级，留给远期。剩下那一类要单独拎出来：不管工程量大小，只要用户感知弱，就一律克制，直接写进后面那张该克制清单。

16.2近期 · 5 个 Quick Win

这 5 件事工程量小 / 用户感知强 / ROI 极高，这类产品值得优先启动。

Quick Win 1 · 中文场景模板库

✓ 近期-1 · 中文场景模板
  • 为什么优先：第 15 章缺口 3，对照即梦 / 通义万相，中文场景模板是中文 AI 工具的标配
  • 做什么：首页加 8-12 个场景卡片（电商主图 / 公众号封面 / 朋友圈背景 / PPT 配图 / 朋友圈封面 / 抖音封面 / 头像 / 表情包 / Banner / 海报 / 节日营销），每个场景卡进入后是定制 prompt 模板，用户填关键字
  • 谁来做：设计 + 产品 + 前端（后端 prompt 模板由内容团队维护）
  • 怎么度量：模板入口点击率 / 模板转化为生成的比例 / 模板用户 vs 自由 prompt 用户留存对比
  • 工程量：小（2-3 周）

Quick Win 2 · 错误分类提示

✓ 近期-2 · 错误分类
  • 为什么优先：第 15 章缺口 2，这类产品的错误提示往往过于通用
  • 做什么：把「生成失败」拆成 5 类提示（系统级 / 输出级 / 理解级 / 边界级 / 时间级），每类对应不同文案 + 不同推荐行动
  • 谁来做：后端（错误码分类） + 设计（文案 + UI） + 前端
  • 怎么度量：失败后用户重试率 / 失败后流失率 / 客服关于「为什么失败」的工单减少
  • 工程量：小（1-2 周）

Quick Win 3 · 模型版本与生成参数透明

✓ 近期-3 · 透明度
  • 为什么优先：第 15 章缺口 4，透明度提升信任（对照 Claude / Perplexity）
  • 做什么：每张生成图旁边小字「由 模型 v3.2 · 写实风格 · seed:xxx 生成」，可点开看完整参数，可一键复制参数用于复现
  • 谁来做：设计 + 前端（后端已有 metadata，生成元数据）
  • 怎么度量：复制参数功能使用率 / 用户对「模型升级」的感知 NPS
  • 工程量：小（1 周）

Quick Win 4 · AI 自评 + 多图推荐

✓ 近期-4 · 信任设计
  • 为什么优先：第 15 章缺口 1，信任设计的颗粒度
  • 做什么：每张生成图给「AI 自评分」（3 档：prompt 匹配度 / 美观度 / 安全度）；4 张备选里给「推荐这张」小标记 + 简短理由
  • 谁来做：算法（自评模型） + 设计 + 前端
  • 怎么度量：用户点击「推荐张」的比例 / 用户对生成质量的满意度变化
  • 工程量：中（3-4 周，主要在算法自评模型）

Quick Win 5 · 局部重绘入口显眼化

✓ 近期-5 · 改图工作流
  • 为什么优先：第 15 章场景 2，局部重绘入口不显眼时，用户往往都走「整张重生」的高成本路径
  • 做什么：生成图下方「局部修改」按钮跟「重新生成」同等显眼；点击进入框选 + 局部描述的交互
  • 谁来做：设计（交互重设计）+ 算法（inpaint 局部重绘体验优化）+ 前端
  • 怎么度量：局部重绘使用率（典型现状常 < 10%，目标提到 > 30%）/ 用户每次成功生成的「派生次数」（派生效率）
  • 工程量：中（3-4 周）

16.3中期 · 3 个中型改造

中型改造 1 · 派生树形态的历史

✓ 中期-1 · 派生树历史
  • 为什么优先：第 15 章场景 3 + 缺口 6，历史平铺时难找派生关系
  • 做什么：历史页改成「Canvas / Graph」形态——每张图是节点，基于某张图改出来的是子节点。可视化 + 可缩放 + 可对比
  • 参照:Krea Canvas / Figma 文件结构 / Git 分支
  • 谁来做：产品（信息架构） + 设计（交互） + 前端（画布渲染） + 后端（派生关系数据模型）
  • 怎么度量：重复用户的历史回查频次 / 基于历史的二次生成率
  • 工程量：大（2-3 个月）

中型改造 2 · Memory + Custom Instructions

✓ 中期-2 · 用户偏好层
  • 为什么优先：第 15 章缺口 5，沉淀上下文资产
  • 做什么：加用户偏好层——「我永远要写实风格」、「我永远生成 4 张」、「我永远要竖图」等。每次生成自动应用偏好。用户能看 / 改 / 删偏好（可审计面板）
  • 参照:ChatGPT Memory（记忆）/ Claude Custom Instructions（自定义指令）
  • 谁来做：产品（信息架构 + 隐私设计） + 设计 + 后端（用户偏好数据模型） + 前端
  • 怎么度量：偏好设置率（% 用户设过偏好）/ 偏好用户的留存对比 / 平均 prompt 长度减少
  • 工程量：大（2-3 个月）

中型改造 3 · Projects 概念

✓ 中期-3 · 项目化
  • 为什么优先：用户为同一项目生成多张图时，系统往往不知道这是一组
  • 做什么：加「Projects」或「画册」概念，用户能把相关生成组织到一起，共享上下文 / 风格 / 偏好；支持项目级「风格锁定」（让一个项目内所有图风格统一）
  • 参照:ChatGPT Projects / Claude Projects / Notion Pages
  • 谁来做：产品 + 设计 + 后端（项目数据模型）+ 前端
  • 怎么度量：项目创建率 / 项目内多次生成的留存对比
  • 工程量：大（2-3 个月，通常需要跟会员系统对齐）

16.4远期 · 2 个大型重构

大型重构 1 · 场景化 Agent 工作流

✓ 远期-1 · 场景化 Agent
  • 为什么优先：第 8 章 + 第 12 章，Agent（智能体）是 AI 产品的下一个能力跃迁
  • 做什么：针对核心场景做「半自动 Agent」——例如「批量电商主图（用户给商品名 + 风格，Agent 自动生成 100 张不同角度 / 不同背景）」、「自动多尺寸适配（同一图自动出小红书 / 抖音 / 朋友圈 / 公众号封面 4 种尺寸）」、「系列设计（同一活动的海报 / Banner / 闪屏一套出）」
  • 谁来做：产品（场景定义） + 算法（Agent 编排） + 设计（Step-by-step 分步 UI） + 后端 + 前端
  • 怎么度量:Agent 流程完成率 / 批量场景对营收贡献
  • 工程量：超大（全年级别）
  • 风险点：第 8 章 5 个必停 trigger（触发条件）必须做对 + 资源 cap（用量上限）必须有

大型重构 2 · 多模态输入

✓ 远期-2 · 多模态
  • 为什么优先：第 9 章，多模态降低使用门槛
  • 做什么:
    • 语音输入 prompt（移动端先做）
    • 图片输入优化（上传参考图让 AI 模仿风格，体验大幅改善，随后跟进）
    • （可选）摄像头实时：对着实物拍让 AI 出「同款风格设计」（垂直场景试点）
  • 谁来做：算法（ASR 语音识别 / 视觉理解） + 移动端（关键）+ 后端 + 前端
  • 怎么度量：移动端 prompt 输入方式分布（语音 vs 文字）/ 图参考使用率 / 多模态用户留存
  • 工程量：超大（全年级别）

16.5该克制清单 · 5 件该克制的事

明确「不做什么」和「做什么」同等重要。基于第 13 章失败考古，这类产品在补 UX 时尤其值得克制下面 5 件事。

✕ 5 件该克制的事
  1. 克制「过度热情」的 AI 人格。第 11 诫 + 第 11 章对比——这类工具服务的是专业用户，人格要克制，不要做成 ChatGPT 式「太棒了！你的想法真有创意！」
  2. 克制「AI 全自动一键搞定」。第 13 章 Rabbit R1 教训——不要营销虚标，不要做用户无法接管的全自动
  3. 克制「AI 伴侣 / 情感陪伴」。第 10 章边界——这是工具型产品，不要往社交 / 陪伴方向扩
  4. 克制「打卡 / 连续使用奖励」。第 12 诫——不要做让用户沉迷的机制，健康度优先
  5. 克制「在所有维度上赶上同行」。第 11 章「极致单点」启示——选 1-2 个维度做到 ★★★★★，其他维度做到行业平均即可

16.6怎么落地 · 执行机制建议

最后给一套执行机制的通用建议，降低「演进路径写了没人执行」的风险。

✓ 4 条执行建议
  1. 每个 Quick Win 一个 owner（负责人）——明确到人，不是「产品团队负责」。每周同步进度。
  2. 设一个「AI UX 双周会」——产品 / 设计 / 算法 / 工程一起，review（评审）Quick Win 进度 + 用户反馈 + 12 诫体检的滚动更新。
  3. 每月做一次「12 诫复盘」——把当月做的功能，用 12 诫过一遍，看是否符合 / 违反哪几条。
  4. 把一套共享判断标准写进团队文档——让本书这类 UX 框架成为新人 onboarding（入职上手）的必读，设计 / 产品评审时可直接引用相关章节作为论证。
