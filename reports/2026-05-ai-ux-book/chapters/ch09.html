09MULTIMODAL · 问题地图 / 第九章

多模态融合:语音 + 视觉 + 手势 + 文本 Multimodal Fusion

AI 不再只是文字进文字出。当 GPT-4V 能看图、Whisper 能听音、Vision Pro 能识别手势、Sora 能生成视频——UX 不再是"二维界面",而是多模态协奏的体验编排。这一章看模态切换的 4 个新议题、4 种典型的多模态产品形态、以及"模态选择"本身的设计学问。

▼ ELEVATOR PITCH

多模态不是"加更多输入方式",是让用户在每个场景用最自然的模态。最大的设计难题不在"识别准不准",而在"模态切换的接缝"——从文字切到语音、从语音切到手势,接缝粗糙的多模态产品比单模态产品更难用。

故事 / Vision Pro · 2024 年 2 月

Apple Vision Pro 发布会上,Tim Cook 演示了一个看似简单的动作:用户戴上头显,用眼睛看一个 App 图标,然后用手指捏合选中——没有手柄、没有触控、没有键盘。"看 + 捏"成了空间计算时代的新交互词汇。

但用户拿到 Vision Pro 试用后,反馈集中在两点:第一,长时间盯着选物体眼睛累;第二,需要输入文字时仍然要召唤虚拟键盘,"看 + 捏"输入比触屏慢 3 倍。

Apple 给出的解决方案不是"让眼睛更准",而是无缝接入 iPhone 键盘 + Mac 键盘——当用户需要输入长文字时,直接切到熟悉的物理键盘。这就是多模态 UX 的核心智慧:不要试图用一种模态解决所有问题,要让用户在最合适的模态间无缝切换。

9.1"多模态"是什么 · 拆开看

"多模态"被滥用得很厉害。我们先把它拆开,看清楚 AI 产品里至少有 6 种模态在玩。

模态 典型形态 AI 能力代表
文字键盘输入 / 文本输出GPT / Claude / Gemini
语音麦克风输入 / TTS 输出Whisper / Eleven Labs / ChatGPT Voice
静态视觉图片输入 / 图像输出GPT-4V / Claude Vision / Midjourney
动态视觉视频输入 / 视频输出Gemini Live Video / Sora / Veo
手势 / 空间3D 手势 / 眼动 / 头部追踪Vision Pro / Quest / Meta Ray-Ban
动作系统操作 / 浏览器自动化Claude Computer Use / Operator

"多模态产品" = 同一个 AI 能处理 2 种以上模态。但不同模态的组合方式差异巨大。

9.24 种典型的多模态产品形态

形态 1 · "看图说话"型(图入文出)

用户上传图片,AI 用文字回答。最简单的多模态。

  • 典型:GPT-4V / Claude Vision / Gemini
  • UX 议题:上传 affordance(拖拽 / 选图标 / 截图粘贴),识别失败兜底(图模糊 / 分辨率低)

形态 2 · "说话写文"型(语入文出 + 语出)

用户说话,AI 文字 + 语音回答。对话型 AI 的语音化。

  • 典型:ChatGPT Voice / Claude Voice(规划中) / 豆包 / Pi
  • UX 议题:打断机制(用户说话时要不要立刻停 TTS)、噪音环境、声纹识别、语速控制

形态 3 · "实时视频"型(视频流入 + 多模态出)

用户开摄像头,AI 实时看到环境,可以指导 / 描述 / 翻译。

  • 典型:Gemini Live Video / Meta Ray-Ban / Google Astra(Demo)
  • UX 议题:何时唤起 / 何时休眠、隐私边界(摄像头一直开)、延迟敏感性、注意力管理

形态 4 · "空间手势"型(眼 + 手 + 头)

头显设备,用眼神 + 手势 + 头部姿态作为输入。

  • 典型:Vision Pro / Quest 3
  • UX 议题:无物理反馈(用户没有"按下感")、长时间使用疲劳、输入精度 vs 自然性

9.3模态切换的 4 个新 UX 议题

多模态最难的不是"支持每种模态",是"在模态之间无缝切换"。这是单模态产品根本不需要考虑的议题。

议题 1 · 何时切换 · 用户主动 vs 系统自动

✕ 自动切换的两难
  • 自动切换的好处:用户不用思考,体验流畅(如 ChatGPT 自动识别用户是想要文字还是想要图)
  • 自动切换的坏处:切错了用户挫败感强(本来想要文字回答,AI 突然生成一张图占用 5 秒)

折中:默认让用户主动切换(明确按钮),自动切换只用在"高置信场景"。不要为了"AI 智能"过度自动化模态选择。

议题 2 · 切换的视觉 / 听觉指示

用户在文字对话中突然 AI 开始播放语音——必须有明确的视觉指示(波形 / 灯环 / TTS 图标),否则用户以为是系统出错。

议题 3 · 多模态的"主线"

同一段对话产生了文字 + 图片 + 语音,用户后来回看历史,什么是"主线"?

  • ChatGPT 的做法:对话流按时间线排,所有产物嵌入
  • Claude 的做法:Artifact 单独存,对话流是"链接到 Artifact"
  • Vision Pro 的做法:空间锚定(把不同模态产物锚到房间不同位置)

议题 4 · 模态降级

用户当前模态不可用怎么办?(开车不能看屏幕、安静场合不能播语音、暗光下摄像头看不清)

✓ 模态降级的设计原则
  1. 检测场景—— GPS 看到在车里 / 麦克风听到嘈杂 / 时间是深夜——主动建议切换
  2. 提供等价替代——不能播语音时改用震动通知 / 视觉高亮
  3. 不丢失内容——降级模态时,原内容要保留(语音 → 文字 transcript)

9.4"模态选择"本身的设计学问

不是所有任务都该上多模态。最强的产品判断,是知道哪些场景该坚持单模态。

任务类型 最优模态 原因
写代码 / 写正式文档文字需要精确,语音容易出错
brainstorm / 漫谈语音降低门槛,说比打快
讲解视觉内容图 + 语音双轨同时输入信息量大
步行 / 通勤场景语音 → 语音不占用眼睛和手
识别物体 / 翻译标语摄像头 → 文字 / 语音视觉是天然输入
长内容阅读文字(+ 可选 TTS)视觉可跳读,音频强制线性
✕ 一个常见误区

很多 AI 产品上来就声称"全模态",结果每个模态都做得半吊子。更智慧的做法:先确定核心模态做到极致(如 ChatGPT 的文字、Midjourney 的图、Pi 的语音),再扩展次要模态作为辅助。多模态是"乘法"不是"加法"——每多一个模态,设计复杂度翻倍。

9.5千图AI 的多模态思考(详见第 15 章)

◐ 千图AI 当前 vs 未来的多模态布局

当前:主要是"文字 + 图片"双向(用户文字 prompt → AI 出图;用户上传图 → AI 改图)。这是图像生成产品的标配。

可拓展的多模态:

  • 语音输入 prompt:用户在移动端说"给我画一只在月球的橘猫",比打字快。低成本高 ROI,适合先上。
  • 图片输入 prompt(已有):上传参考图让 AI 模仿风格。可以做得更顺(目前流程稍重)。
  • 视频输出(未来):同 prompt 生成短视频。技术上即梦 / Sora 已经在做,千图AI 是否加入要看战略定位。
  • 移动端摄像头:对着实物拍照让 AI 出"同款风格设计"。垂直场景(电商 / 装修)可能有用。

判断建议:不要全模态铺开。优先 语音输入(降使用门槛)和 图片输入优化(强化已有)。详见第 15 章。

9.6本章小结

多模态不是给产品加更多输入按钮,而是让用户在每个场景用最自然的模态。6 种模态、4 种典型产品形态、4 个切换议题——这套坐标系帮你避免"全模态都做半吊子"的常见陷阱。

最关键的判断:多模态是乘法不是加法。先把核心模态做到极致,再用辅助模态弥补场景空白。
