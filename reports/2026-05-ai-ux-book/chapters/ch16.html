  1. 团队内部讨论 ROI 与现状匹配度

故事 / 「诊断 → 行动」的桥

所有体检报告(医疗 / 产品 / 公司)都有一个共同失败模式:诊断写得很好,但读完没人知道下周该做什么。

这一章我们刻意避免这个陷阱——把第 15 章的「得分 + 缺口」翻译成「谁、什么时候、做什么、做完怎么判断「四件套。每个建议都能直接进 Jira / Linear 当 ticket。

同时明确「不做什么「——好的路线图一半价值在」做什么「,另一半价值在」明确不做什么「,避免团队精力发散。

16.1路线图方法论 · ROI 矩阵

我们用一个简单矩阵决定「什么进 90 天 / 什么进 180 天 / 什么进 365 天」。

象限说明处理
高 ROI + 低工程量用户感知强 + 工程量小进 90 天 quick win
高 ROI + 中工程量用户感知强 + 工程量中等进 180 天中型改造
高 ROI + 高工程量战略级,需要长期投入进 365 天大型重构
低 ROI + 任何工程量用户感知弱不做(写进「不做清单」)

矩阵的逻辑其实就两根轴:一根是用户感知有多强(也就是 ROI),一根是要花多少工程量。两根轴一交叉,该往哪个时间窗放就自己定了——用户感知强、工程量又小的,进 90 天先做掉;感知强但工程量中等的,放 180 天;感知强可工程量很大、得长期投入的战略级,留给 365 天。剩下那一类要单独拎出来:不管工程量大小,只要用户感知弱,就一律不做,直接写进后面那张不做清单。

16.290 天(Q1)· 5 个 Quick Win

这 5 件事工程量小 / 用户感知强 / ROI 极高,建议立刻启动。

Quick Win 1 · 中文场景模板库

✓ 90D-1 · 中文场景模板
  • 为什么:第 15 章缺口 3,对照即梦 / 通义万相,中文场景模板是中文 AI 工具的标配
  • 做什么:首页加 8-12 个场景卡片(电商主图 / 公众号封面 / 朋友圈背景 / PPT 配图 / 朋友圈封面 / 抖音封面 / 头像 / 表情包 / Banner / 海报 / 节日营销),每个场景卡进入后是定制 prompt 模板,用户填关键字
  • 谁做:设计 + 产品 + 前端(后端 prompt 模板由内容团队维护)
  • 度量:模板入口点击率 / 模板转化为生成的比例 / 模板用户 vs 自由 prompt 用户留存对比
  • 工程量:小(2-3 周)

Quick Win 2 · 错误分类提示

✓ 90D-2 · 错误分类
  • 为什么:第 15 章缺口 2,当前错误提示过于通用
  • 做什么:把「生成失败」拆成 5 类提示(系统级 / 输出级 / 理解级 / 边界级 / 时间级),每类对应不同文案 + 不同推荐行动
  • 谁做:后端(错误码分类) + 设计(文案 + UI) + 前端
  • 度量:失败后用户重试率 / 失败后流失率 / 客服关于「为什么失败」的工单减少
  • 工程量:小(1-2 周)

Quick Win 3 · 模型版本与生成参数透明

✓ 90D-3 · 透明度
  • 为什么:第 15 章缺口 4,透明度提升信任(对照 Claude / Perplexity)
  • 做什么:每张生成图旁边小字「由 千图AI v3.2 · 写实风格 · seed:xxx 生成」,可点开看完整参数,可一键复制参数用于复现
  • 谁做:设计 + 前端(后端已有 metadata)
  • 度量:复制参数功能使用率 / 用户对「模型升级」的感知 NPS
  • 工程量:小(1 周)

Quick Win 4 · AI 自评 + 多图推荐

✓ 90D-4 · 信任设计
  • 为什么:第 15 章缺口 1,信任设计的颗粒度
  • 做什么:每张生成图给「AI 自评分」(3 档:prompt 匹配度 / 美观度 / 安全度);4 张备选里给「推荐这张」小标记 + 简短理由
  • 谁做:算法(自评模型) + 设计 + 前端
  • 度量:用户点击「推荐张」的比例 / 用户对生成质量的满意度变化
  • 工程量:中(3-4 周,主要在算法自评模型)

Quick Win 5 · 局部重绘入口显眼化

✓ 90D-5 · 改图工作流
  • 为什么:第 15 章场景 2,局部重绘入口不显眼,用户都走「重生」高成本路径
  • 做什么:生成图下方「局部修改」按钮跟「重新生成」同等显眼;点击进入框选 + 局部描述的交互
  • 谁做:设计(交互重设计)+ 算法(inpaint 体验优化)+ 前端
  • 度量:局部重绘使用率(从当前 < 10% 提升到 > 30%目标)/ 用户每次成功生成的「次数」(派生效率)
  • 工程量:中(3-4 周)

16.3180 天(H1)· 3 个中型改造

中型改造 1 · 派生树形态的历史

✓ 180D-1 · 派生树历史
  • 为什么:第 15 章场景 3 + 缺口 6,历史平铺难找派生关系
  • 做什么:历史页改成「Canvas / Graph」形态——每张图是节点,基于某张图改出来的是子节点。可视化 + 可缩放 + 可对比
  • 参照:Krea Canvas / Figma 文件结构 / Git 分支
  • 谁做:产品(信息架构) + 设计(交互) + 前端(画布渲染) + 后端(派生关系数据模型)
  • 度量:重复用户的历史回查频次 / 基于历史的二次生成率
  • 工程量:大(2-3 个月)

中型改造 2 · Memory + Custom Instructions

✓ 180D-2 · 用户偏好层
  • 为什么:第 15 章缺口 5,上下文资产
  • 做什么:加用户偏好层——「我永远要写实风格」、「我永远生成 4 张」、「我永远要竖图」等。每次生成自动应用偏好。用户能看 / 改 / 删偏好(可审计面板)
  • 参照:ChatGPT Memory / Claude Custom Instructions
  • 谁做:产品(信息架构 + 隐私设计) + 设计 + 后端(用户偏好数据模型) + 前端
  • 度量:偏好设置率(% 用户设过偏好)/ 偏好用户的留存对比 / 平均 prompt 长度减少
  • 工程量:大(2-3 个月)

中型改造 3 · Projects 概念

✓ 180D-3 · 项目化
  • 为什么:用户为同一项目生成多张图时,系统不知道这是一组
  • 做什么:加「Projects」或「画册」概念,用户能把相关生成组织到一起,共享上下文 / 风格 / 偏好;支持项目级「风格锁定」(让一个项目内所有图风格统一)
  • 参照:ChatGPT Projects / Claude Projects / Notion Pages
  • 谁做:产品 + 设计 + 后端(项目数据模型)+ 前端
  • 度量:项目创建率 / 项目内多次生成的留存对比
  • 工程量:大(2-3 个月,需要跟会员系统对齐)

16.4365 天(Y1)· 2 个大型重构

大型重构 1 · 场景化 Agent 工作流

✓ 365D-1 · 场景化 Agent
  • 为什么:第 8 章 + 第 12 章,Agent 是 AI 产品的下一个能力跃迁
  • 做什么:针对核心场景做「半自动 Agent」——例如「批量电商主图(用户给商品名 + 风格,Agent 自动生成 100 张不同角度 / 不同背景)」、「自动多尺寸适配(同一图自动出小红书 / 抖音 / 朋友圈 / 公众号封面 4 种尺寸)」、「系列设计(同一活动的海报 / Banner / 闪屏一套出)」
  • 谁做:产品(场景定义) + 算法(Agent 编排) + 设计(Step-by-step UI) + 后端 + 前端
  • 度量:Agent 流程完成率 / 批量场景对营收贡献
  • 工程量:超大(全年)
  • 风险点:第 8 章 5 个必停 trigger 必须做对 + 资源 cap 必须有

大型重构 2 · 多模态输入

✓ 365D-2 · 多模态
  • 为什么:第 9 章,多模态降低使用门槛
  • 做什么:
    • 语音输入 prompt(移动端,Q3)
    • 图片输入优化(上传参考图让 AI 模仿风格,体验大幅改善,Q4)
    • (可选)摄像头实时:对着实物拍让 AI 出「同款风格设计」(垂直场景试点)
  • 谁做:算法(ASR / 视觉理解) + 移动端(关键)+ 后端 + 前端
  • 度量:移动端 prompt 输入方式分布(语音 vs 文字)/ 图参考使用率 / 多模态用户留存
  • 工程量:超大(全年)

16.5不做清单 · 5 件主动不做的事

明确「不做什么」和「做什么」同等重要。这 5 件事是基于第 13 章失败考古 + 千图AI 当前阶段判断,不建议做。

✕ 5 件主动不做的事
  1. 不做「过度热情」的 AI 人格。第 11 诫 + 第 11 章对比——千图AI 服务专业用户,人格要克制不要做成 ChatGPT 式「太棒了!你的想法真有创意!」
  2. 不做「AI 全自动一键搞定」。第 13 章 Rabbit R1 教训——不要营销虚标,不要做用户无法接管的全自动
  3. 不做「AI 伴侣 / 情感陪伴」。第 10 章边界——千图AI 是工具型产品,不要往社交 / 陪伴方向扩
  4. 不做「打卡 / 连续使用奖励」。第 12 诫——不要做让用户沉迷的机制,健康度优先
  5. 不在所有维度上「赶上同行」。第 11 章「极致单点」启示——选 1-2 个维度做到 ★★★★★,其他维度做到行业平均即可

16.6怎么落地 · 执行机制建议

最后给一个执行机制的建议,降低「路线图写了没人执行」的风险。

✓ 4 条执行建议
  1. 每个 Quick Win 一个 owner——明确到人,不是「产品团队负责」。每周同步进度。
  2. 设「AI UX 双周会」——产品 / 设计 / 算法 / 工程一起,review Quick Win 进度 + 用户反馈 + 12 诫体检的滚动更新。
  3. 每月一个「12 诫复盘」——把当月做的功能,用 12 诫过一遍,看是否符合 / 违反哪几条。
  4. 把这本书放在团队文档区——新人 onboarding 必读,设计 / 产品评审引用本书章节作为论证。
