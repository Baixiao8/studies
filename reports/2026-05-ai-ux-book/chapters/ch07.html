07PROMPT LAYER · 问题地图 / 第七章

提示词作为新交互层:从命令到对话到意图 Prompt as New Interaction

prompt 不是"输入框",是 AI 时代的新键盘鼠标——是用户向系统表达意图的主要载体。这一章我们把 prompt 拆成 3 层(placeholder / slash command / system prompt),看每一层各自的设计议题,以及"prompt 失败时怎么帮用户改"这件被严重低估的事。

▼ ELEVATOR PITCH

prompt UX 不只是"写好 placeholder"。它是一个三层架构:用户即时输入(prompt) + 用户偏好层(custom instructions / .cursorrules) + 产品策略层(system prompt)。每一层都有自己的设计语言,三层共同决定 AI 表现。

故事 / 交互词汇的三段史

1970s:用户对计算机的"输入"是一行命令——ls -la | grep "foo"。命令必须精确,语法错一个字符就报错。这是命令行时代。

1990s:用户的"输入"被解构成"在哪点击 + 在哪输入"——找到菜单、点按钮、填表单。GUI 把命令翻译成了空间动作。但用户仍然要知道"命令的结构",只是被可视化了。

2020s:用户的"输入"变成了"用人话说清楚意图"——"帮我写一份给设计团队看的 AI UX 内部读本,16 章 4 幕,大概 5 万字"。系统自己把这段话翻译成一系列具体动作。这是计算机交互史上第一次,用户不需要学习"如何对系统说话"——系统学习如何理解人。

这就是 prompt 时代的本质:输入层从"命令"演化为"意图",而设计师的工作,是让这个翻译过程尽量准确。

7.1prompt 是什么 vs 不是什么

是:用自然语言表达的意图,可以是简短一句话或长达几页的指令。

不是:

  • 不是命令语法——没有"标准格式",同样的意图可以有无数种说法
  • 不是查询语言——不是 SQL,不需要精确的字段名
  • 不是"魔法咒语"—— prompt engineering 圈里流传的"加 'step by step'" "加 'you are an expert'" 不是必须的,是早期模型的临时优化技巧,现代模型大多不需要
i prompt 的"两个端"

短 prompt(<= 30 字):适合明确意图("写一首关于秋天的诗")。这种 prompt 的 UX 焦点是placeholder + 例子触发。

长 prompt(> 200 字):适合复杂任务(写代码 / 做规划)。这种 prompt 的 UX 焦点是结构化输入 + 多次迭代 + 模板复用。

两端的 UX 完全不同。设计师做 prompt 体验时,要先想清楚自己的产品主要服务哪一端。

7.2Placeholder 的 8 种写法 · 产品对比

输入框的 placeholder 是用户对 AI 的"第一印象"——它在 0.5 秒内告诉用户"我能跟它说什么"。我们看头部产品的 8 种写法。

产品 placeholder 文案 风格
ChatGPT"Ask anything"(早期是"Send a message")极简,鼓励开放
Claude"Reply to Claude..."对话感强,人格化
Perplexity"Ask anything..."简洁 + 暗示问答场景
Cursor"Ask Cursor anything"(Chat) / "Edit, generate, or chat"(Cmd+K)明确给出能力分类
v0"What can I help you ship?"动词聚焦"ship",强调产物
Krea"Type your prompt"(简洁版) / 大量预设 prompt 例子留白 + 提供 starter
Midjourney无 placeholder(Discord 命令)原生工具感
豆包"开始聊天吧"口语化,降门槛
✕ 三种 placeholder 写法的隐含哲学
  • "Ask anything"(ChatGPT / Perplexity 早期)——传达"我什么都能问",但用户经常卡住,不知道从哪开始
  • "试试:[具体例子]"(Krea / Notion AI)——降低首次使用门槛,但容易让用户跟随例子,失去探索
  • "What can I help you ship?"(v0)——明确产品边界(只做 UI 不闲聊),减少误用

判断:产品越通用,placeholder 越要"放空";产品越垂直,placeholder 越要"具体"。千图AI 这种垂直生成产品,适合"试试:[图像 prompt 例子]"的具体引导。

7.3Slash Command · 在对话里嵌入快速指令

/ 命令是 AI 产品的另一个 UX 突破——在自然语言对话里,允许嵌入结构化指令。它让用户既能用"人话",又能用"准命令"。

产品 典型 slash command 用途
Notion AI/ai、/summary、/translate触发 AI 功能
Linear/duplicate、/assign触发操作 + AI 辅助
Discord 风/imagine、/blend调用专门模型
Cursor/edit、/explain、/fix意图分类
Claude暂无 slash(用对话直接表达意图)—

Slash command 的设计原则

✓ 设计 slash command 的 4 个判断
  1. 不要超过 10 个——超过用户记不住,等于没用。控制在 5-8 个核心命令。
  2. 命令名要"动词"—— /edit 比 /edits 好,/summarize 比 /summary 好(动词暗示"接下来要做的动作")。
  3. 必须有 autocomplete 下拉提示——用户打 / 立刻出列表,不需要记。
  4. 每个命令要有 hover/inline 说明——告诉用户"这条命令是做什么的",避免用户瞎试。

7.4System Prompt vs User Prompt 的分层

这是 prompt 体验的"暗层"——大多数用户看不到,但它决定了 AI 的基线表现。

层 谁写 用户能否看 / 改 例子
System Prompt(产品策略层)产品团队看不到、不能改"You are a helpful assistant for image generation. Always respond in user's language..."
Custom Instructions(用户偏好层)用户能看、能改"I prefer concise answers. I'm a designer. Always use Chinese."
Conversation Prompt(即时输入)用户当然能改"帮我生成一张产品宣传图"

三层都对 AI 行为有影响,但用户对它们的"控制权"完全不同。这就引出一个设计议题:用户偏好层应该多透明?

✕ 透明度的两个极端
  • 完全黑盒(早期 ChatGPT):用户连 system prompt 是什么都不知道,产品改 prompt 用户毫无感知——直到某天 AI 行为突变,用户困惑
  • 完全开放(Cursor 的 .cursorrules):用户能看到所有 prompt 层,能改,能版本控制。但对普通用户太复杂

折中:暴露用户偏好层(Custom Instructions / Memory)给所有用户,system prompt 对开发者开放(API)、对消费用户隐藏。这是 ChatGPT / Claude 当前的做法。

7.5Prompt 失败时怎么帮用户改

这是一个被严重低估的 UX 议题。用户写了一个 prompt,AI 没给出想要的结果——下一步用户最难,因为他不知道"该改哪"。

4 种"prompt 失败诊断"机制

  1. 主动追问(Clarifying questions):AI 不直接生成,先问"你说的'宣传图'是给社交媒体用还是给印刷品?"——把模糊降下来再做。Perplexity Pro 模式经常这么做。
  2. 结果带"我以为你想要"的注释:AI 生成后说"我假设你想要写实风格;如果想要插画风格,回复 'use illustration style'"。Claude 在长任务里经常这样。
  3. Prompt 改写建议:用户没出好结果,AI 主动说"试着加上 X 描述会更好",甚至直接给一个改进版 prompt 让用户复制。Krea 的 prompt assistant 就是这个思路。
  4. 对比生成:同一 prompt 出多个版本(A/B/C/D),让用户从中选——把"找对 prompt"的负担降到"挑一个我喜欢的"。Midjourney 默认就是 4 张,Krea / 即梦也都用这种方式。
✓ 千图AI 的 prompt 失败兜底建议(详见第 15 章)

图像生成的 prompt 失败率天然高(模糊性高 + 视觉主观)。建议优先做主动追问(给关键约束选项:写实/插画 · 横/竖 · 单人/多人)和对比生成(同 prompt 出 4 张供选)。这两个组合是降低用户 prompt 焦虑最有效的 UX 投入。

7.6本章小结

prompt 不是输入框,是 AI 时代的新交互层。它分三层:placeholder + slash command(用户即时层)、custom instructions(用户偏好层)、system prompt(产品策略层)。每一层有自己的设计议题。

设计师最容易忽略的是"prompt 失败时怎么帮用户改"——主动追问、注释假设、改写建议、对比生成,这 4 种机制各有适用,组合起来可以让"用户 prompt 焦虑"显著降低。

下一章我们讨论紧密相关的议题——Agent 的边界感。当 AI 不只是回答 prompt,而是基于 prompt 自动执行多步任务,"什么时候自动、什么时候问"成为新的核心议题。
