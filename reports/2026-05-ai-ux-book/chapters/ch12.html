AI 辅助生产力对比组:同一个问题的 4 种不同解法 AI DevTools Comparison

Cursor / GitHub Copilot / Bolt / Lovable——四款 AI 辅助生产力工具,服务相似用户(工程师 / 半工程师),但 UX 选择截然不同。这一章不是"评开发工具",是看"同一个 AI 能力(写代码)在不同的产品形态下,UX 会被设计成完全不同的东西"——这是给千图AI 的对照启示。

▼ ELEVATOR PITCH

4 个产品 = 4 种"AI 介入级别"的精准对应:Copilot 是 L1(单步建议),Cursor 是 L2-L3(短链到长链可切换),Bolt 是 L4(端到端 app),Lovable 是 L3(UI-first 半自动)。这 4 个产品共同证明:AI 产品 UX 没有"通用方案",必须先选定"介入级别"再设计。

故事 / 2021-2026 五年开发者 AI 工具进化

2021 年 6 月:GitHub Copilot 公测,把 AI 代码补全做成 IDE 插件。模式是 L1——AI 建议,用户审批。

2023 年:Cursor 诞生,把 Copilot 模式扩展到"多文件改"(Composer),走到 L2-L3。同时给 Agent mode 升到 L3。

2024 年:Bolt 出现,目标用户从"工程师"扩展到"产品经理 / 设计师",做 prompt-to-app,走到 L4。Lovable 同期切 UI-first 路径。

2025-2026 年:四个产品共存,各占细分。它们的差异不是"模型谁更强"——基础模型很多用同一个,差异在 UX 选择。

12.2GitHub Copilot · 极简渐进 + IDE 原生

Copilot 是这个赛道的"祖师",2021 年定义了"AI 代码补全"的产品形态。

  • 介入级别:L1(inline 补全)→ L2(Copilot Chat)→ L3(Workspace + Agent,2024 后)
  • 用户群:纯工程师,IDE 用户
  • 撤销机制:Esc 拒绝补全 / git 兜底
  • 验证机制:运行用户原生 IDE 的 build / test
  • 核心 UX 选择:不抢戏——AI 是 IDE 的辅助层,不是主角。灰色字补全 + Tab 接受这套基础范式至今没变
i 核心 insight

Copilot 的"不抢戏"是它能存活的核心——它没改 IDE 工作流,只是在缝隙里加 AI。这种"渐进式 AI 化"是企业用户最容易接受的路径。启示:面向已有工作流的用户,AI 应该作为"增强",不是"替代"。

12.3Cursor · 多介入级别可切换

Cursor 的核心创新不是某个功能,是"让用户在 3 个 AI 介入级别间无缝切换"(我们在第 4 章详细讨论过)。

  • 介入级别:Tab(L1)/ Composer(L2-L3)/ Agent(L3-L4)三梯度
  • 用户群:工程师 + 部分半工程师
  • 撤销机制:git diff 完整集成 + accept/reject 每个 hunk
  • 验证机制:用户自跑 IDE / terminal
  • 核心 UX 选择:把 IDE fork 后重做,完全围绕 AI 重设计交互——这是高风险高回报的策略,赌"AI 工作流足够不同,值得专用 IDE"
i 核心 insight

Cursor 的成功验证了"AI-native 工具有市场"——用户愿意离开 VS Code 用一个"为 AI 重设计"的 IDE。但代价是: Cursor 用户基本只能用 Cursor,不能切回 VS Code(因为 Cursor 特有的功能 VS Code 没有)。这是 vendor lock-in。启示:做 AI-native 产品要敢"重做"工作流,不要怕"破坏现有习惯"——但要承诺持续投入。

12.4Bolt · 把"app 生成"做成消费品

StackBlitz Bolt 把目标用户从"工程师"拉到"任何会描述需求的人"。介入级别直接跳到 L4。

  • 介入级别:L4(给一个 prompt,出端到端 app)
  • 用户群:产品经理 / 设计师 / 创业者(可以读代码但写不深)
  • 撤销机制:Checkpoint 系统(每次大改一个 checkpoint,可回滚)
  • 验证机制:右侧实时预览 + 一键 deploy
  • 核心 UX 选择:把"看代码"降权,把"看结果"升权——大多数用户不需要懂代码,看 app 跑起来即可
✕ Bolt 的两难

Bolt 想服务"非工程师",但当 AI 生成的 app 出 bug 时,非工程师无法 debug——他们只能让 Bolt 自己修,修不好就卡住。这是当前 prompt-to-app 类产品的共同瓶颈:目标用户没有"兜底能力"。启示:做 L4-L5 产品时,要预设"AI 卡住时用户该怎么办"——要么提供 fallback 到人,要么给"逃逸到工程师"的入口。

12.5Lovable · UI 优先 + 强 figma 感

Lovable 跟 Bolt 同期出现,选了一条更"设计师友好"的路径——UI 优先,后端可选。

  • 介入级别:L3(给一个 UI 描述,出可交互前端;后端可选)
  • 用户群:设计师 / UI/UX 工作者 / 产品经理
  • 撤销机制:版本历史 + section-level edit
  • 验证机制:右侧实时 web 预览
  • 核心 UX 选择:把"UI 调整"做成 Figma 般的精细操作——可以选中某个元素让 AI 单独改它,而不是整页重生
i 核心 insight

Lovable 的差异化在"精细到元素级别的 AI 编辑"——它假设设计师不需要"重生整页",而是"我喜欢这个布局,但这个按钮改成红色"。这种"局部编辑"的颗粒度是它跟 Bolt 的关键差异。启示:做设计类 AI 工具,"局部编辑"比"整体重生"更重要——设计师的工作流就是局部精调。

"AI 辅助生产力"不是一个赛道,是 4 个不同的赛道:AI 增强既有工作流(Copilot)、AI-native 重做工作流(Cursor)、AI 端到端产物(Bolt)、AI 精细局部编辑(Lovable)。每一个赛道服务不同用户、有不同的护城河。没有谁能"全要"。

12.7给千图AI 的 3 条启示

千图AI 不是 IDE,但工作流设计思维高度可借鉴。

✓ 从 4 个 DevTools 提炼,给千图AI 的 3 条启示
  1. 明确"介入级别定位"。Cursor 的 3 梯度切换给了我们启发:千图AI 是否要做"快速生图(L1)→ 多张备选(L2)→ 系列设计自动化(L3-L4)"的梯度?用户能在不同任务复杂度下选合适的 AI 介入级别。
  2. 学习 Lovable 的"局部编辑"。当用户生成一张图喜欢 70%,他要的是"改其中的 30%",不是"整张重生"。Inpaint / 局部重绘的 UX 入口必须做得跟"重生"按钮一样显眼——这是设计师的核心工作流。
  3. 不要走 Bolt 的"完全自动化"陷阱。给用户"逃逸到精细控制"的入口——当 AI 不达期望时,用户能切到"自己拉滑块 / 改 prompt 关键字 / 局部 inpaint"。L4 不能没有 L1-L2 的退路。

12.8本章小结

4 款 AI 辅助生产力工具不是"竞品",是4 个不同介入级别的精准对应。它们共同证明:AI 产品 UX 的第一个设计决策,是"我的产品在 L1-L5 哪个级别"——级别不同,工作流、用户群、撤销机制、验证方式都不同。
