Prompt Caching · 把账单降到 1/10 Production-Grade Cost Control

前面九章把模型怎么想、Agent 怎么跑、prompt 怎么写都讲完了——到这一章换个维度，讲一个不浪漫但要命的东西：钱。任何把 Claude 接到生产环境的人，第一周就会被账单吓一跳。Anthropic 在 2024 年发布的 Prompt Caching 是把这笔账单砍到 1/10 的核心机制，本章拆开它的工作原理、五条使用规则、跟 Extended Thinking 的成本陷阱，以及三种最常见的误用。

▼ 三十秒看懂这一章

Prompt Caching 干一件事：把 prompt 里重复出现的前缀（长 system prompt、工具定义、检索片段）缓存在 Anthropic 服务端，下次请求只算「新增部分」的 token，缓存命中部分按原价 10% 计费。听上去美好，但有五条死规则——必须从消息开头开始、最小 1024 token、5 分钟 TTL、缓存创建反而比原价贵 25%、要显式打 cache_control。开对了月账单从 10 万降到 1 万；开错了反而比不开还贵。这一章的判断：caching 不是「越多越好」，是「热路径上才开」。

故事 / 2024 年 8 月 14 日 · Anthropic 发布 Prompt Caching beta

2024 年 8 月 14 日，Anthropic 在官方博客发了一篇标题平淡的公告：《Prompt caching with Claude》。文章本体只有几屏，但底下挂了三张定价表和一个 5 分钟 TTL 的说明 [Anthropic Prompt Caching 2024]。这一晚，北美的 AI 工程师群组开始疯转——因为有人算了一下，他们公司基于 Claude 跑的客服 Agent，每条对话要附带一份 12000 token 的 SOP 手册，每天 50 万条对话，光这一块的 token 成本就是六位数美元。Caching 一开，这部分降到原来的 10%，公司一个月省的钱够再雇两个工程师。

后续几周，社区开始反过来吐槽——一些人开了 caching 之后账单反而更贵了。原因是他们的应用场景是「一次性长 prompt 查询」，每次 prompt 都不一样，缓存写入费 1.25 倍照付，命中率 0%，纯亏 25%。Anthropic 在文档里加了一段醒目的警告：「Caching is only economical when the cached prefix is reused.」（只有缓存前缀被复用，caching 才划算。）

到 2025 年中，Anthropic 又加了一个 1 小时 TTL 的付费选项，把 5 分钟之外的「中频复用」场景也覆盖进来。同期 Skilljar Academy 上线的《Building with the Claude API》课程，把 prompt caching 列为「生产环境必学的三个 API feature 之一」（另两个是 streaming 和 batch processing）[Anthropic Academy 2025]。这一章要讲清楚的，就是这套机制每个旋钮的意义，以及怎么判断要不要拧。

10.1机制 · 缓存的不是答案，是 prefix 计算 What Actually Gets Cached

第一个要纠正的误解：Prompt Caching 不是缓存「答案」——你问同样的问题，Claude 仍然要重新生成回答。它缓存的是 prompt 前缀经过模型 transformer 层之后的中间计算状态。下次请求带着同样的前缀过来，这部分计算可以跳过，只需要对新增的「后缀部分」做计算。

从计费视角看更直观：一次请求的 token 成本现在分成三段——缓存写入(cache write)、缓存命中(cache read)、新增 token。Sonnet 4.5 / Opus 4.7 当前的定价大致是：缓存写入 = 原价 × 1.25（贵 25%）；缓存命中 = 原价 × 0.10（便宜 90%）；新增 token = 原价（不变）[Anthropic Pricing 2026]。

一个具体例子

假设你给 Claude 一个 10000 token 的 system prompt（里面是公司 SOP）+ 一条 500 token 的用户问题，Claude 生成 1000 token 答案。一天调 1000 次，system prompt 每次都一样：

方案第 1 次成本第 2-1000 次每次成本当天总成本（相对）
不开 caching10000 输入 + 1000 输出10000 输入 + 1000 输出100% 基准
开 caching10000 × 1.25（写入）+ 1000 输出10000 × 0.10（命中）+ 500 输入 + 1000 输出约 15% — 省掉 85%

第一次请求贵了 25%，但从第二次开始，前缀那 10000 token 只按 10% 计费，日均下来节省接近 85%。规模越大、前缀越长、复用频率越高，省得越多。这是为什么客服 Agent、长文档问答、多轮对话这三个场景是 caching 最大的受益者。

i 跟 GPU 上的 KV cache 不是一回事

很多人看到 caching 第一反应是「这不就是 transformer 推理时本来就有的 KV cache 吗」——不是。GPU 上的 KV cache 是单次推理内部的状态复用，请求结束就丢；Anthropic 的 Prompt Caching 是跨请求的服务端持久化，把前缀的中间表示存在服务端 5 分钟（或 1 小时），让多个独立请求复用。两者发生在不同的层次，机制不同、生命周期不同、计费方式不同。把它们混为一谈，会导致对成本和延迟的预估全错。

10.2五条死规则 · 必须先背下来 The Five Hard Rules

Caching 的所有踩雷，本质都是不熟悉这五条规则。先把五条一次摆全，后面几节再逐条展开：必须从消息开头起、最小 1024 token、默认 5 分钟 TTL、首次写入反而贵 25%、要显式打 cache_control。

RULE 1
从开头起
缓存的是前缀 · 必须从 system 或第一条 user 开始 · 中间断了缓存就废
RULE 2
≥ 1024 tokens
Sonnet / Opus 最小 1024 · Haiku 最小 2048 · 小于这个不缓存
RULE 3
5 分钟 TTL
默认 5 分钟没人调用就过期 · 可付费延长到 1 小时
RULE 4
写入费 1.25×
首次写入贵 25% · 命中后才便宜 · 一次性请求开了反而亏
RULE 5
显式 cache_control
不会自动缓存 · 要在 message 里打 cache_control 字段 · 最多 4 个 breakpoint

规则一深讲：为什么必须从开头起

Prompt Caching 是基于前缀匹配的——服务端拿到一个请求，会从消息开头开始逐 token 比对历史缓存，一旦发现不一致就停止匹配，后续 token 全部按新输入算。这意味着：

  • ✅ 改 prompt 末尾不影响缓存 — 前面的 system + tools + 历史对话仍然命中，只有新增部分按原价
  • ❌ 改 prompt 开头一个字，后面所有缓存全废 — 哪怕只换了一个空格，从那个 token 之后全部按原价重算
  • ❌ system prompt 里塞个时间戳就毁掉 — "当前时间: 2026-05-27 14:32:01" 每次都不一样，放在前面就 0% 命中

这条规则决定了 prompt 的结构必须重排：稳定的放前面，易变的放后面。Anthropic 推荐的标准布局是 system prompt → tools → 长文档/RAG 片段 → 对话历史 → 当前 user message，稳定性逐段递减。

规则三深讲：5 分钟 TTL 的真实含义

「5 分钟」不是从首次写入算 5 分钟就过期，而是从最后一次命中算 5 分钟——只要 5 分钟内有请求命中，TTL 就会刷新延长。这意味着一个高频被用的缓存可以无限延寿，只要每 5 分钟内至少有一次请求。但一个偶尔被用的缓存（比如每 10 分钟一次）就会反复过期、反复写入、反复付 1.25 倍写入费，得不偿失。

1 小时 TTL 是 2025 年加的付费选项，适用于中频复用场景——比如一个内部知识库 Agent，工作时段每小时被问几次。多付的钱要换算一下，通常超过 4-5 次复用就划算，低于这个还不如默认 5 分钟 [Anthropic Extended Cache 2025]。

✕ 最常见的初学者错误 · 在 system prompt 开头写时间戳 / 用户名

「亲爱的 [user_name]，今天是 [date]——」这种模板化开头，如果时间或用户名每次不一样，等于把缓存击穿在第二个 token 上。所有动态变量必须从前缀里挪到后面去，放到当前 user message 的末尾，或者作为变量注入到 system 的最后一段（而不是开头）。结构上，你要把 prompt 想成一个三明治：稳定的底面包（system + tools）→ 半稳定的夹心（对话历史）→ 易变的顶面包（当前 user message），caching 从底面包开始，沿稳定的部分一路命中，在易变的部分自然结束。

10.3什么时候开 · 三种高 ROI 场景 When Caching Pays Off

不是所有 Claude 应用都该开 caching。这一节给你三种「开了就赚」的标准场景，以及判断标准。

场景一 · 长 system prompt + 多轮对话

客服 Agent、教学助手、知识库问答——典型特征是 system prompt 写得很长（几千到上万 token），每个用户聊一聊就是十几二十轮。这种场景下，system prompt 在 5 分钟 TTL 内被命中几十次，caching 节省接近 90%。这是 caching 设计时的第一目标场景。

场景二 · 工具定义 + Agent 多步执行

用 第 8 章 讲的 Claude 工具调用搭 Agent，工具定义通常几千 token，一次 Agentic Loop 里要被读 5-20 次。给 tools 字段打 cache_control，工具定义部分按 10% 计费。Anthropic 在内部测 Claude Code 时，tools 的 caching 是显著的成本项。

场景三 · 长文档分段问答

RAG（第 11 章）场景：把检索到的长文档片段作为 context，用户连续问几个相关问题。文档片段在 5 分钟内被复用几次，caching 把这部分压到 10%。Anthropic 自己的 Projects 功能背后大概率用了类似机制。

场景四 · Few-shot 例子库

把几十个示例（input/output 对）放在 system 里给 Claude 学。这一段 few-shot 通常 5000-20000 token，如果调用频率高，caching 是必开。例子库越稳定、调用越频繁，caching 收益越大——这正是 第 9 章 推荐的 prompt 工程模式的天然搭配。

判断标准 · 一条简单公式

对于一段长度为 N token 的前缀，5 分钟内调用次数为 K，是否开 caching 的判断：

  • 不开 caching 的总成本 ≈ K × N × 原价
  • 开 caching 的总成本 ≈ N × 1.25 × 原价 + (K-1) × N × 0.10 × 原价
  • 令两者相等，解出临界 K ≈ 1.28

也就是说，理论上只要前缀在 5 分钟 TTL 内被命中超过 1-2 次，caching 就开始划算；超过 5 次，节省接近 80%；超过 20 次，几乎是「免费用」。所以判断标准简化成一句话：「这段前缀在 5 分钟内会不会被重用两次以上？」会，就开。

10.4什么时候不开 · 四种反场景 When Caching Hurts You

前一节是「开了就赚」，这一节是反过来——四种「开了亏钱」的场景，工程上要主动绕开。

反场景一 · 一次性查询

用户问一个问题，Claude 回一个答案，会话就结束了。这种场景下没有「下一次」复用，缓存写入费 1.25 倍照付，命中率 0%，纯亏 25%。批处理 / 一次性脚本 / 个人偶尔问问题，默认不开 caching。Anthropic 提供的 batch API 是这类场景的正解（50% 折扣，但是异步），跟 caching 是两条不同的优化路径，不要混。

反场景二 · prompt 短于 1024 token

这是规则二的硬限制——Sonnet / Opus 必须至少 1024 token 才能缓存（Haiku 是 2048）。短 prompt 即使打了 cache_control，服务端也会忽略，直接按原价计费。所以「小型工具调用」「短 system + 短 user」这种场景下，caching 字段是空操作，既不省也不亏（但容易让你误以为开了）。判断方法：看 API 返回的 usage 里有没有 cache_creation_input_tokens 大于 0，如果是 0，说明没真的缓存上。

反场景三 · 前缀变动频繁

有些应用场景看起来像高频复用，但 prompt 前缀其实每次都微变——比如：

  • ❌ system 开头写「现在是 [时间戳]」——每次时间不一样，缓存击穿在第二 token
  • ❌ system 里嵌入「用户 ID: [user_id]」在前面——每个用户都不一样，命中率 0%
  • ❌ tools 字段每次按用户权限动态裁剪——工具列表序列化后 byte 级不一致

这些场景理论上能复用（同一用户），但实际上跨用户、跨请求的随机性破坏了前缀稳定性。修复办法：把动态变量从前缀里挪到 user message 末尾（规则一的应用），或者按用户 ID 分桶维护多个缓存。

反场景四 · 复用频率低于 5 分钟 TTL

每 10 分钟才被调一次的低频 prompt——缓存每次都过期，每次都重新写入付 1.25 倍，命中率近 0%。这种情况下不如不开 caching，或者升级到 1 小时 TTL（但 1 小时 TTL 的写入费更贵，需要算清楚临界复用次数）。判断方法：在生产环境跑一周，看 cache_read_input_tokens / cache_creation_input_tokens 的比值——低于 1:1 说明命中率不到一半，要么换 TTL 要么关 caching。

✕ 反共识 · 「缓存命中率 100% 是最优」 — 错

命中率是个相对指标，绝对成本节省才是目标。一个调用频率极低的 prompt 哪怕做到 100% 命中，省下来的钱可能比写入费多付的还少。反过来，一个高频调用的 prompt 哪怕命中率只有 60%，绝对节省也可能是六位数美元。不要追求命中率的虚荣指标，看月账单的实际下降。Anthropic 在 API console 里提供了 cache hit 和实际节省的统计——按那个看，不要按直觉。

10.5Extended Thinking 的成本陷阱 · The Hidden Bill of Reasoning

Extended Thinking 是 Claude 4 系列引入的「显式推理」模式——开了之后，Claude 会先输出一段不可见的 thinking 内容（用 <thinking> 块包裹），再生成最终回答。在复杂推理任务上质量显著提升，但成本上有一个所有人初次接触都会踩的陷阱：thinking tokens 也算 token，而且不能被 cache。

Anthropic 在 Extended Thinking 的发布博客里把这一点写得很清楚，但很多人没注意：thinking 内容是模型每次重新「想」出来的，具有不可复现的随机性，所以服务端不缓存这部分。每次请求，thinking tokens 都按原价计费，数量还往往很大（简单问题几百，复杂任务几千到上万）[Anthropic Extended Thinking 2025]。

组合陷阱：Extended Thinking + 长 prompt

这两个 feature 单独开都没问题，组合开就有惊喜——一个典型场景：

配置输入 tokenthinking token输出 token单次成本（相对）
普通模式 + 短 prompt50005001×
普通模式 + 长 prompt 不开 cache100000500约 10×
普通模式 + 长 prompt 开 cache（命中）10000（按 10% 计）0500约 2×
Extended Thinking + 长 prompt 开 cache10000（按 10% 计）5000（原价）500约 7×

最坑的是两个一起开：你以为开了 caching 把长 prompt 砍到一成，实际上 thinking 那 5000 token 是按百分之百原价算的，反而成了主要成本项。Extended Thinking 不是「免费的更聪明」，是「贵几倍的更聪明」。

i 怎么用 Extended Thinking 不被账单咬

三条经验：一，只在真正需要复杂推理的请求上开——简单事实查询、格式化输出、模板化任务别开。二，配合 thinking budget 参数（API 里能限制 thinking 最大 token 数），给个上限避免它失控漫游。三，先用 Sonnet 4.5 + Extended Thinking 试，效果不够再切 Opus 4.7——Opus 的 thinking 单价更高，踩雷代价更大。Anthropic 文档里专门有一节叫《Cost considerations for Extended Thinking》，生产前必读。

10.6实战 · Python SDK 标准用法 The Code You Actually Run

理论讲完，落到能直接跑通的写法上，核心就一件事：cache_control 怎么打到 system prompt 和 tools 上。做法基于 Anthropic Python SDK 标准用法，跟官方文档保持一致。

场景：一个产品咨询 Agent,system prompt 里塞了一份 8000 token 的产品手册，带两个工具（查库存、查订单），需要被频繁调用。

from anthropic import Anthropic

client = Anthropic()

# 长 system prompt — 稳定不变 — 打 cache_control
SYSTEM_PROMPT = """你是 ACME 公司的产品咨询助手。

[此处省略 8000 token 的产品手册:产品规格、FAQ、退换货政策……]
"""

# 工具定义 — 稳定不变 — 也打 cache_control
TOOLS = [
    {
        "name": "check_inventory",
        "description": "查询某个 SKU 当前库存数量",
        "input_schema": {
            "type": "object",
            "properties": {
                "sku": {"type": "string", "description": "产品 SKU"}
            },
            "required": ["sku"],
        },
    },
    {
        "name": "check_order",
        "description": "根据订单号查询订单状态",
        "input_schema": {
            "type": "object",
            "properties": {
                "order_id": {"type": "string"}
            },
            "required": ["order_id"],
        },
        # 最后一个工具上打 cache_control — 一次性覆盖前面所有 tools
        "cache_control": {"type": "ephemeral"},
    },
]

def ask(user_message: str):
    response = client.messages.create(
        model="claude-opus-4-7",
        max_tokens=1024,
        system=[
            {
                "type": "text",
                "text": SYSTEM_PROMPT,
                # 关键:在 system 上打 cache_control
                "cache_control": {"type": "ephemeral"},
            }
        ],
        tools=TOOLS,
        messages=[
            {"role": "user", "content": user_message}
        ],
    )

    # 检查缓存有没有真的生效
    usage = response.usage
    print(f"cache_creation_input_tokens: {usage.cache_creation_input_tokens}")
    print(f"cache_read_input_tokens:     {usage.cache_read_input_tokens}")
    print(f"input_tokens (new):          {usage.input_tokens}")

    return response

# 第一次调用 — cache_creation_input_tokens 会很大(约 8000+)
ask("你们的退换货政策是什么?")

# 第二次调用(5 分钟内)— cache_read_input_tokens 会很大,cache_creation 应该是 0
ask("订单 ORD-12345 现在什么状态?")

三个细节值得点出

  • cache_control 打在最后一个要缓存的内容块上 — 它的作用是「从这里及之前的内容都缓存」，不是「只缓存这一块」。所以 tools 列表里打在最后一个工具上，等于覆盖全部 tools
  • type 目前只有 ephemeral — 5 分钟 TTL 默认值。如果要 1 小时 TTL，加 "ttl": "1h"（需要付费层级支持）
  • 最多 4 个 breakpoint — 一次请求里最多打 4 个 cache_control，通常的分布是：system 一个、tools 一个、长文档一个、对话历史一个。超过 4 个会报错
✓ 武器 · 生产环境 caching 健康度自检

每条请求的 usage 里有四个数：input_tokens（新增）、cache_creation_input_tokens（写入）、cache_read_input_tokens（命中）、output_tokens（输出）。把这四个数打到 metrics 里，看一周的趋势——cache_read / (cache_read + cache_creation) 这个比值越接近 1，说明命中率越高，caching 越赚。如果一周下来这个比值低于 0.5，说明你的复用模式有问题，要么是 prompt 变动太频繁，要么是调用频率低于 5 分钟 TTL，得回去查规则一和规则三。

10.7三个反共识 · 别被直觉带偏 Three Counterintuitive Truths

Caching 这套机制有几个表面合理但其实错的直觉，拎出来单独讲。读完这一节，你应该能在团队讨论时挡住几次错决策。

反共识一 · 「命中率越高越好」

前面 10.4 节已经点过：命中率是相对指标，绝对成本节省才是目标。一个被调一万次但每次复用 100 次的高频 prompt，即使命中率「只有」90%，绝对节省是六位数；一个被调十次但每次都复用的低频 prompt，即使命中率 100%，绝对节省只有几美分。看 dashboard 时用美元金额排序，不要用命中率排序——这一条对团队 review 的优先级影响很大。

反共识二 · 「TTL 越长越好」

1 小时 TTL 听上去比 5 分钟 TTL 好，但前者要额外付费（写入费更贵）。判断公式跟开不开 caching 同理：1 小时内的复用次数，要够偿付 1 小时 TTL 比 5 分钟 TTL 多付的钱。Anthropic 的建议是先用默认 5 分钟跑两周，看 metrics，只有那些「确实每 10-30 分钟被调用一次」的低频但稳定模式才升级到 1 小时——其他的要么保持 5 分钟，要么干脆关掉 caching。

反共识三 · 「Prompt Caching 等同于 KV cache」

10.1 节的 callout 已经点过，这里再单独强调：这是 API 层抽象，不是 GPU 上的 KV cache。两者的生命周期、计费、可见性、控制方式都不同。一个直接后果是：你不能假设 caching 命中会带来「延迟显著下降」——延迟改善是有的（prefill 阶段计算量减少），但不像直觉里 KV cache 那样大；主要收益是成本，不是延迟。如果团队 review 时有人说「开 caching 是为了让响应更快」，这话错了——是为了让账单更小。

✕ 还有一条反共识 · 「写入费 1.25 倍不算什么」

这个 25% 的写入溢价，在「调一次就丢」的场景下是纯亏的 25%——不是「省一点」是「贵 25%」。很多团队上线时图省事，在所有请求上都打 cache_control，以为不命中也无所谓——其实每条不命中的请求都比不开 caching 贵 25%。如果你的应用里有一半请求是一次性查询，这部分等于持续在多付钱。判断是否该打 cache_control 的标准只有一个：「这段 prefix 在 5 分钟内大概率会被复用两次以上吗？」不确定就先别打，跑一段时间看 metrics 再决定。

10.8这一章给你的判断武器 Takeaways

读完这章，合上书，你应该带走以下三个能直接拿去用的判断武器。

✓ 武器一 · 把 prompt 重排成「稳定→易变」三明治结构

system + tools + 长文档（稳定底层）→ 对话历史（半稳定）→ 当前 user message + 动态变量（易变顶层）。所有动态变量（时间戳、用户名、user_id）都从前缀里挪到末尾。这一条做完，caching 的命中率会从 0% 跳到 80% 以上——其他所有优化都基于这条结构。如果项目里目前 system prompt 开头还有「现在是 [date]」，先回去改这个，改完再谈别的。

✓ 武器二 · 用「5 分钟内复用 ≥ 2 次」做开 / 不开判断

不是所有 prompt 都该开 caching。判断标准简化成一句：这段 prefix 在 5 分钟内会不会被复用两次以上？会，开；不会，关。批处理任务、一次性查询、用户偶尔提问类应用，默认关 caching——开了反而每次多付 25%。第 1 章讲过 LLM 是个概率引擎，这里再加一句：caching 也是个概率游戏——你赌的是 prefix 会被复用，赌赢省 90%，赌输亏 25%。下注前看清牌。

✓ 武器三 · Extended Thinking 跟 caching 不是叠加优化，thinking tokens 不能 cache

看到「Extended Thinking + Prompt Caching 双开」的项目方案，先问一句：thinking 的 token 大概多少？如果是每条几千上万，那 caching 省下来的 90% 大概率被 thinking 的原价抵消掉。Extended Thinking 是「贵几倍的更聪明」，不是「更聪明的免费午餐」。要么用 thinking budget 限上限，要么只在真正需要复杂推理的请求上开——其他场景关掉。

Prompt Caching 讲完，你掌握了 Claude API 生产环境最有杠杆的成本优化武器。但 caching 解决的是「重复部分」的成本，prompt 里另一大块——动态注入的检索内容——还没讲。检索内容怎么组装、怎么排序、怎么不毒化模型？这是 第 11 章 RAG 要解决的问题。RAG 跟 caching 是一对天然搭档：RAG 决定塞什么进去，caching 决定塞进去的部分怎么省钱。两者合在一起，才是生产级 Claude 应用的完整工程图。
