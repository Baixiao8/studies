Projects / Artifacts / Skills / Connectors 桌面端四部件 The Desktop Stack

第 5 章讲完三件套(Claude.ai · Claude Code · API),你可能以为产品篇就结束了。但 Claude 桌面端真正让长期协作变成可能的,是横跨这三个壳的四个部件:Projects 长期上下文容器 / Artifacts 持续编辑容器 / Skills 团队知识包 / Connectors 外部世界接口。理解这四个部件怎么配合,才理解 Claude 从「单次对话工具」升级成「长期协作平台」的工程实现。

▼ 三十秒看懂这一章

Anthropic 把「让 Claude 记住你」这件事拆成四个独立部件,而不是做一个 OpenAI Memory 那样的全局记忆。Projects 装长期上下文,Artifacts 装持续输出物,Skills 装重复工作流,Connectors 装外部数据接口。四个部件互补不替代——你可以只用其中一两个,也可以全部组合。这种「组件式记忆」是 Anthropic 跟其他厂商最不同的产品哲学:不替你决定记什么,把记忆的层级显式摆出来,让你自己装配。

故事 / 2024 年 6 月 25 日 · Claude 3.5 Sonnet 发布日

2024 年 6 月 25 日,Anthropic 发了一篇看起来不大的产品公告。标题里有 Claude 3.5 Sonnet,所有人盯着模型版本号——比上一代 Opus 还强、价格只有五分之一。但博客最后藏着一段几乎被淹没的话:「我们同时推出 Artifacts,一个让 Claude 生成的代码、文档、图表能在对话旁边的窗口里持续存在的容器。」[Anthropic 2024c]

这句话当时没几个人在意。直到一个月后,Reddit 和 X 上开始出现一类截图:用户拿 Claude 写了一个 React 组件,Claude 把组件放进右侧的 Artifact 窗口里,用户对着窗口说「这个按钮颜色改深一点」「再加个 hover 动效」「现在把它打包成一个可下载的 HTML」——每一句对话都在改同一个 Artifact,而不是每次都重新生成一段新代码。这种「持续在同一个对象上协作」的感觉,跟之前所有 Chat 形态的 AI 都不一样。

三个月后的 9 月 30 日,Anthropic 上线 Projects——把对话、文件、自定义指令打包成一个长期容器,Claude 跨多次对话保持记忆 [Anthropic 2024d]。又过半年,2025 年 5 月 Connectors 上线,Claude 能直接接 Gmail / Calendar / Linear / Notion / GitHub [Anthropic 2025a]。再过五个月,2025 年 10 月 Agent Skills 上线,SKILL.md 成为新的团队知识载体 [Anthropic 2025b]。

回头看,这四个部件不是同一时刻设计出来的,是 Anthropic 在 18 个月里一点一点把「Claude 怎么从对话工具变成协作平台」想明白的。它们不替代彼此,各管一段——这就是 Anthropic 产品观最不同的一点:不做全局抽象,做正交部件。

6.1为什么需要四个部件 · Why Four, Not One

最直观的疑问是——为什么不做一个「Claude 自动记住所有」的全局记忆,要拆成四个?回答这个问题,要回到第 1 章讲过的工作记忆性质。

LLM 的工作记忆是上下文窗口,200K tokens 是硬上限。「让 Claude 跨对话记住你」这件事在物理上不可能——下一次对话开始时,上下文窗口被清空,模型完全不记得你是谁。所有看起来像「记忆」的产品功能,本质都是把上次的关键信息自动塞回这次的上下文窗口。这是个工程问题,不是模型问题。

一旦你接受「记忆是工程,不是能力」,下一个问题就来了:每次塞什么进去?这才是 Anthropic 跟 OpenAI 在产品哲学上分叉的地方。

✕ 反共识 · OpenAI Memory vs Anthropic 四部件

OpenAI 的 Memory 是一个全局开关——开了之后,ChatGPT 在所有对话里都会自动记下「关于你」的事实,下次自动召回。听起来很方便,但代价是用户失去了对「记忆什么」的控制权。它可能记下你随口的一句吐槽,也可能漏掉你认真想保留的偏好。Anthropic 选了相反的路:把记忆显式拆成四种容器,每种容器有明确边界。Projects 装这个项目相关的长期上下文,Artifacts 装这次任务的持续产物,Skills 装你团队的重复工作流,Connectors 装实时外部数据。这种拆法的代价是用户要学习四个概念,但好处是你永远知道 Claude 在「记什么、从哪记的」——可控性大于自动化,这就是 Anthropic 产品观的底色。

四部件怎么分工

把四个部件按「时间维度」和「来源维度」摆开,就能看清它们的正交关系:

PROJECTS
长期上下文
跨对话沉淀 · 项目级背景
ARTIFACTS
持续输出物
本次任务 · 反复迭代同一对象
SKILLS
重复工作流
跨项目复用 · 团队知识沉淀
CONNECTORS

这四个部件不是层级关系,是四个独立的存储 / 调用机制,可以单独用、可以叠加用。一个最丰富的 Claude 使用场景可能同时用上所有四个:在一个 Projects 里、对着一个 Artifact 干活、调用了几个 Skills、连接了几个 Connectors 拉取数据。但你也可以只用 Artifacts、不开 Projects,完全合法。

6.2Projects 项目 · 把上下文打包 Long-Term Context Container

Projects 是 Claude.ai 在 2024 年 9 月推出的第一个「长期容器」。它的核心承诺只有一句:在这个项目里的对话,共享同一份长期上下文。每开一次新对话,Claude 自动把项目里的文件、自定义指令、过往关键对话塞回上下文窗口的开头。

不要把 Projects 想成「聊天记录文件夹」——那是它最浅的用法。Projects 的真正杀手锏在于它由三层东西组成,而每一层都对应一种工程意图。

Projects 的三层结构

层级装什么对应的工程意图
System Prompt(项目级指令)「你是一位资深康复理疗师,回答时要分循证等级,术语首次出现给中英对照」把「Claude 在这个项目里应该是谁」固化下来,避免每次对话都重复声明
Project Knowledge(项目文件)PDF 文档、Markdown 笔记、CSV 数据、过往关键代码片段把跨对话需要反复引用的资料放进来,Claude 在每次对话开头自动检索
Conversations(对话历史)这个项目下所有过往对话的标题和摘要避免「上次跟你聊的 XX 你忘了吗」这种事,Claude 能召回过去做过的决定

三层加起来,效果是什么?举个具体场景。你在写一本运动康复手册,开了一个 Projects 叫「跑步康复书 v4」。System Prompt 写了术语规范、风格基调、目标读者画像。Project Knowledge 里放了 12 篇核心论文 PDF、3 份已经写完的章节、1 份大纲。每次开新对话写一个新章节,Claude 自动知道:你是谁、你想写给谁、术语怎么处理、前面写过什么、当前章节在大纲里的位置。所有「描述上下文」的胶水时间被压缩到几乎为零——这跟第 2 章讲的 Agent 形态优势是一回事,只是 Projects 解决的是「跨对话级别」的胶水。

✕ 反共识 · 大多数人在用 Projects 的方式是错的

Anthropic 自己的数据显示:有 60% 以上的 Projects 用户,把 Projects 当成「聊天记录归档夹」——只是把相关对话归在一起,没配 System Prompt,没传 Project Knowledge [Anthropic 2025c]。这等于花钱买了一辆跑车,只用来从家开到便利店。Projects 的真正杠杆在三层加起来的复合效应——没有 System Prompt 的 Projects,跟普通对话窗口没差别;没有 Project Knowledge 的 Projects,Claude 还是只能看你这次打的字。要拿到 Projects 的全部价值,三层都得配。

Projects 的工程实现

Projects 在底层做了什么?Anthropic 的工程博客 [Anthropic 2025c] 给出了一个简化的描述:

  • 每次在 Projects 里发起新对话,系统先把 System Prompt 塞到上下文窗口最前面
  • 然后用 RAG(检索增强生成)从 Project Knowledge 里捞出跟当前问题最相关的几个片段,塞进窗口
  • 再附上过往对话的摘要索引(不是全文,是摘要 + 标题),让 Claude 知道「这个项目讨论过哪些话题」
  • 最后才接你这次的输入

这就是为什么 Project Knowledge 不能塞太多东西——RAG 检索质量在文件数 100 个以下表现最好,超过 200 个后开始稀释。Anthropic 官方推荐每个 Projects 的 Knowledge 总量控制在 30-50 个文件以内,单个文件不超过 30 万 tokens [Anthropic Docs 2026]。

6.3Artifacts 工件 · 内容跟对话分离 Persistent Content Container

Artifacts 是 2024 年 6 月跟 Claude 3.5 Sonnet 一起发布的功能,设计哲学只有一句:AI 生成的内容不应该被埋在对话流里,应该有自己的容器,可以被持续编辑。

这听起来像个小改动,但反映了一个深层的产品判断——Anthropic 在 Artifacts 发布博客里写得很清楚:「在 ChatGPT 时代,我们让用户把所有东西都装进对话泡泡。但当用户开始用 Claude 写代码、写文档、做图表时,这些产物天然有自己的生命周期,跟对话的颗粒度不一致。Artifacts 是把这两层分开」[Anthropic 2024c]。

Artifacts 解决的具体问题

在没有 Artifacts 之前,用 Claude 写一个网页是这样的:

  • 你说「写个登录页」 → Claude 在对话里输出 200 行 HTML
  • 你说「按钮改成蓝色」 → Claude 重新输出 200 行 HTML,只是改了一行
  • 你说「再加个忘记密码链接」 → Claude 重新输出 200 行 HTML,还得你自己对比哪里变了
  • 滚 10 屏对话后,你彻底不知道现在哪个版本是最新的

有了 Artifacts 之后:

  • 你说「写个登录页」 → Claude 把代码放进右侧 Artifact 窗口
  • 你说「按钮改成蓝色」 → Claude 直接原地编辑那个 Artifact,你看到的是 diff 而不是重抄一遍
  • 你说「再加个忘记密码链接」 → 同一个 Artifact,版本号 +1
  • 对话泡泡里只剩下「你跟 Claude 在聊什么」,产物本身在另一个窗口持续演化

Artifacts 支持的内容类型

代码与可执行内容

React 组件、HTML/CSS 页面、Python / JavaScript 脚本、SVG 图形——Claude 在 Artifact 窗口里直接渲染并可交互。一个 React 组件改完之后,你能在右侧立刻看到效果,不用复制到自己的环境里跑。

文档与结构化文本

Markdown 文档、Mermaid 图表、长文写作。Artifact 提供「编辑模式」和「预览模式」,你可以像用普通文档编辑器一样选中某段让 Claude 重写。

数据与可视化

表格数据、Chart.js 图表、D3 可视化。把一份 CSV 丢进对话,让 Claude 在 Artifact 里画图,图变成可下载、可分享的独立产物。

跨 Artifact 协作(2025 年新增)

2025 年中,Anthropic 上线了 Artifact 之间相互引用——一个 React 组件 Artifact 可以引用另一个 CSS Artifact,改 CSS 时组件自动更新。这把单文件 Artifact 升级成了「迷你工程」。

i 设计哲学 · 「容器」是 Anthropic 产品观的核心隐喻

注意 Artifacts、Projects、Skills 都被设计成「容器」——一个有边界、有版本、可独立操作的东西。这不是巧合。Anthropic 的产品观跟 OpenAI 最大的区别在于:OpenAI 倾向于把所有东西捏成一个无缝的对话流(Memory 也是 invisible 的),Anthropic 倾向于把每个抽象单独立成容器,让用户能看见、操作、组合。这种「容器化产品观」的代价是认知成本上升,好处是可控性、可解释性、可组合性大幅提升——这跟 Constitutional AI 的设计哲学(回看 第 1 章)一脉相承,都是「显式优于隐式」。

6.4Skills 能力包 · 团队知识装进 Claude Skill Packages

Skills 是 2025 年 10 月发布的功能,也是四部件里最年轻、被讨论得最热烈的一个。它的核心定义只有一句:一个 Skill = 一个文件夹 + 一个 SKILL.md。Claude 在合适场景下自动加载这个文件夹里的指令和资源。

这听起来跟 System Prompt 没差别。差别在三点:

  1. Skill 是「按需加载」的——只有 Claude 判断当前任务匹配 SKILL.md 描述时,才把 Skill 里的内容塞进上下文窗口。System Prompt 是「永远加载」的
  2. Skill 是「可复用 + 可分享」的——一个 Skill 可以跨 Projects 使用,也可以打包发给团队成员、放进公司知识库。System Prompt 绑死在一个 Projects 里
  3. Skill 可以包含「可执行资源」——SKILL.md 里可以引用 Python 脚本、模板文件、外部工具调用。Claude 加载 Skill 时把这些资源一起带进上下文

SKILL.md 长什么样

Anthropic 在 Skills 发布博客和 Skilljar Academy「Introduction to Agent Skills」课程里给的标准格式 [Anthropic 2025b][Anthropic Academy 2025]:

  • YAML frontmatter — 给 Skill 名字、描述、触发条件。Claude 用这段判断「当前任务要不要加载这个 Skill」
  • Skill 主体 — Markdown 写的指令,告诉 Claude 「遇到 X 场景时按 Y 步骤处理」
  • 引用的文件 / 工具 — 可以是同目录下的模板、脚本、配置;Claude 在执行 Skill 时会自动加载

一个真实例子:Anthropic 内部的 brand-guidelines Skill。它告诉 Claude:「写涉及 Anthropic 品牌的内容时,用这些官方色值、这套字体、这些视觉规范。」当用户在 Claude 里画一张涉及 Anthropic logo 的卡片时,这个 Skill 自动被触发,Claude 输出的设计直接符合品牌规范——用户不需要每次都重新解释「Anthropic 的颜色是什么」。这就是 Skill 的杠杆:把「团队反复要解释的事」沉淀成一个文件夹,放进 Claude 的工具箱,从此自动生效。

✓ Skills 的「双层加载」机制

Anthropic 在 Skilljar Academy 课程里讲过一个常被忽视的细节:Claude 加载 Skill 是分两步的。第一步,Claude 只看 SKILL.md 的 frontmatter(几十个字),判断要不要加载——这一步 token 消耗极小,所以你可以在 Claude 里挂上几百个 Skills 不影响性能。第二步,只有匹配的 Skill 才被加载主体内容——这时才消耗 token。这种「轻量索引 + 按需加载」的设计,让 Skills 实质上成了一个可无限扩展的知识库系统,而不是一堆挤占上下文窗口的指令。

Skills 的完整讲解会在 第 8 章「四把刀」里展开——包括怎么写一个高质量的 SKILL.md、怎么设计触发条件、怎么避免 Skill 之间冲突、Anthropic 自己用 Skills 替代了哪些过去靠工程团队维护的工作流。这一章先把它在「四部件」里的位置摆清楚。

6.5Connectors 连接器 · 接住外部世界 External Data Bridges

Connectors 是 2025 年 5 月上线的功能,本质很简单:Anthropic 官方维护的一组预置工具,让 Claude 能直接读 Gmail、Calendar、Linear、Slack、Notion、GitHub 等主流 SaaS 应用的数据。

这听起来像 ChatGPT 的 Plugins,但底层完全不同。Connectors 是 MCP 协议(Model Context Protocol)的「官方应用商店版」——Anthropic 把社区里成熟的 MCP server 包装成开箱即用的连接器,放在 Claude.ai 的设置面板里,用户一键开启即可。

Connectors 跟 MCP 的关系

层级是什么谁来做
MCP 协议Anthropic 2024 年底开源的工具调用协议,定义了 LLM 怎么跟外部工具通信开源标准,任何人可实现
MCP Server实现 MCP 协议的具体程序,比如「Gmail MCP Server」「Notion MCP Server」社区 + 厂商各自贡献,数百个已发布
ConnectorsAnthropic 在 Claude.ai 上预置的、官方维护的一批 MCP Server,用户一键开启Anthropic 官方挑选 + 包装

所以 Connectors 不是「新东西」,是 MCP 这个底层协议的「官方精选商店」。一个不在官方 Connectors 列表里的 SaaS 应用,只要有人写了对应的 MCP Server,你也能在 Claude Desktop 里手动接入——只是需要点配置工作。MCP 协议本身的完整讲解在 第 12 章。

主流 Connectors 一览

  • Google Workspace — Gmail、Calendar、Drive、Docs。Claude 能读邮件、查日程、改文档
  • Microsoft 365 — Outlook、Teams、SharePoint、OneDrive。企业用户用得最多
  • 项目管理 — Linear、Asana、Atlassian(Jira)、ClickUp、Monday、Notion。Claude 能查任务、写 PR、更新看板
  • 沟通 — Slack、Intercom。Claude 能读频道、查历史、回复客服工单
  • 开发 — GitHub、Datadog、PagerDuty。Claude 能查 PR、看监控、响应告警
  • 销售与运营 — HubSpot、Salesforce、Apollo、Outreach、Zoominfo

这个列表会持续增长。Anthropic 在 2025 年 11 月的更新里宣布,Connectors 已覆盖 70+ 主流 SaaS 应用,这个数字预计 2026 年突破 200 [Anthropic 2025d]。

i 一个被低估的细节 · Connectors 是「读」优先,「写」谨慎

大部分 Connectors 默认只开了「读」权限——Claude 能查 Gmail、能查 Calendar、能查 Linear,但不能未经确认就发邮件、删事件、关任务。要让 Claude 能写,你得在设置里单独开「Write」权限,而且每个写操作 Claude 都会在对话里明确告诉你「我要做 X,确认吗」。这是回到 第 2 章「监工模式」的工程实现——Anthropic 把「关键节点拍板」做进了 Connectors 的默认流程,而不是依赖用户记得提醒 Claude。

6.6四部件配合的真实场景 · When All Four Click

单独讲每个部件,你只能理解它们各自的能力边界。要看到它们的复合杠杆,得看一个把四个都用上的真实场景。

场景 · 一个产品经理的「项目周报」工作流

设想一个产品经理小李,她要每周交一份项目周报。在没有 Claude 之前,她的工作流是:打开 Linear 看本周完成的任务、开 Slack 翻团队动态、打开 Notion 看产品文档、在 Google Docs 里写周报、贴图、发给老板。前后 90 分钟。

用上 Claude 四部件之后,她的工作流变成:

  1. Projects — 她建了一个「项目 X 周报」Projects,System Prompt 里写了「我是项目 X 的 PM,我的周报要包含进度、风险、下周计划三个部分,语气直接、不要装饰词」。Project Knowledge 里放了项目 OKR、过往周报、利益相关方画像
  2. Connectors — 她开了 Linear、Slack、Google Drive 三个 Connector。Claude 能直接读这周的 Linear 任务、Slack 频道、Drive 文档
  3. Skills — 她装了团队内部一个叫「project-x-weekly」的 Skill,SKILL.md 写了周报的固定结构模板、措辞规范、必须包含的指标
  4. Artifacts — 她让 Claude 把周报草稿写进一个 Artifact,在 Artifact 里反复改,而不是在对话里来回滚屏

每周一早上 9 点,她在 Claude 里说一句「写本周周报」。Claude 自动:Skill 触发,加载周报模板;Connectors 拉数据,从 Linear 拿任务列表、从 Slack 拿团队消息、从 Drive 拿文档变更;Projects 提供长期背景,知道项目目标和风格;Artifact 生成草稿,小李对着 Artifact 改三轮,定稿。从 90 分钟压缩到 12 分钟——这就是四部件配合的杠杆。

6.7跟 OpenAI / Google 的产品哲学对比 · Different Philosophies

最后用一节横向对比 Anthropic、OpenAI、Google 三家在「让 AI 长期协作」这件事上的产品选择。这种对比能让你看清 Anthropic 选「四部件」不是技术限制,是产品哲学的主动选择。

厂商长期记忆方案外部接入方案哲学概括
AnthropicProjects(显式容器)+ Skills(团队知识)Connectors(精选 MCP)+ MCP 协议(开放)显式 · 可组合 · 用户可见
OpenAIMemory(全局自动记忆)+ Custom GPTsPlugins(已废弃) → GPT Actions(私有协议)隐式 · 自动 · 平台主导
GoogleGemini 跟 Workspace 深度耦合(自动调 Gmail / Drive)原生绑死 Google 生态,第三方接入弱耦合 · 垂直 · 生态封闭

这三种哲学没有对错,只有取舍。OpenAI 的「隐式自动」最省心,适合不愿意学习概念的普通用户;Google 的「生态耦合」最丝滑,前提是你已经在 Google 生态里;Anthropic 的「显式容器」最累但最透明,适合愿意为可控性付出学习成本的工程化用户——这跟 Anthropic 的整体用户画像高度一致(回看 第 5 章三件套的用户画像分析:Claude 的用户里程序员、知识工作者占比远高于 OpenAI)。

✕ 一个常被忽视的代价 · 「显式容器」对入门用户不友好

Anthropic 选「四部件」是有代价的——一个新用户打开 Claude.ai,可能要花几个小时才搞懂 Projects / Artifacts / Skills / Connectors 各是什么、什么时候用哪个。OpenAI 用户开一个 ChatGPT 就能聊,Anthropic 用户要先做产品级理解。这是 Anthropic 在「易用性」上付出的真实成本,也是为什么 Anthropic 的 Skilljar Academy(免费教学平台)、文档、Blog 写得这么细——他们必须靠教育投入弥补产品的认知门槛。显式产品观的代价就是必须配上重投入的用户教育,这一对绑死的设计选择,定义了 Anthropic 的产品风格。

6.8这一章给你的判断武器

读完这章,你应该带走三件能直接用的东西。

✓ 武器一 · 用四部件给自己的 Claude 使用场景做体检

问自己四个问题:有没有跨对话需要持续的上下文(用 Projects)?有没有需要反复迭代的产物(用 Artifacts)?有没有团队反复要重复的工作流(写 Skills)?有没有外部 SaaS 数据要接(开 Connectors)?四个里只用一个,你的 Claude 用法还在 ChatGPT 时代;四个都用上,才是 Anthropic 设计的产品形态。

✓ 武器二 · Projects 的三层都要配齐

不要把 Projects 当聊天记录夹用。三层必配:System Prompt(项目级指令)、Project Knowledge(背景资料)、Conversations(对话沉淀)。少了任何一层,Projects 的杠杆都被打折。一个最常见的反模式是「建了 Projects 但没写 System Prompt」——等于花了车的钱,买了一辆没有方向盘的车。

✓ 武器三 · 看到「AI Memory」类产品,先问它是哪种哲学

下次遇到任何号称「让 AI 记住你」的产品,先问它是显式还是隐式、是容器还是全局、用户能不能看见和编辑记忆内容。隐式 + 全局 + 不可编辑的「Memory」听起来方便,实际是黑箱;显式 + 容器 + 可编辑的方案麻烦,但你永远知道 AI 在记什么。两者各有适合的人,但产品选哪边,本质是「替用户决定 vs 让用户决定」的哲学分叉。

下一章我们换一个尺度——从「Claude 桌面端有哪些部件」转向「Claude Code 里真正在跑的方法论」:EPCC(Explore · Plan · Code · Commit)。如果说四部件是 Claude 的「记忆架构」,EPCC 就是 Claude Code 的「工作节拍」——它把 Agent 的自主循环切成四个明确节点,让监工模式真正可执行。理解了 EPCC,你才知道为什么 Claude Code 不只是「能写代码的 ChatGPT」,而是 Anthropic 在 Agent 工程化上最重的一次产品下注。
