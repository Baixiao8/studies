MCP 三大原语 · Tools / Resources / Prompts Model Context Protocol

第八章把 MCP 当作四把刀里「通往外部世界的协议桥梁」一笔带过,这一章把这座桥梁拆开看清楚。MCP 不是一种新工具调用方式,而是一份开放规范——它把「LLM 怎么接外部系统」这件事,从「每家自己造一套」改成「全行业共用一套」。要用好 MCP,关键不在于「能不能跑通」,而在于把每一个能力归到 Tools / Resources / Prompts 三大原语里的哪一类。归错了,Claude 用不上你的服务。

▼ 三十秒看懂这一章

MCP 的核心抽象是三大原语:Tools 由模型决定何时调用(查数据库、跑命令、发邮件)、Resources 由应用主动提供(文件、配置、上下文数据)、Prompts 由用户从模板里选(/translate /summarize 这类预设)。设计一个 MCP server 时,真正难的不是写代码,是想清楚每一个能力归哪一类——错位会让 Claude 永远调不到、或者反过来污染工具选择。除了三大原语,还有四个进阶题(Sampling 反向 LLM 调用、Roots 文件边界、Notifications 实时反馈、StreamableHTTP 取代旧 SSE)、一个生态(几百个官方 + 社区 server)和三条反共识(MCP 不是 Function Calling、不是越多越好、不一定取代 RAG)。

故事 / 2024 年 11 月 25 日 · Anthropic 发布 MCP

2024 年 11 月 25 日,Anthropic 在工程博客发了一篇看起来不太起眼的文章——Introducing the Model Context Protocol。规范开源、SDK 同步放出(Python / TypeScript)、五个官方参考 server(Filesystem / Git / Slack / GitHub / Postgres)同时上线 [Anthropic 2024c]。文章的措辞很克制,说这是「一个开放标准,让 AI 助手连接外部数据源和工具」——但行业里读懂的人都明白这意味着什么。

故事的另一面发生在一个真实团队里。某个做内部知识管理 SaaS 的小公司,2024 年中花了两个季度做一件事:写「Claude 适配层」「ChatGPT 适配层」「Cursor 适配层」「Cline 适配层」——每家 Agent 厂商的工具协议都不同,他们的同一组 API(搜文档、读文档、编辑文档)要在四个适配层里各写一份,各自维护、各自 bug、各自版本不一致。两个工程师的时间几乎全砸在适配上,真正的功能迭代被卡死。

MCP 发布的第二周,他们把四个适配层全删了。改成一个 MCP server,实现一次,Claude Desktop 能用、Claude Code 能用、Cursor 能用、Cline 也能用——只要那家 Agent 实现了 MCP 客户端。半年后,这家公司的 CTO 在一场技术 Meetup 里说:「MCP 不是省了我们一点工时,是让我们从『集成商』变回了『产品商』。」

这就是 MCP 的真实价值——它不是新发明了「LLM 调外部服务」这件事(Function Calling 早就能干),它发明的是一个让所有 LLM 跟所有外部服务对话的统一插头。类比起来:HTTP 之于网络、USB-C 之于充电、ODBC 之于数据库——MCP 想做的是 LLM 时代的同等位置 [modelcontextprotocol.io 2024]。

12.1MCP 是什么 · A Standard Plug for LLMs

MCP 全称 Model Context Protocol,是 Anthropic 2024 年 11 月开源的一份协议规范。它的设计目标只有一句:让「LLM 接外部系统」从「每家自己造一套」变成「行业共用一套」。

这件事之所以重要,要先看看 MCP 出现之前是什么样。在 2022-2024 这两年,几乎每一家 LLM 应用都在做同一件事——自己定义「工具调用格式」、自己写「外部服务适配」、自己处理「鉴权和状态」。OpenAI 有 Function Calling、Anthropic 有 Tool Use、Google 有 Gemini Functions、Cursor 有自己的 Agent 协议、Cline 也有一套——格式各不相同,字段命名各不相同,错误处理各不相同。结果是:任何想做 LLM 集成的外部系统(GitHub、Notion、Slack、Postgres)都要给每一家 LLM 厂商单独写一份对接。这是行业层面巨大的重复劳动。

MCP 把这件事重构了。它定义了一组协议层抽象,所有 LLM 客户端按这套规范调用,所有外部系统按这套规范提供能力。一边一份代码,组合自由——就像 USB-C 让所有充电器和所有设备解耦那样。

MCP 的三层架构

规范层面,MCP 是 client-server 架构,但术语跟传统 web 不太一样:

HOST
宿主应用
Claude Desktop / Claude Code / Cursor — 用户直接看到的 LLM 应用
CLIENT
客户端
宿主内的 MCP 客户端模块 — 负责跟 server 通信
SERVER
服务端
独立进程 — 暴露 Tools / Resources / Prompts 三类能力
TRANSPORT
传输层
stdio(本地)/ StreamableHTTP(远程) — 协议无关具体形态

这套架构有四个角色,记清楚后面就顺了。Host 宿主,是你直接用的那个应用,Claude Desktop、Claude Code、Cursor 都算;Client 客户端,是宿主内部那个负责跟服务端通信的模块;Server 服务端,是一个独立进程,真正暴露 Tools、Resources、Prompts 这三类能力;Transport 传输层,管它俩怎么连——本地走 stdio,远程走 StreamableHTTP,协议本身不关心具体走哪条。

这套架构的好处:server 一次实现,所有合规的 host 都能用。坏处:多了一层进程间通信,本地启动有几百毫秒级开销。所以判断要不要上 MCP,先看这事值不值得跨进程——本地一个 Bash 一行能做的,绝对不必上 server。

i 关键观察 · MCP 不发明能力,只规范接口

MCP 不让 Claude 多做任何「以前做不到」的事——它做的是把「以前每家各自定义的事」统一成「一份规范说了算的事」。这条很重要:它意味着 MCP 的价值不来自技术突破,来自行业协调。HTTP 不发明计算,但让 Web 成立;ODBC 不发明数据库,但让数据库可移植;MCP 不发明 Agent,但让 Agent 生态可以横向打通 [modelcontextprotocol.io 2024]。理解了这一层,就能理解为什么 Anthropic 把它放出来作为开源标准而不是私有协议——这是策略性让步,目的是把市场做大、把生态绑回 Anthropic。

12.2Tools 工具 · 模型决定何时调用 Model-Controlled

三大原语里,Tools 是开发者最熟悉的——它对应传统意义上的「Function Calling」「Tool Use」。一个 Tool 是 server 暴露的可调用函数,有名字、有参数 schema、有返回值,由 Claude 自己决定何时调用、传什么参数。

典型场景:查数据库、调外部 API、跑命令、发邮件、创建工单。任何「需要主动执行一个动作」的能力,都该归 Tools。

Tools 的几个关键特征

特征一 · Model-controlled · 模型自主决策。Tools 的核心权力交给模型——它判断当前任务需要哪个工具、传什么参数、什么时候调。这意味着 Tool 的 description 写得好不好,直接决定模型用不用它。Anthropic 官方建议 Tool description 里讲清楚:这个工具做什么、什么时候该用、什么时候不该用、参数怎么填、返回什么 [Anthropic 2024c]。description 写糊了,模型大概率绕过它。

特征二 · 副作用要明确。Tool 可能产生真实世界的副作用(发邮件、改数据库、调外部 API)。规范里没有「dry-run」内建机制,但工程上强烈建议把读和写分开——list_emails 和 send_email 拆成两个 Tool,而不是一个 email 工具靠参数区分。这样模型在权衡风险时更准,Hook 拦截写操作也更容易。

特征三 · 输入有 schema、输出无 schema。Tool 的输入参数必须用 JSON Schema 描述清楚——这是模型决定怎么传参的唯一依据。输出则是自由文本(或结构化字符串),Claude 把它当上下文继续推理。这条不对称很关键:输入 schema 越严越好,输出反而该给 Claude 留解释空间。

一个真实例子

拿一个查飞书日历的 MCP server 来说,它的 list_events 工具,description 会写明这是用来列出某个日期范围内的日程、专门回答「今天有什么会」「下周日程」这类问题,同时点明它不负责创建日程也不负责查空闲;参数则要求传入起止日期,日历 ID 可选、默认主日历。

{
  "name": "list_events",
  "description": "列出指定日期范围内的日程事件。
    用于:用户问『今天有什么会』『下周日程』『还有哪些约』。
    不用于:创建日程(用 create_event)、查找空闲(用 check_freebusy)。",
  "inputSchema": {
    "type": "object",
    "properties": {
      "start_date": { "type": "string", "format": "date" },
      "end_date":   { "type": "string", "format": "date" },
      "calendar_id": { "type": "string", "default": "primary" }
    },
    "required": ["start_date", "end_date"]
  }
}

Claude 看到这份 description + schema,就有了判断依据:用户说「今天的会」,它会推断出 start_date 和 end_date 都是今天,调 list_events 拿结果,把摘要给用户。整个流程对用户透明——他根本不知道背后调了一个 MCP server。

✕ 常见误解 · 「Tool 越多越好,把所有能力都做成 Tool」

这是新手最容易犯的错。一个 server 暴露 50 个细粒度 Tool,模型在权衡时会变迷糊——同时有 list_events / get_event / find_events_by_keyword / search_events 四个看起来都能查日程的工具,模型选哪个全凭运气。Anthropic 的实测经验是:单 server 的 Tool 数量控制在 5-15 个之间,每个 Tool 的职责严格单一,description 把适用场景和反例都写清楚,准确率最高 [Anthropic Academy 2025d]。超过 20 个就该想想是不是有些能力该挪到 Resources(数据)或者拆成多个 server。

12.3Resources 资源 · 应用主动提供 Application-Controlled

Resources 是三大原语里最容易被忽略也最容易被误解的一类。它对应的是「server 主动告诉 client 有哪些数据可读」——典型场景:文件内容、配置、文档、上下文数据、当前选中的代码片段、当前打开的工单。

Resources 和 Tools 最核心的差别,在于「谁决定要不要用」。Tool 是模型决定调不调;Resource 是应用主动塞给模型。这是 application-controlled 的含义——宿主应用(Claude Desktop / Cursor)决定哪些 Resource 自动塞进上下文、哪些等用户挑选。

Resource 的典型形态

每个 Resource 有一个 URI(类似 file:///path/to/file.md 或 calendar://primary/today)、一个 MIME type、一段文本或二进制内容。client 可以:

  • ✅ list resources — 拿到当前 server 提供的所有 Resource 清单
  • ✅ read resource — 按 URI 取某个 Resource 的内容
  • ✅ subscribe — 订阅某个 Resource 的变更,变了主动收到通知

宿主应用拿到这些能力之后,可以选择不同策略:

  • 自动注入:用户打开 Claude Desktop,自动把「当前桌面的笔记内容」作为 Resource 塞进上下文
  • 用户挑选:Claude Desktop 在 UI 上展示可用 Resources,用户点击「附加」把某份文件加进对话
  • 引用占位:Resource 不展开内容,只在上下文里放一个 URI,模型需要时显式读

Resources 和 Tools 的边界判断

新手最常问的问题是:既然 Tool 也能读数据,为什么还要 Resources?这两个的区分,本质是「主动权归谁」。

判别角度该用 Tool该用 Resource
触发主导 模型判断「我现在需要查这个」 应用判断「这份数据用户当前肯定要参考」
调用频率 条件性 / 偶尔 / 按需 每次任务都用 / 全程上下文
数据形态 动作的产物(查询结果) 静态或半静态的引用资料
典型例子 search_docs("xxx") 返回搜索结果 文件内容、当前选中代码、配置
UI 暴露 用户看不到调用过程 用户能在 UI 里看见可附加资源

真要区分这两个,有五个角度可以一条条问下去。一问谁来触发:是模型自己判断「我现在得查这个」,还是应用判断「这份东西用户当前肯定要参考」?前者是 Tool,后者是 Resource。二问多久用一次:偶尔、按需、看条件的,归 Tool;每次任务都要、全程都得挂着的,归 Resource。三问数据是什么形态:是一个动作跑出来的产物比如查询结果,那是 Tool;是一份静态或半静态的参考资料,那是 Resource。四看典型例子:按关键词搜文档、拿回结果,是 Tool;一份文件内容、当前选中的代码、一段配置,是 Resource。五看用户能不能感知:Tool 的调用过程用户通常看不见,Resource 则会在界面上明摆着让用户知道有哪些可以附加进来。

一个具体对照:VS Code 风格的「读当前打开的文件」该是 Resource(应用知道用户正在编辑哪个文件,主动提供),「搜索整个代码库找某个函数定义」该是 Tool(模型按需触发,搜索关键词由模型决定)。两个都关于「读代码」,但归属不同。

i 关键观察 · Resources 是「Agent 时代的剪贴板」

把 Resources 理解成「应用层的剪贴板」很贴切——用户在 IDE 里选了某段代码、在 Notion 里圈了某条记录、在浏览器里高亮了某段文字,这些「当前关注的东西」如果通过 MCP Resource 暴露,Claude 就能精准地把上下文聚焦到这里。没有 Resource 的话,你只能靠用户手动复制粘贴或者靠 Tool 全文搜索去猜。Resources 把「现在我关心什么」这条隐式语境变成了协议层的一等公民——这是 MCP 区别于纯 Function Calling 协议的最大设计 [modelcontextprotocol.io 2024]。

12.4Prompts 提示 · 用户从模板里选 User-Controlled

第三类原语 Prompts,在三个里关注度最低,但用对了非常省事。它的本质是server 提供的「预设任务模板」,用户在客户端 UI 里挑一个,客户端把模板填好后发给模型。

典型场景:Claude Desktop 输入框里的 /translate、/summarize、/code-review 这些 slash command。它们对用户而言是「快速指令」,对 server 而言是「我推荐的高频任务,提前帮你 prompt 好」。

Prompt 跟前两个原语的差别

Prompt 是 user-controlled——「用户从清单里挑」,既不是模型自己调(像 Tool),也不是应用自动塞(像 Resource)。它对应的人类直觉是「快捷指令」「模板」「Recipe」。这种归属决定了几个特征:

  • ✅ 可参数化:用户挑了 /translate 之后,client 弹出一个表单让用户填「源语言」「目标语言」,然后把完整的 prompt 发给模型
  • ✅ 对用户可见:Prompt 列表在 UI 上展示,用户能浏览所有可用模板,知道这个 server 给我能用什么
  • ✅ 不调用工具:Prompt 本身只是文本模板,它生成的 prompt 发给模型后,模型再自主决定要不要调 Tool / 用 Resource

一个真实例子

假设有个「代码 review」MCP server,它暴露一个 review-pr Prompt,接收三个参数:仓库、PR 编号是必填的,还有一个可选的「重点关注」,可以填 security、perf 或 style。

{
  "name": "review-pr",
  "description": "对一个 GitHub PR 做完整 code review",
  "arguments": [
    { "name": "repo",     "description": "GitHub 仓库", "required": true },
    { "name": "pr_number","description": "PR 编号",    "required": true },
    { "name": "focus",    "description": "重点关注(可选:security / perf / style)" }
  ]
}

用户在 Claude Desktop 输入 /review-pr,UI 弹出表单,用户填好仓库、PR 编号和关注点。client 拿到参数,从 server 取出真正的 prompt 模板——模板里预先写好了一套 review 流程,等着填参数。

"请对 {repo} 仓库的 #{pr_number} 做 code review,重点关注 {focus}。
步骤:
1. 用 github_mcp::get_pr 拿到 PR diff
2. 逐文件分析,检查 {focus} 相关问题
3. 按严重程度排序,输出报告"

client 把模板填好,发给模型。模型按这份 prompt 跑流程——它会自动调用 github_mcp 的 Tool 拿 PR、自动按要求格式输出。Prompt 的价值,是把「用户高频任务的 prompt」沉淀成 server 提供方的最佳实践,而不是让每个用户自己摸索怎么写 prompt。

✕ 常见误解 · 「Prompts 是给开发者用的」

恰恰相反——Prompts 是给终端用户用的快捷入口。开发者写 prompt 走的是代码或者直接 prompt 输入框,不需要 Prompts 原语。Prompts 的真正受众是非技术用户:他们不会写 prompt,但能从一个清单里挑「我要做这件事」,然后填一些显而易见的参数。设计 server 时,如果你的 server 主要服务于自动化场景(Agent 自己调),那 Tools 就够了,Prompts 是浪费;如果你的 server 服务于人在 Claude Desktop 这种 UI 里使用,那 Prompts 是给用户提供入口的关键——没有它,用户不知道这个 server 能干嘛。

12.5三大原语对比 · 「谁控制」是核心区分 The Defining Question

把前三节收束成一句话。设计 MCP server 时,真正难的不是写代码,是想清楚每一个能力归哪一类——归错了,Claude 永远调不到、或者反过来污染工具选择。

维度 Tools 工具 Resources 资源 Prompts 提示
谁控制 Model-controlled 模型决定 Application-controlled 应用决定 User-controlled 用户决定
何时触发 模型推理中按需调用 应用主动塞或用户附加 用户从 UI 清单里挑
典型场景 查数据库、跑命令、发邮件、调外部 API 文件内容、配置、当前选中代码、文档 /translate、/summarize、/review-pr
有无副作用 可有(执行动作) 通常只读 无(只生成 prompt)
设计陷阱 暴露太多 Tool 让模型选迷糊 该当 Resource 的当成 Tool 让模型靠搜索找 开发者自动场景里硬塞 Prompts 占位
类比 员工自己决定调用的工具 领导主动发到员工桌面的资料 菜单上预设好的套餐

说到底,「谁控制」就是设计 MCP server 时必须先回答的那个问题。看到一个能力,问自己:这件事什么时候发生?

  • 「模型在推理中按需用」 → Tool
  • 「应用层判断现在该塞给模型」 → Resource
  • 「用户主动选要做这件事」 → Prompt

同一个外部系统,可以同时提供三类:Linear MCP 可以暴露 create_issue Tool(模型按需建工单)、linear://current-sprint Resource(应用主动塞当前 sprint 概览)、/triage-bugs Prompt(用户挑这个模板批量分类未处理 bug)。三者不替代,各管一段。

✓ 武器 · 设计 server 时按「谁控制」一问到底

拿到一个外部系统(假设是 Notion),不要直接想「这玩意儿要暴露哪些功能」,先列所有能力,再对每个能力问:模型自己决定何时用 → Tool;应用层根据当前场景塞 → Resource;用户从菜单挑 → Prompt。一遍走完,server 的形态就清楚了。这条提问框架,是把 MCP server 从「可以跑」做到「Claude 用得上」的关键差别。

12.6进阶四题 · Beyond the Three Primitives

三大原语解释完之后,MCP 还有四个进阶特性值得讲清楚——它们解决的是「server 跟 client 协作时的几个棘手问题」,在 server 做大之后才会真正用到。

题一 · Sampling — server 反向请求 client 调 LLM

常规 MCP 流程是 client → server(client 调 Tool、读 Resource)。但有一类需求反过来:server 在执行 Tool 时,自己也想用 LLM 推理一下——比如一个「自动生成单元测试」的 Tool,执行过程里需要让 LLM 分析代码、产生测试用例。

朴素方案是 server 自己持有一份 API key 调 Claude API。但这有问题:server 自己付费、自己管 key、用户多了成本失控。MCP 设计了一个反向请求机制——Sampling:server 把「我需要 LLM 推理一下」这个请求发回给 client,client 用宿主应用的现有模型连接执行,把结果回给 server [modelcontextprotocol.io 2024]。

好处:

  • ✅ 不用 server 持有 API key — 所有 LLM 调用都走宿主应用的额度
  • ✅ 统一的安全审计 — 用户能看到所有 LLM 调用
  • ✅ 模型可替换 — 用户在 Claude Desktop 切换模型,server 自动跟着切

题二 · Roots — 文件访问权限边界

Roots 解决的是「server 该有哪些文件访问权」。当一个 Filesystem MCP server 跑起来,它技术上能读整个磁盘——这显然不安全。MCP 规范定义了 Roots 抽象:client 在握手时告诉 server「你只能在这几个目录里活动」,server 按这个边界跑。

典型场景:Claude Code 在某个项目里启动,把这个项目目录作为唯一 Root 告诉 Filesystem server,server 后续所有 read/list/write 都被约束在这个目录里。这相当于操作系统级 chroot 的 MCP 版,但是协作式的——server 自觉遵守,client 也可以二次校验。

题三 · Notifications — 实时反馈与进度

有些 Tool 跑得久(几十秒到几分钟):跑一个大型测试套件、做一次完整代码扫描、上传一个大文件。如果 Tool 调用是「阻塞式等结果」,用户体验很差——前端没法显示进度,Claude 也不知道现在到哪一步。

MCP 的 Notifications 机制让 server 在 Tool 执行过程中主动推送进度消息到 client。client 收到消息后,可以在 UI 上更新进度条、在对话里显示「正在跑第 3/10 个测试」、让模型决定是不是要等下去。这对长跑任务的协作体验是质的提升 [modelcontextprotocol.io 2024]。

题四 · StreamableHTTP Transport — 取代旧 SSE

MCP 早期支持两种传输:stdio(本地进程)、SSE(Server-Sent Events,远程)。SSE 是单向流,适合服务器推送,但有几个工程问题:不容易做无状态水平扩展、防火墙穿透差、连接管理复杂、断线重连体验差。

2025 年规范升级引入 StreamableHTTP 替代 SSE,作为生产级远程传输的推荐方案 [modelcontextprotocol.io 2025]。它的核心改进:

  • ✅ 无状态水平扩展 — 同一个 server 可以由多个实例服务,client 任意连
  • ✅ 普通 HTTP 友好 — 走标准 HTTP,所有反向代理 / 负载均衡 / CDN 都能透传
  • ✅ 双向流 — 既能 server push、也能 client 主动 invoke
  • ✅ 明确的会话语义 — 通过 session ID 关联多次请求,而不是依赖长连接

对开发者的实际影响:2025 年之后写新的远程 MCP server,直接用 StreamableHTTP,SSE 看作 legacy。已有的 SSE server 会有过渡期,但新生态主推 StreamableHTTP。

i 关键观察 · 这四个特性都不是「核心原语」,但都是「生产化必需」

三大原语解决的是「能不能跑」,这四个进阶题解决的是「跑得好不好、跑得安不安全、跑得稳不稳」。Sampling 解决成本与安全、Roots 解决权限边界、Notifications 解决长跑体验、StreamableHTTP 解决远程部署。把这四题想清楚,server 才能从「demo」做到「生产」。新人做 MCP server 时容易只关注三大原语,把这四题留到出问题再说——能跑的 demo 跟能撑住的生产系统之间,就是这四题的工程量 [Anthropic Academy 2025e]。

12.7生态全景 · Hundreds of Servers, One Protocol

MCP 发布一年多以来,生态扩张速度超出 Anthropic 预期。到 2026 年初,官方 + 社区维护的 server 数量已经过千。这一节不可能一一列举,只挑几条主轴讲清楚生态的形态。

官方参考实现

Anthropic 自己维护一组「reference server」作为生态种子和样板。这些 server 不只是给用户用,更是给后来者抄作业用——展示一个标准 MCP server 长什么样、frontmatter 怎么写、错误处理怎么做、Roots 怎么用 [Anthropic 2024c]:

  • Filesystem — 本地文件读写,带 Roots 边界控制
  • Git — 仓库历史、diff、blame
  • GitHub — issue / PR / repo 操作
  • Slack — 消息收发、频道管理
  • Postgres — 数据库查询,只读
  • Google Drive / Maps / Memory / Puppeteer / Sentry 等

第三方主流 server

第三方厂商和社区贡献的 server 覆盖几乎所有常见 SaaS 和开发工具:

  • 协作类:Notion、Linear、Jira、Atlassian Confluence、ClickUp、Asana、Monday
  • 开发类:Cursor、Sentry、PagerDuty、Datadog、Vercel、Cloudflare
  • 设计类:Figma、Framer、Sketch
  • 数据类:Snowflake、BigQuery、ClickHouse、ElasticSearch、MongoDB
  • 沟通类:飞书(Lark)、企业微信、Discord、Telegram、Zoom
  • AI 工具类:Replicate、HuggingFace、OpenAI、Perplexity

这份名单还在持续增长。MCP 官方有一个 server registry 索引(modelcontextprotocol.io/registry),作为生态的中心目录,但社区里也有 awesome-mcp-servers 这类清单,补充更多边缘场景。

跨 Agent 厂商的接纳

MCP 真正成为「行业标准」的标志,是其他 Agent 厂商也实现了 MCP 客户端。到 2026 年初,主流 Agent IDE / Desktop 大多兼容 MCP:

  • ✅ Anthropic Claude Desktop / Claude Code — 原生支持
  • ✅ Cursor — 2025 年中接入 MCP 客户端
  • ✅ Cline / Continue.dev — 早期就跟进
  • ✅ Zed / JetBrains AI Assistant — 2025 年下半年加入

这意味着你给 Claude 写的 MCP server,大概率也能给 Cursor / Cline 用,无需重写。这就是开放协议的网络效应——server 作者只需要实现一次,服务范围却随客户端生态指数级扩展。这种正反馈是 MCP 一年内能起势的根本原因 [modelcontextprotocol.io 2025]。

12.8反共识三条 · Three Counterintuitive Truths

行业里关于 MCP 有几个流传很广的误区。这一节专门拆解三条最常见的反共识,让读者跳过这些坑。

反共识一 · 「MCP 跟 Function Calling 是一回事」

这是最常见的误解,也最容易导致后续判断全错。Function Calling 和 MCP 在形式上确实像——都涉及模型调用外部能力——但它们解决的问题不在一个层级:

维度Function CallingMCP
范围单一 LLM API 内的功能扩展跨进程、跨厂商的开放协议
定义方每家 LLM 厂商自己定义格式统一规范,所有家共用
实现位置函数代码跟调用方耦合在一起独立 server 进程,跨应用复用
能力类型只有 Tool 一类三大原语 + 进阶四题
跨厂商OpenAI 格式 ≠ Anthropic 格式同一份 server 服务多家 host

简单类比:Function Calling 像是「在一个 app 内部加一些插件 API」,MCP 像是「定义了一份标准协议,所有 app 跟所有插件都按这套对话」。前者是应用内功能,后者是生态层基础设施。

具体到工程影响:用 OpenAI Function Calling 写的工具代码,换到 Claude 上要重写;用 MCP 写的 server,Claude / Cursor / Cline 都能直接用。这就是协议层抽象的价值。

反共识二 · 「MCP server 装得越多越好」

很多用户拿到 Claude Desktop 就开始疯狂装 MCP server——GitHub 装一个、Slack 装一个、Notion 装一个、Linear 装一个、Filesystem 装一个、Postgres 装一个... 装到 20+ 之后,发现一个奇怪现象:Claude 在简单任务上反而变笨了。问它一个明明该用现有信息就能答的问题,它非要绕一圈调某个 MCP Tool;问它写一段代码,它先调五次 MCP 看看有什么资源可用。

原因:过多 MCP server 同时挂着,Tool 总数从 5 个膨胀到 80 个,模型每一次推理都要在 80 个 Tool 里挑选——这会显著污染工具选择,降低准确率,也大幅消耗注意力 [Anthropic Academy 2025d]。

实操经验:

  • ✅ 按场景分组挂载:写代码场景挂 GitHub + Filesystem,开会场景挂日历 + Notion,不要全开
  • ✅ 用「项目级配置」隔离:Claude Code 的 settings.json 按项目配 MCP,每个项目只挂当下需要的
  • ✅ 临时开启 / 关闭:Claude Desktop 支持运行时开关,需要时再开
  • ❌ 不要一次性挂 20+ server 在全局:这会让所有任务都变慢、变迷糊

反共识三 · 「MCP 取代 RAG」

有人看到 Resources 能提供文件内容,就觉得 MCP 把第十一章讲的 RAG 全干掉了——不用做索引、不用 embedding、不用 vector DB,直接让 server 提供 Resources 就完了。这个判断对小规模数据成立,对大规模数据不成立。

原因:Resources 的工作方式是「整份内容塞进上下文」。上下文窗口有上限(200K tokens / ~15 万中文字),一份 100 页的文档塞进来还有余量,但一个有 10 万文档的知识库——绝对塞不下。这时候 RAG 还是必要的:先 embedding 检索找到 top-K 相关片段,再把片段作为 Resource 塞给模型。

两者的真实关系是互补,不是替代。

场景数据规模推荐方案
当前打开的几份文件≤ 上下文 1/3MCP Resources 直接塞
一个项目的所有代码能塞,但占满窗口Resources(精选)+ Tool(grep)按需补
公司全部内部文档百万级 chunksRAG(向量检索)+ MCP Tool 包装检索
千万级 / 实时数据无论怎么裁都塞不下RAG + 数据库 + MCP Tool 暴露查询

怎么互补,按数据量分四档就清楚了。当前打开的那几份文件,小到占不满上下文三分之一,直接用 Resources 塞进去就行;一个项目的全部代码,塞得下但会占满窗口,这时候精选一部分走 Resources、剩下的靠一个 grep 类的 Tool 按需补;到了公司全部内部文档、百万级片段这个量级,就得上 RAG 向量检索,再用一个 MCP Tool 把检索能力包起来;而千万级甚至实时变动的数据,怎么裁都塞不进上下文,只能 RAG 加数据库,再用 MCP Tool 暴露查询接口。一句话,数据越大,越往检索那头靠。

更精准的说法:MCP 提供了「数据接入的协议层」,RAG 提供了「数据检索的算法层」。两层叠加,各管一段——MCP 解决「LLM 怎么调到数据」,RAG 解决「数据怎么按相关度筛选」。看到 MCP 出现就预言 RAG 死亡的判断,是混淆了协议层和算法层(第十一章详谈)。

✕ 反共识汇总 · 三条最常见的 MCP 误判
  • ❌ 「MCP = Function Calling 的新名字」 → MCP 是跨厂商协议,Function Calling 是单 API 功能,不在一个层级
  • ❌ 「MCP server 越多越好」 → 过多 Tool 污染模型选择,按场景分组挂载、临时开关
  • ❌ 「MCP 取代 RAG」 → 两层互补,中小数据用 Resources,大数据还得 RAG + MCP Tool 包装

12.9这一章给你的判断武器

读完这一章合上,你应该带走以下几个能直接用的判断武器。

✓ 武器一 · 设计 MCP server,先回答「谁控制」

对每个能力问:模型自己决定何时用 → Tool;应用主动判断该塞 → Resource;用户从清单挑 → Prompt。这条提问框架,是把 MCP server 从「能跑」做到「Claude 真用得上」的核心差别。归错了原语类型,Claude 永远调不到、或者一直误调。

✓ 武器二 · Tool 数量控制在 5-15,description 把适用场景写透

单 server 的 Tool 数量过多,模型选不准。Anthropic 实测的甜区是 5-15 个 Tool / server,每个 Tool 的 description 把「什么时候用、什么时候不用、参数怎么填」都写清楚。同一个 Claude 桌面上挂的 MCP server 总数也别超过 5-7 个常驻,按场景启停。

✓ 武器三 · MCP 跟 RAG 不是替代关系

中小规模数据(≤ 上下文 1/3)可以靠 MCP Resources 直接塞;大规模数据必须 RAG 检索 + MCP Tool 包装。两层互补:MCP 是协议层、RAG 是算法层,各管一段。看到 MCP 出现就放弃 RAG 是常见误判。

这一章把 MCP 的协议本质、三大原语、进阶四题、生态全景、反共识三条全讲完了。回看脉络:第 六章 提过 Claude Desktop 的 Connectors,本质就是 MCP client 的产品化呈现;第 八章 把 MCP 当作四把刀里「通往外部世界的协议桥梁」一笔带过,这一章是它的展开;第 十一章 RAG 跟本章的关系刚刚说清楚——协议层和算法层的互补。下一章 第十三章 我们换到 Agent 工作流模式——MCP 提供了「触达外部」的能力,但 Agent 怎么把这些能力组装成稳定可维护的工作流(Prompt Chaining / Routing / Parallelization / Orchestrator-Workers / Evaluator-Optimizer),那是另一个完整的话题。
