Skills / Subagents / Hooks / MCP 四把刀 When to Use Which

Claude Code 给开发者留了四个自定义机制——Skills、Subagents、Hooks、MCP。新人最常踩的坑是把它们当成同一类东西、随手抓一个用。这一章把四把刀彻底分开：每一把的本质、触发方式、配置方式、典型场景，以及遇到一个需求该怎么 30 秒内选对刀。

▼ 三十秒看懂这一章

四把刀的边界一句话：Skill 是「教 Claude 怎么做某件事」的工作流载体——按需自动加载；Subagent 是「派分身去做一件事、主线不受打扰」的并行机制——独立上下文；Hook 是「在特定事件前后强制跑某段脚本」的策略机制——不依赖模型自觉；MCP 是「让 Claude 接入外部系统」的协议桥梁——通向数据库 / API / 内部服务。把这四个分清，你才能不滥用 Subagent 拖慢主任务、不把 Skill 当 CLAUDE.md 用、不用一段裸 prompt 去做本该用 Hook 强制的事。

故事 / 2025 年 10 月 16 日 · Anthropic 推出 Agent Skills

2025 年 10 月，Anthropic 在官方博客发布了一项看似不起眼的功能：Agent Skills。文档很短——SKILL.md 文件，放在 ~/.claude/skills/ 或项目 .claude/skills/ 目录下，头部 frontmatter 写明这个技能在什么场景下触发，正文写做这件事的步骤。Claude 在跑任务时，会扫描所有可用 Skills，匹配场景就把对应的 SKILL.md 加载到上下文里 [Anthropic 2025a]。

这条消息发布后的两周里，开发者社区出现了一种典型的迁移行为。之前他们把所有跨项目的偏好、模板、流程都堆在 CLAUDE.md 里——发邮件的模板、生成 PR 描述的格式、把 Figma 导出的 SVG 转 React 组件的步骤、内部 API 的鉴权方式——CLAUDE.md 越涨越长，一些团队的根目录 CLAUDE.md 已经超过 3000 字，每次对话都全量塞进上下文。

Skills 出现之后，这些散乱的「教程式」内容开始被拆成一个个独立 Skill 包。send-email/SKILL.md 只在「用户提到发邮件、起草通知」时加载；figma-to-react/SKILL.md 只在「涉及 SVG 转组件」时加载。CLAUDE.md 回归本来的角色——项目级硬性规则和团队约定，而不是所有知识的垃圾桶 [Anthropic Academy 2025a]。

但与此同时，另一批人栽在了 Subagent 上。Claude Code 1.0 之后，/agents 命令让创建子代理变得很简单，有人就把所有任务都开 subagent：写一个函数派一个 subagent、跑一个测试派一个 subagent、改一行配置派一个 subagent。结果主任务变得比直接做还慢——subagent 启动、上下文准备、总结回传，每个动作都有 overhead，简单事被它放大成复杂事 [Anthropic Academy 2025b]。

这一年里，这四把刀的边界在社区被反复讨论、反复试错，最终沉淀出一些共识。这一章把这些共识讲清楚，你就不用从头踩一遍。

8.1四把刀全景 · The Four Knives

Claude Code 的四种自定义机制，从抽象到具体可以这样排：MCP 是地基（让 Claude 触达外部世界）、Hook 是底盘（让特定动作前后跑强制脚本）、Subagent 是工位（让任务在独立上下文里跑）、Skill 是工作手册（教 Claude 怎么做某类活）。它们不是替代关系，而是分层关系——一个高级 Claude Code 项目通常四个全用，各管一段。

四把刀的本质、触发、配置、典型场景，下面五节会一把一把拆开讲透，先把彼此的边界放在心里。

维度 Skill Subagent Hook MCP
本质 工作手册 独立工位 策略闸口 外部桥梁
解决什么 「怎么做这类活」 「主线别被卡住」 「必须每次都做」 「碰不到的外部系统」
触发方式 场景匹配自动加载 主代理显式派活 事件触发（PreTool / PostTool / UserPrompt 等） 工具调用通过 MCP server
配置位置 SKILL.md 文件 + frontmatter /agents 命令 + agent 配置文件 settings.json 里的 hooks 段 MCP server 进程 + settings.json 注册
上下文 挂到主上下文里 独立上下文（隔离） 脚本独立运行，不进上下文 调用结果进上下文
是否能拒绝 模型决定要不要看 主代理决定派不派 强制 · 模型无法绕过 模型决定调不调
典型场景 发邮件 / 写 PR / 转 PDF 跑长任务 / 并行调研 commit 前 lint / edit 后 format 查 Linear / 读 PostgreSQL
典型反例 把通用上下文当 Skill 写 简单任务也开 subagent 把业务逻辑塞进 Hook 能本地做的事也走 MCP

表里有几个关键差别值得单独点出来：

  • ✅ 只有 Hook 是强制的。Skill / Subagent / MCP 都是「模型自己决定要不要用」——它可以选择不加载某个 Skill、不派 subagent、不调 MCP 工具。Hook 不一样，只要事件触发，脚本必跑，模型连知道都不一定知道。
  • ✅ 只有 Subagent 有独立上下文。这是它的核心价值，也是它的成本来源——独立上下文意味着主代理拿不到 subagent 的中间过程，只能拿到它最后的总结，有些信息会丢。
  • ✅ 只有 MCP 是协议，其他三个都是 Claude Code 内部机制。MCP 是 Anthropic 在 2024 年开源的标准 [Anthropic 2024c]，其他厂商的 Agent 也能用，这意味着你为 Claude Code 写的 MCP server 也能给 Cursor、Cline 用。
i 关键观察 · 「按需加载」是 Skill 的核心设计哲学

Skill 之所以能跟 CLAUDE.md 区分开，核心就是按需加载。CLAUDE.md 是每次对话都全量塞进上下文的——它消耗的 token 是固定成本。Skill 是匹配到场景才加载——它的 token 消耗是按需成本。一个项目可以有 50 个 Skill，但任何一次对话通常只触发 1-2 个。把不需要每次都讲的事写进 Skill，把每次都需要的硬规则写进 CLAUDE.md——这条边界是 Skills 设计的隐含意图，也是社区一年来反复确认的最佳实践 [Anthropic Academy 2025a]。

8.2Skill · 教 Claude「怎么做某类活」 Skills as Workflow Containers

Skill 是一个文件夹，里面放一个 SKILL.md，以及任何辅助文件（脚本、模板、示例）。SKILL.md 头部是 YAML frontmatter，描述这个 Skill 在什么场景下应该被触发；正文是这个任务怎么做的工作流——前置检查、步骤、产出格式、注意事项。

一个具体例子：发飞书邮件的 Skill

以一个真实的飞书发邮件 Skill 为例，感受它的形态：一个文件夹，根上放 SKILL.md，旁边可以带模板和脚本；SKILL.md 头部的 frontmatter 声明触发场景，正文写这件事的标准流程——先确认登录态、再查收件人、再按模板发。

~/.claude/skills/lark-mail/
├── SKILL.md
├── templates/
│   ├── notification.md
│   └── weekly-report.md
└── scripts/
    └── send.sh

# SKILL.md frontmatter
---
name: lark-mail
description: 飞书邮箱 — 起草、发送、回复邮件。
  Use when user mentions 起草邮件、写一封邮件、发邮件、
  draft email, send email, reply email
---

# SKILL.md 正文
## 触发场景
当用户提到发邮件 / 起草通知 / 回复邮件 ...

## 标准流程
1. 先用 lark-cli auth check 确认登录态
2. 用 lark-cli contact +search 找收件人 open_id
3. ...

## 模板
notification 类邮件用 templates/notification.md ...

Claude 在跑任务时，会扫所有 Skill 的 frontmatter description，匹配上当前场景就把对应 SKILL.md 加载进上下文。整个过程对用户透明——你说「帮我给团队发个周报」，Claude 内部自动加载 lark-mail/SKILL.md，按里面的步骤跑；你说「帮我转一下这个 PDF」，Claude 加载的是 pdf-tools/SKILL.md。

Skill 的几条设计原则

原则一 · description 决定触发准确率。Skill 能不能被加载，完全靠 frontmatter 的 description 写得好不好。Anthropic 官方文档给的建议是：description 里写明触发词、典型场景、不该触发的反例 [Anthropic 2025a]。description 写得越具体，误触和漏触就越少。

原则二 · 一个 Skill 只做一件事。不要写一个「通用助手」的 Skill，把所有偏好都塞进去——那不叫 Skill，那是把 CLAUDE.md 换了个名字。一个 Skill 应该聚焦一个清晰的任务族（发邮件 / 转 PDF / 创建工单），边界清楚，匹配条件明确。

原则三 · Skill 可以调用 MCP server。这一点最容易被忽略。Skill 是工作流容器，里面的步骤可以调外部工具——而这些工具往往就是 MCP 提供的。例如 lark-mail Skill 的步骤里会调 lark-cli（它可以是 MCP server），deploy-prod Skill 会调一个连到 K8s 的 MCP server。Skill 是「怎么做」，MCP 是「拿什么做」，两层叠在一起才完整。

原则四 · 用户 / 项目 / 团队三层组织。Skill 文件夹可以放在三个地方：

  • ~/.claude/skills/ · 用户级，对所有项目生效（发邮件、转 PDF 这种通用工作流）
  • .claude/skills/ · 项目级，只在当前项目生效（项目特有的发布流程、内部 API 调用方式）
  • 插件包 · 通过 Claude Code Plugin 分发（团队级，统一安装）
✕ 常见误解 · 「Skill 就是放在 .claude 里的 CLAUDE.md」

这是社区里最高频的误解，也是 Skill 用不出价值的根本原因。CLAUDE.md 是每次都加载的，它适合放硬规则（命名约定、必须用的工具、绝对禁止的操作）。Skill 是按需加载的，它适合放条件性知识（某类任务的工作流）。两个的差别不在内容形式，而在加载机制——你把通用上下文塞进 Skill,Claude 在不该触发的场景也加载它，token 浪费、注意力分散；反过来，你把条件性工作流写进 CLAUDE.md，每次对话都塞 3000 字进上下文，所有任务都被它拖累。判别标准：这段内容是不是每次对话都必须知道？是 → CLAUDE.md；不是 → Skill。

Skill 的合适场景 vs 不合适场景

  • ✅ 跨项目通用的工作流：发邮件、写 commit message、把 Figma 导出转 React、生成会议纪要
  • ✅ 有标准化步骤的任务：跑测试、做 code review、发布到 staging
  • ✅ 需要附带模板 / 脚本 / 示例文件:Skill 文件夹本身就是个容器，模板和脚本可以放在同一目录
  • ❌ 项目硬规则（永远必须用 TypeScript、不准动某个文件夹）→ CLAUDE.md
  • ❌ 一次性任务（只在这次需要）→ 直接 prompt
  • ❌ 需要外部访问能力（查数据库 / 调 API）→ Skill 不提供能力，要配 MCP

8.3Subagent · 派分身去跑、主线不卡 Subagents as Parallel Workers

Subagent 是主代理派出去的小代理，它有自己独立的上下文窗口，跑完一个任务后把结论摘要回报给主代理。主代理拿到摘要，继续往下走——subagent 的中间过程不会污染主代理的上下文。

Anthropic 的 Skilljar 课程把 Subagent 的价值压成一句话：「保持主上下文干净 + 实现并行 + 任务隔离」 [Anthropic Academy 2025b]。这三件事是 Subagent 的全部价值，也是判断要不要用它的标尺。

什么时候 Subagent 真的有用

USE CASE 1
独立子模块
3 个以上彼此独立的章节 / 文件块 / 调研子题，并行派 subagent 同时跑
USE CASE 2
脏活
读 50 个文件找一条信息 — subagent 干完只回报结论，主代理不必把 50 个文件全看一遍
USE CASE 3
大输出
长报告分段写 — 主代理装配，subagent 各负责一段产出
USE CASE 4
探索性发散
「先扫一遍代码库回报观察」— 这种任务 subagent 跑完就丢，不必污染主上下文

真正用得上的场景大致四类：一是手头有三个以上彼此独立的子模块——不同章节、不同文件块、不同调研子题，这种适合并行派出去同时跑；二是脏活，比如要翻五十个文件就为找一条信息，派出去的分身干完只回报结论，主代理不必把五十个文件全看一遍；三是大输出，长报告分段写，主代理做装配、分身各负责一段；四是探索性发散，先扫一遍代码库回报观察，这种跑完就丢，不必污染主上下文。

这四个场景的共同特征：任务可切片、产出可摘要、主代理不需要看中间过程。任何一个不满足，就该重新想要不要开 subagent。

什么时候 Subagent 反而拖慢

Subagent 不是免费的。开一个 subagent 的成本包括：启动延迟、上下文准备、独立 LLM 调用、摘要回传——加起来通常比直接做多 30-60%。所以有这么几类场景千万别滥用：

  • ❌ 简单任务也派 subagent：「写一个函数」开 subagent、「跑一次测试」开 subagent——这种事直接做更快
  • ❌ 任务彼此依赖：第二个 subagent 需要看第一个 subagent 的中间过程才能做——这种应该是顺序任务，不是并行 subagent
  • ❌ 需要主代理看到完整产出:subagent 只回报摘要，如果你需要看 subagent 的完整思考链，那它该是主代理
  • ❌ 用户期待持续可见:subagent 跑的时候用户看不到细节，等结果出来才能 review——交互延迟显著
✕ 反共识 · 「越多 subagent 越快」

很多人把 subagent 当成「并行万能药」，任何任务都派分身。实际上 subagent 的并行收益只有在任务真的独立 + 单任务足够大时才大于 overhead。Anthropic 内部的经验是：派 subagent 的门槛是「这个子任务单独跑预计 5 分钟以上 + 跟其他子任务完全独立 + 主代理只需要它的总结」三个条件全占，才派。不满足就直接在主代理里串行做。「派分身」不是流程必修，是协作工具，能用 Bash 一行解决的事别开 agent。

Subagent 的配置方式

Claude Code 用 /agents 命令创建和管理 subagent 配置 [Anthropic Docs 2025]。一个 agent 配置文件包含：

  • name · agent 的标识
  • description · 什么时候用它（类似 Skill 的 frontmatter，主代理靠它决定要不要派）
  • system prompt · 这个 subagent 的角色定位、产出格式约束
  • tools · subagent 能用哪些工具（可以比主代理少，实现权限隔离）
  • model · 可选，subagent 可以用更小的模型（比如 Haiku）做简单子任务，降本

这个最后一条是 Subagent 的隐藏价值——主代理用 Opus,subagent 跑简单调研用 Haiku，成本降一个数量级。这条用法 Anthropic 在 Claude Code Foundations 课程里专门讲过 [Anthropic Academy 2025b]。

8.4Hook · 在特定事件前后强制跑脚本 Hooks as Policy Gates

Hook 是绑在 Claude Code 事件上的脚本——某个事件触发，脚本必跑，模型无法绕过、无法商量。这是它跟其他三把刀最大的差别：Skill / Subagent / MCP 都依赖模型自己决定要不要用，Hook 是强制的。

Hook 支持的事件

Claude Code 在 2025 年下半年逐步开放了一组事件 hook，目前主要的几个 [Anthropic Docs 2025b]:

事件触发时机典型用途
PreToolUseClaude 调任何工具之前权限检查、输入校验、危险操作拦截
PostToolUse工具调用结束后结果日志、副作用清理
UserPromptSubmit用户发送一条消息时注入额外上下文、敏感词检查
StopClaude 准备结束这一轮回复时最终输出审查、强制收尾流程
SubagentStop某个 subagent 结束时验证 subagent 产出格式

这些挂钩点大致落在一次任务的几个节骨眼上：工具调用之前、工具调用之后、用户每发一条消息时，还有两个收尾位置——Claude 准备结束这一轮回复时，以及某个分身跑完时。前几个用来做权限检查、日志、注入上下文；后两个常用来做最终输出审查，或者验证分身交回来的产出格式对不对。

Hook 配置在 settings.json 里。一个典型例子是「commit 前强制跑 lint」——在 PreToolUse 上挂一个 hook，匹配工具是 Bash 且命令含 git commit，跑 npm run lint，失败就阻断 commit 操作。Claude 根本没机会绕过——它发起 commit，事件触发，lint 跑，不过就直接被拦下。

Hook 的真正价值：把「必须做」从模型自觉变成系统强制

这一点值得展开。Skill 和 CLAUDE.md 里也能写「commit 前要跑 lint」，但这是规则——模型可能忘、可能漏、可能在某个特殊场景里跳过。Hook 是机制——只要事件触发，脚本必跑，无关模型怎么想。

这条差别在严肃工程里特别重要：

  • ✅ 合规 / 安全场景：任何 commit 必须扫密钥、任何 edit 必须走 git 备份 → 用 Hook，模型不能选择跳过
  • ✅ 自动化粘合:edit 后自动 prettier、写完文件自动 reload — 用 Hook，不依赖模型记得
  • ✅ 注入上下文：每条用户 prompt 来时，自动把当前 git status 塞进上下文 — 用 UserPromptSubmit hook
  • ❌ 业务逻辑：复杂的判断该不该做某事 — 这种该交给模型，Hook 只做硬规则
  • ❌ 调试性输出：想看模型在干什么 — 用日志，不要乱挂 hook 影响正常流程
✓ 武器 · Hook 是把工程纪律「写死」的唯一办法

当你发现自己反复在 CLAUDE.md 里强调一条规则、但模型还是隔三差五漏掉，就该想：这条规则要不要变成 Hook？写在 prompt 里靠模型自觉的规则，有失守的可能；挂在事件上靠系统执行的 Hook，没有失守的可能。但代价是 Hook 写多了，Claude Code 的行为会变僵——所以 Hook 的边界要清晰：只挂硬性策略，不挂软性偏好。一个项目通常有 2-5 个 Hook 就够了，超过 10 个说明你在用 Hook 当业务逻辑写，方向跑偏。

8.5MCP · 通往外部世界的协议桥梁 MCP as Bridge to the World

前三把刀都是 Claude Code 内部机制——它们扩展的是 Claude 自身的行为方式。MCP(Model Context Protocol)不一样，它是一个外部协议，定义了 Claude 怎么跟外部系统对话——数据库、API、内部服务、IDE、各种 SaaS。

Anthropic 2024 年 11 月开源了 MCP 规范 [Anthropic 2024c]。规范的核心抽象：一个 MCP server 暴露一组 tools（可调用的函数）、resources（可读的资源）、prompts（预定义的提示模板），Claude 通过标准协议跟 server 通信。任何系统只要实现了 MCP 规范，Claude 就能用它。

MCP 长什么样

一个真实的 MCP server 例子——查飞书日历：

  • tools:list_events、create_event、check_freebusy — Claude 调这些函数
  • resources:calendar://primary/today — Claude 可以读今日日程作为上下文
  • prompts:plan_meeting — 预定义的「规划会议」流程模板

当 Claude 在跑「明天有哪些会」这个任务时，它发现自己有 list_events 这个工具可用，就调用 MCP server,server 拿用户的 access token 去查飞书 API，把结果通过 MCP 协议回给 Claude,Claude 整合进回复。整个过程对用户透明，但 Claude 触达了一个没有 MCP 就根本碰不到的世界。

MCP 的几个关键判断

判断一 · 不是所有外部访问都需要 MCP。Claude Code 自带 Bash、Read、Write 等工具，本地文件、本地命令、curl 一个公开 API，都不需要 MCP。MCP 的价值在于：外部系统有持久身份（token/session）、需要结构化协议、希望被多个 Agent 复用。比如查 Linear、读 PostgreSQL、调内部 RPC 服务、操作 Figma——这些是 MCP 的本职。

判断二 · MCP server 是独立进程，要单独跑。MCP server 通常是个长驻进程（stdio 模式或 HTTP），Claude Code 通过 settings.json 注册要连哪些 server，启动时建立连接。这意味着配 MCP 比配 Skill 重——你要管 server 的生命周期、鉴权、错误处理、版本升级。能用 Skill + Bash 拼出来的事，不必上 MCP。

判断三 · MCP 是跨 Agent 标准。这是 MCP 跟其他三把刀最大的战略差别。Skill、Subagent、Hook 都是 Claude Code 专有，你写的只能给 Claude Code 用。MCP 是 Anthropic 推动、其他 Agent 厂商（Cursor、Cline、Continue.dev 等）逐渐采纳的开放协议——你为飞书写一个 MCP server,Claude Code 能用、Cursor 也能用、未来其他 Agent 都能用。这是 Anthropic 在 Agent 时代的关键布局，详细在 第 12 章 讲。

i 一个常被混淆的概念 · 「MCP server」≠「Skill」

有人觉得 lark-mail 既然是个 Skill，飞书的 MCP server 也叫 lark-cli，两个是不是一回事？——不是。MCP server 是能力提供方,Skill 是能力使用方。lark-cli 这个 MCP server 暴露「发邮件、查日历、找联系人」这些原子能力；lark-mail/SKILL.md 是教 Claude怎么用这些原子能力组装出「给团队发周报」这种业务流程。同一个 MCP server 可以服务多个 Skill（lark-mail、lark-calendar、lark-task 都调 lark-cli），同一个 Skill 也可以用多个 MCP server。两层职责清楚，组合才灵活。

8.6遇到一个需求，该用哪把刀 The Decision Flow

把前面五节的内容压成四个递进的问题。下次遇到一个新需求，顺着这四问走一遍，基本不会选错刀。

问题 1
需要访问外部系统？
查 Linear / 读 PostgreSQL / 调内部 API → 配 MCP server
问题 2
需要强制每次都跑？
commit 前 lint / edit 后 format / 注入上下文 → 写 Hook
问题 3
需要并行 / 隔离上下文？
3+ 独立子任务 / 大调研 / 探索性发散 → 派 Subagent
问题 4
教 Claude 怎么做某类活？
按需触发的工作流（发邮件 / 转 PDF） → 写 Skill

这四个问题不是互斥的。一个复杂场景可能同时触发多个答案——这时四把刀一起用，各管一段。拿几个容易选错的需求对一对就清楚了。

需求该用为什么不是别的
每次 commit 前自动扫密钥 Hook(PreToolUse) 必须强制，模型不能跳过 → 不能只放 Skill
用统一格式生成 PR 描述 Skill 跨项目通用流程，按需加载，不是每次都用
同时调研 5 家竞品的官网 Subagent（并行） 5 个子任务彼此独立，各自结果摘要回报，主代理负责整合
从 Figma 拉最新设计稿 MCP(Figma server) 需要触达 Figma 服务，有持久 token，要走标准协议
把项目部署到 Production Skill + MCP + Hook Skill 教流程、MCP 访问部署系统、Hook 在 deploy 前强制安全扫描
整理 50 篇论文的核心结论 Subagent + Skill Subagent 并行读论文回摘要，Skill 教整合格式
项目必须用 TypeScript 严格模式 CLAUDE.md（不是 Skill） 每次对话都必须知道的硬规则，不该按需加载

几个最容易选错的，值得各点一句：每次 commit 前扫密钥，得用 Hook——因为它必须强制、模型不能跳过，只放进 Skill 就守不住；用统一格式生成 PR 描述，该写 Skill 而不是塞 CLAUDE.md——它是跨项目通用、按需加载，不是每次对话都得知道；反过来，项目必须用 TypeScript 严格模式这种，恰恰要写进 CLAUDE.md——因为它每次对话都得在，不该等场景触发才加载。一句话：看这件事是「强制」「按需」还是「常驻」，刀就选出来了。

✕ 反共识 · 「能不能只用一把刀解决所有问题」

很多人（尤其是刚接触 Claude Code 的工程师）倾向找一把「万能刀」——只用 Skill / 只用 Subagent / 只用 MCP。这种倾向通常导致工程畸形：什么都塞 Skill（Skill 变成 CLAUDE.md 的兄弟）、什么都开 Subagent（主任务被 overhead 拖垮）、什么都走 MCP（本来 Bash 一行能做的事被包装成 server）。四把刀各有不可替代的位置，正确的姿势是分层组合而不是单点放大。回到上一章 EPCC 框架里看——E 阶段可以用 Subagent 并行探索，P 阶段用 Skill 加载流程模板，C 阶段调 MCP 真正动外部系统，Hook 在每一步前后做强制检查。四个全用上，EPCC 才跑得稳。

8.7组合使用 · 完整高级用法 Composing the Four

最后看一个完整例子——把四把刀都用上，做一个真实的「自动化发布到 Production」流程。这个例子源自 Anthropic Skilljar 的 Claude Code in Action 课程，展示了高级用法的全貌 [Anthropic Academy 2025c]。

场景

用户说：「把 main 分支当前的代码发布到 production。」期望 Claude 完成完整的发布流程，包括预检、构建、部署、烟雾测试、回滚预案。

四把刀各管一段

1. Skill — 教 Claude 怎么做完整发布

.claude/skills/deploy-prod/SKILL.md

---
name: deploy-prod
description: 发布到 production。
  Use when user mentions 发布、部署、上线、deploy to production、go live
---

## 标准流程
1. 预检 — 跑测试、检查依赖、确认 main 是绿的
2. 派 subagent 并行 — 构建产物、跑回归测试
3. 调 MCP 走 deploy — 通过 k8s-mcp server 执行
4. 烟雾测试 — 验证 production 端点
5. 失败回滚 — 自动 rollback 到上一个 tag

2. Subagent — 把构建和测试并行跑掉。流程第二步派两个 subagent：一个跑 build（读源码、生成 artifact、上传到 registry），一个跑回归测试（跑全套 e2e 测试、生成报告）。两件事彼此独立、各跑 5-10 分钟，并行 subagent 把总耗时砍半。主代理拿到两份摘要，继续走第三步。

3. MCP — 通过 k8s-mcp server 实际操作集群。Claude 不需要懂 kubectl 的语法，它调 MCP server 暴露的 deploy_to_prod 工具，server 负责把请求翻译成 kubectl apply，管理鉴权、token 轮换、操作日志。同一个 MCP server 可以服务 staging、prod、disaster-recovery 多套集群。

4. Hook — 在 deploy 前强制跑安全扫描。settings.json 里挂一个 PreToolUse hook，匹配 MCP 工具 deploy_to_prod，先跑安全扫描脚本（检查镜像漏洞、检查环境变量泄露、检查 RBAC 配置）。扫描失败，Claude 调 MCP 的请求就被 hook 阻断，deploy 根本不会发生。Claude 模型层面不需要懂这个 — Hook 是系统层面的保险。

这四层叠起来，完整的流程是这样跑的：

  • 用户说「发布到 production」
  • Claude 匹配 deploy-prod Skill，加载流程
  • 按 Skill 步骤 1 跑预检
  • 按步骤 2 派两个 subagent 并行 build + test
  • 按步骤 3 调 k8s-mcp::deploy_to_prod
  • Hook 触发，跑安全扫描 — 通过
  • MCP server 执行 deploy
  • 步骤 4 跑烟雾测试
  • 结束，主代理向用户汇报
✓ 武器 · 「分层 + 组合」是 Claude Code 工程化的最终形态

这一章的所有内容，可以浓缩成一句话：不要把四把刀当成「四个选项」，要当成「四个层」。MCP 是底层能力供给、Hook 是策略约束、Subagent 是任务编排、Skill 是工作流封装。一个成熟的 Claude Code 项目通常长这样：.claude 目录下有十几个 Skill 教不同场景的工作流、settings.json 里挂 3-5 个关键 Hook、registers 几个 MCP server 提供外部访问，主代理在跑任务时按需派 Subagent 并行。四层叠合后，Claude Code 不再是一个聊天工具，而是一个被你按团队工程规范定制过的协作平台。

8.8这一章给你的判断武器

读完合上，带走以下几个能直接用的判断武器。

✓ 武器一 · 遇到新需求，先按四问决策

「要访问外部系统吗？」→ MCP。「要强制每次都跑吗？」→ Hook。「要并行或隔离上下文吗？」→ Subagent。「要教 Claude 一类工作流吗？」→ Skill。这四问不是互斥的，可以多个都为是，这时就组合用。不要先想「我要写个 Skill」，先想「我要解决什么」，再选对应的刀。

✓ 武器二 · Skill 跟 CLAUDE.md 的判别标准只有一条

这段内容是不是每次对话都必须知道？是 → CLAUDE.md；不是 → Skill。CLAUDE.md 是常驻成本，Skill 是按需成本。把这条想清楚，就不会再把 CLAUDE.md 写成 3000 字的垃圾桶，也不会再把通用上下文塞进 Skill 让所有任务都加载它。

✓ 武器三 · Subagent 的派发门槛比你想的高得多

「单任务预计 5 分钟以上 + 跟其他子任务真的独立 + 主代理只需要总结」三条同时满足，才派 Subagent。不满足任何一条就直接在主代理里串行做。Subagent 是协作工具，不是流程必修——能 Bash 一行解决的事，绝对不开 agent。
